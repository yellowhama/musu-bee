# NOTE — Chat is a Web GUI surface, not MUSU core (2026-04-09)

요지: “MUSU에서 ‘대화(채팅)’는 필수가 아니다. 채팅은 `musu.pro`에서 어디서든 접속 가능한 웹 GUI/원격 운영면을 만들기 위해 필요했던 상위 인터페이스다.”

---

## 제품 포지션 재정의

**MUSU의 본체는 채팅 제품이 아니다.**  
채팅은 기본 사용 방식이 아니라, **원격 접속 가능한 웹 GUI를 만들기 위한 상위 인터페이스**다.

---

## 지금 말로 재정의한 MUSU

### 1) 기본 사용 경로

핵심 사용은 아래다.

- 각 컴퓨터에서 `Codex CLI`
- 각 컴퓨터에서 `Claude Code`
- MCP나 CLI를 통해 MUSU 기능 사용

즉 로컬 작업 환경 안에서 바로 붙는 **작업 운영 레이어**가 본체다.

### 2) 채팅 기능의 진짜 이유

채팅은 “MUSU는 메신저 제품입니다”가 아니라,

- `musu.pro` 웹사이트에서
- 어느 기기에서든 접속해서
- 원격으로 상태를 보고
- 지시하고
- 작업을 이어받기 위한
- **웹 GUI 진입점**

이 목적 때문에 필요한 것이다.

---

## 이 관점으로 보면 이전 문서들의 해석도 수정돼야 한다

이전에는 `PRODUCT_VISION`을 보고 “업무 메신저가 제품의 중심”으로 읽힐 여지가 있었는데,
정확히는 아래로 정리돼야 한다.

### 수정된 해석

- **본체**: AI 운영체계 / control layer / execution layer
- **주 사용 인터페이스**: Codex CLI, Claude Code, MCP
- **채팅 UI**: 선택적 원격 운영면
- **웹 GUI**: Pro/유료 버전의 접근성과 관리성 강화 수단

즉 채팅은 코어가 아니라 **관리 인터페이스**다.

---

## 이게 왜 중요한가

이 한 문장 차이로 제품 우선순위가 바뀐다.

### 잘못 잡으면

- 채팅 UX를 너무 일찍 만든다
- 메시지 UI에 집착한다
- 메신저처럼 보여야 한다고 착각한다
- 본체인 control/runtime 설계가 늦어진다

### 제대로 잡으면

먼저 만들어야 할 것은:

1) **CLI/MCP에서 MUSU 기능을 잘 쓰게 하는 것**
2) **각 컴퓨터의 runtime/control surface를 안정화하는 것**
3) **원격 접속용 웹 GUI는 그 위에 얹는 것**

---

## 제품 구조를 깔끔하게 말하면

**MUSU는 여러 컴퓨터에 흩어진 AI 실행 능력을 운영하는 control plane이고,  
CLI/MCP가 기본 인터페이스이며, 채팅형 웹 GUI는 원격 접속과 운영 편의를 위한 상위 인터페이스다.**

이 정의가 좋은 이유:

- Codex CLI / Claude Code와 충돌하지 않음
- 오히려 그 위에서 더 잘 작동하는 운영 레이어가 됨
- 웹 GUI의 존재 이유가 분명해짐
- 유료화 포인트와도 자연스럽게 연결됨

---

## 그러면 문서도 이렇게 고쳐야 한다(제안)

### 1) `PRODUCT_VISION.md`

- 채팅은 **하나의 인터페이스**
- 기본 인터페이스는 **CLI / MCP**
- 웹 채팅은 **원격 운영 및 접근성 레이어**

즉 “유저는 말만 한다”를 유지하되, 그 말이 채팅창에만 갇힐 필요는 없음을 명시한다.

### 2) `MUSU_BLUEPRINT.md`

Layer 4 UI의 우선순위를 “개발/도그푸딩 UI”와 “제품 UI”로 분리해서 더 명확히 적는 것이 좋다.

- Claude Code / CLI / MCP = 주 사용 경로
- Web GUI / Chat = 원격 접속용 운영면
- Desktop = 장기 제품 UX

### 3) 제품 카피

“AI 팀 업무 메신저”를 전면에 두면 오해를 부를 수 있다. 더 정확한 카피는:

- AI 운영체계
- 분산 AI control plane
- CLI/MCP 기반 운영면
- 웹에서 어디서든 접속 가능한 원격 control UI

메신저는 보조 비유로만 쓰는 편이 낫다.

---

## 유료 버전 관점에서도 자연스럽다

설명 기준으로 채팅 기능은 무료/코어의 본질이 아니라, **유료 버전에서 가치가 커지는 원격 운영 기능**으로 읽힌다.

- 무료: 로컬 CLI/MCP 중심
- 유료: 웹 GUI, 원격 접속, 어디서든 운영, 상태 관찰, 지시, 이어받기

즉 채팅 기능은 “예쁜 UI”가 아니라 **접근성 + 운영성 + 원격성**에 돈을 받는 이유가 된다.

---

## 결론

- **채팅은 필수가 아니다**
- **핵심은 Codex CLI / Claude Code / MCP에서 쓰는 운영 레이어다**
- **채팅/웹 GUI는 `musu.pro`에서 어디서든 붙기 위한 원격 control surface다**
- **채팅 기능은 본체가 아니라 상위 운영 인터페이스다**
