## 1. 큰 그림

V24는 musu Layer 1 (machines) 구현체를 Python 25,885 LOC → Rust 단일 binary로 옮긴 작업. 사양은 그대로, substrate만 바뀐 implementation migration. 4-layer (CoS / machines / companies / agents) thesis는 V23.5에서 lock된 상태에서 진입했고 V24 panel debate에서도 손대지 않았다.

핵심 변화 두 가지:
- **단일 binary 6 subcommand**: `musu bridge|control|writer|indexer|install|supervise` — 기존 Python 5 패키지 (musu-bridge / musu-core / musu-control / musu-indexer / musu-writer) + scripts/install.{sh,ps1} 통합.
- **per-workspace SQLite + FTS5**: 인덱서는 `<work_dir>/.musu_dev.db` 단위로 끊김. 기존 musu-indexer Python 가 process-wide 였던 것과의 결정적 차이.

→ Phase 0 Researcher가 "spec verifier"가 아니라 "scope cutter" 역할을 했다. [[feedback-strategic-critic-gate]] panel-as-thesis-stress-tester 와 같은 패턴이 Phase 0 에서도 valid.

## 10. 정성 verdict

**원래 가설**: "Rust로 옮기면 startup latency + memory + cross-platform install 셋이 동시에 좋아진다."
**검증**: R8 (4060Ti E2E) closure 가 cold-start `musu bridge` 가 Python 보다 빠르다고 했고 (operator-observed, 정량 측정 안 됨), single binary install 은 `cargo build --release` 한 번으로 6 subcommand 동시 ship. cross-platform 은 R6 가 systemd / launchd / Windows SCM 3개 backend 다 커버.

**미검증**: cross-machine routing (R9). memory footprint 정량 (operator instrumentation 없음). Python 25k LOC 가 사라진 뒤 git history pruning 효과 (R10 이후).

**위험 정리**:
- HIGH 잔여 없음 (Critic/Auditor union 14 모두 audit-fix or plan-revise)
- 잔여 MED: 위 §8 "Python dead code 가 working tree 에 남아있어 file 탐색 confusion" 단 1건 — operator R10 으로 해소

**[[feedback-no-python]] 위반 해소**: 본 master plan 자체가 6주에 걸친 violation accumulation 청산. 새 sub-WS 는 Rust only. 위반 재발 방지는 `MODE_Agent_Team.md` Phase -1 + Phase 0 Researcher frame check.
