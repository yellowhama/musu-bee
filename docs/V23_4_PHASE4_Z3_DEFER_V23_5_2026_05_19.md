# V23.4 Phase 4 T2-Z3 omnibus defer — wiki/442

**Date**: 2026-05-19
**Branch**: `v23/phase4`
**Scope**: 0 LOC code, single defer doc covering 4 V23.3 forward-pointers
**Status**: deferred to V23.5

Four forward-pointers in V23.3 wiki/396 §5 share the same root deferral cause: their dependency targets (worker-sidecar runtime, `musu-health` CLI binary, in-toto attestation harness) do not yet exist in the V23.4 codebase. Implementing the pointers ahead of those targets would produce dead code that never executes.

## F-A1c-5 — worker-sidecar bench coverage

**Original intent**: Extend `bridge-bench.sh` (Z4a, this phase) with a "with-sidecar" run that measures the per-workspace sidecar latency tax.

**Why defer**: The worker-sidecar runtime pattern does not exist. V23.4 ships a single `musu-bridge` Deployment per K3s node serving all workspaces from one process. A sidecar pattern would require ratio decisions (one sidecar per workspace? per company?) that belong to V23.5/V23.6 architecture work.

**Reactivation criterion**: When V23.5+ adds a sidecar runtime pattern, this scenario becomes a 30-60 LOC extension to `bridge-bench.sh` — add a third per-run mode alongside the existing in-cluster bridge bench.

## F-A1c-6 — musu-health CLI status command

**Original intent**: A user-facing `musu-health` Windows CLI binary that reports gateway + bridge + signaling reachability in one shot. The forward-pointer text in V23.3 wiki/396 §5 explicitly marks it "V23.5-horizon" — the binary itself is not in V23.4 scope.

**Why defer**: The conditional defer is the right call. There is no `musu-health` binary on disk in the V23.4 codebase, so any V23.4 work on a status command would be premature.

**Reactivation criterion**: When V23.5 ships the `musu-health` CLI scaffold, fold this status-command work in as a same-PR feature (not a follow-up forward-pointer).

## F-A1c-7 — in-toto attestation harness

**Original intent**: Generate in-toto attestations for `musu-backend.tar` so downstream installers can verify the supply-chain provenance of the airgap tar.

**Why defer**: The in-toto attestation harness (key management, attestation generator, verification path) is not present in V23.4. Adding `in-toto-run` wrapper calls without the harness produces orphan attestation files that no installer path consumes. This is a V23.5 supply-chain workstream, not a Z-batch residual.

**Reactivation criterion**: When V23.5 commits to the supply-chain workstream and selects an attestation format (in-toto-attestation vs SLSA provenance vs sigstore cosign), wire the attestation step into `build-musu-backend.sh` step 9 (right next to the existing sha256 sidecar).

## F-A1c-8 — installer telemetry signal-loss audit

**Original intent**: Audit the installer telemetry write-path for silent drops (e.g., `_Invoke-MusuInstallAttemptTelemetry` failures that swallow exceptions).

**Why defer**: The original forward-pointer is marked "V23.5-horizon" in V23.3 wiki/396 §5 because the audit shape depends on V23.5's telemetry schema decisions (whether install_attempt remains the only failure channel or whether a separate `install_telemetry_loss` counter joins it). Auditing the current path now would re-audit again post-V23.5; defer once.

**Reactivation criterion**: When V23.5 freezes the install-telemetry counter schema, run the audit against the new shape.
