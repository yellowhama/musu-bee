## §3 Rate-limit design

### 3.1 Substrate decision (resolves OQ-1)

Per Researcher Phase 0 R5 finding: `fly.toml` lines 32-36 declares only connection-concurrency soft 200/hard 250; no per-endpoint rate-limit primitive. No in-code rate-limit middleware exists in the musu-relay codebase. B2 implements an **in-memory token-bucket per Fly instance**, scoped per `(musu_install_id, source_ip)` tuple.

**Trade-off accepted for V23.3** (per OQ-1 resolution + wiki/379 §7.2 mitigation #3 precedent): the rate-limit is per-Fly-instance and stateless. With `min_machines_running = 1` in fly.toml:26 the cluster usually has exactly one machine, so per-instance ≈ global; if Fly auto-scales (auto_stop_machines = "off" + bursty traffic) a determined caller could amortize their per-tuple budget by hopping machines via Fly's edge proxy. For pre-bootstrap installer telemetry the abuse surface is small: max payload 16KB, 1KB DB row, ~5-10 POSTs per failed install, and the caller has no incentive to spam (no auth bypass, no oracle leak). Global rate-limit is a V23.4 forward-pointer.

### 3.2 Token-bucket spec

Implementation lives as a module-private object in `telemetry.ts` (sibling to `_warnedNoSharedSecret` / `_warnedNoAdminSecret` state at telemetry.ts:212-213):

```typescript
// telemetry.ts — install-attempt rate limiter (V23.3 B2 / wiki/390)
// Per-instance in-memory token bucket. Scoped per (install_id, source_ip)
// tuple. NOT global — fly.toml min_machines_running=1 makes per-instance
// approximately global at steady state; auto-scale hops are a V23.4
// follow-on (see plan §3.1 trade-off).
//
// Memory bound: a Map<string, BucketState> with ~10K active entries cap.
// Each entry ~80 bytes (key ~50 bytes + state ~30 bytes); ~800KB worst
// case. Reaper sweeps entries older than REAPER_MAX_AGE_MS every
// REAPER_INTERVAL_MS. Test isolation via _resetInstallAttemptLimiter().

interface BucketState {
  tokens: number;       // float, refilled lazily on hit
  lastRefillMs: number; // Date.now() of last refill
}

const RL_CAPACITY = 20;            // max burst per tuple
const RL_REFILL_PER_HOUR = 20;     // 20 tokens/hr ≈ one every 3min
const RL_REFILL_PER_MS = RL_REFILL_PER_HOUR / (60 * 60 * 1000);
const RL_REAPER_INTERVAL_MS = 5 * 60_000;
const RL_REAPER_MAX_AGE_MS = 60 * 60_000;
const RL_MAP_HARD_CAP = 10_000;

const _installAttemptBuckets = new Map<string, BucketState>();
let _lastReaperRunMs = 0;

function _consumeInstallAttemptToken(
  installId: string,
  sourceIp: string,
): { allowed: boolean; retryAfterSec: number } {
  const now = Date.now();
  // Opportunistic reaper — runs at most once per RL_REAPER_INTERVAL_MS
  // on the hot path; bounded work per call.
  if (now - _lastReaperRunMs > RL_REAPER_INTERVAL_MS) {
    _lastReaperRunMs = now;
    for (const [k, v] of _installAttemptBuckets) {
      if (now - v.lastRefillMs > RL_REAPER_MAX_AGE_MS) {
        _installAttemptBuckets.delete(k);
      }
    }
  }
  // Hard cap — if the map is at the cap and the key is new, refuse with
  // retry-after=60s and DO NOT insert. Prevents adversarial Map growth.
  const key = `${installId}|${sourceIp}`;
  let st = _installAttemptBuckets.get(key);
  if (!st) {
    if (_installAttemptBuckets.size >= RL_MAP_HARD_CAP) {
      return { allowed: false, retryAfterSec: 60 };
    }
    st = { tokens: RL_CAPACITY, lastRefillMs: now };
    _installAttemptBuckets.set(key, st);
  } else {
    // Lazy refill: add (elapsed_ms * refill_per_ms) tokens, clamp to capacity.
    const elapsed = now - st.lastRefillMs;
    st.tokens = Math.min(RL_CAPACITY, st.tokens + elapsed * RL_REFILL_PER_MS);
    st.lastRefillMs = now;
  }
  if (st.tokens < 1) {
    const deficit = 1 - st.tokens;
    return {
      allowed: false,
      retryAfterSec: Math.max(1, Math.ceil(deficit / RL_REFILL_PER_MS / 1000)),
    };
  }
  st.tokens -= 1;
  return { allowed: true, retryAfterSec: 0 };
}

// Test isolation — call from beforeEach() in install-attempt.test.ts.
export function _resetInstallAttemptLimiter(): void {
  _installAttemptBuckets.clear();
  _lastReaperRunMs = 0;
}
```

### 3.3 Limits + response codes

- **Capacity**: 20 tokens per tuple, refill 20/hour (~one every 3 minutes)
- **Burst**: full 20 absorbed immediately on first contact (cold start case — installer fails fast at step 3 and emits one POST per failure)
- **Sustained**: 20/hour comfortably covers the 9 trigger sites × 3 max retries = 27 POSTs per install (still under the long-tail 20/hour after the first burst is consumed)
- **Memory bound**: ~10K active tuples = ~800KB worst case; reaper sweeps entries older than 1h every 5min on hot path
- **Hard cap on Map size**: 10K entries — beyond which new keys are refused with `429` + `Retry-After: 60`

Response codes:
- `204 No Content` — accepted and inserted
- `400 Bad Request` — field validation failed (see §4.3)
- `429 Too Many Requests` — token bucket empty; response includes `Retry-After: <seconds>` header
- `500 Internal Server Error` — DB write failed (better-sqlite3 throws); should be impossible in steady state

### 3.4 Source IP derivation

**Per Critic C-B2-H1 (HIGH, resolved)**: Express 4 with default `trust proxy = false` makes `req.ip` always the socket-peer IP — which on Fly.io is ALWAYS the Fly edge proxy, not the client. Without the fix below, `(install_id, source_ip)` collapses to `(install_id, <fly-proxy-ip>)` which is functionally per-install_id-only, and an attacker who rotates `install_id` (a free `[Guid]::NewGuid().ToString("N")` call) gets unlimited 20-token buckets. **Mandatory Builder action**:

1. **Add to `server.ts:310`** (immediately after `const app = express()`):
   ```typescript
   // V23.3 B2 (wiki/390 §3.4 + Critic C-B2-H1): Fly proxy is single-hop;
   // trust exactly one hop so req.ip consumes the right-most X-Forwarded-For
   // entry (the actual client IP), not the Fly proxy socket peer.
   // Side effect: /install, /nat_pierce, /agent_spawn now also see real
   // client IPs if they ever read req.ip — positive, defensive change.
   app.set("trust proxy", 1);
   ```
2. **Test case T16** in `install-attempt.test.ts`: supertest POST with `X-Forwarded-For: 1.2.3.4` header — assert the stored `source_ip_hash` equals `sha256("1.2.3.4").substring(0,8)`, NOT the hash of `::ffff:127.0.0.1` or `127.0.0.1`. This proves the wiring end-to-end.
3. **Curl smoke** (§6.2 addition): a same-install_id, same-(real)-IP 21-burst test that proves the bucket exhausts after 20 under the new trust-proxy regime.

Privacy posture below (§3.5) still holds: only the 8-char hash lands in the DB; raw IP stays in Fly proxy logs only.

### 3.5 Privacy posture

`source_ip_hash` stored is `sha256(req.ip).substring(0, 8)` (32-bit prefix; ~4B-cardinality budget — collision-tolerant for analytics). Raw IP never written. Matches the telemetry.ts header promise at telemetry.ts:6-9 ("NEVER stores user identity, workspace contents, agent outputs, file paths, code, chat history. Plumbing health ONLY").

### 4.2 Handler signature

```typescript
// POST /v1/telemetry/install_attempt — UNAUTH (V23.3 B2 / wiki/390)
//
// Pre-bootstrap failure telemetry. Hosts that never reach a working
// install have no account_key, so /install (HMAC-authed) is unreachable.
// This endpoint accepts an unauth POST per failure step from
// install-wsl2.ps1, rate-limited per (musu_install_id, source_ip).
//
// Auth: deliberately none. See plan §1.2 + §3.5 privacy posture.
// Validation: see §4.3.
// Rate-limit: 20 tokens/hr per tuple; 429 + Retry-After on exhaustion.
// Schema: install_attempt table (v42), see plan §2.
router.post("/install_attempt", (req, res) => {
  // No requireInstallHmac() / requireTelemetrySecret() — UNAUTH by design.
  const b = req.body || {};

  // §4.3 field validation
  const installId = asStr(b.musu_install_id);
  if (!installId || !INSTALL_ID_RE.test(installId)) {
    res.status(400).json({ error: "missing or malformed musu_install_id" });
    return;
  }
  const step = asStr(b.step);
  const errorClass = asStr(b.error_class);
  const elapsedMs = asInt(b.elapsed_ms);
  if (!step || !errorClass || elapsedMs === null) {
    res.status(400).json({ error: "missing required fields" });
    return;
  }
  // Length cap on free-form strings (defense against junk payloads even
  // under the 16KB express.json limit — small string × many rows is
  // still cheap to defeat the cap):
  if (step.length > 64 || errorClass.length > 64) {
    res.status(400).json({ error: "step/error_class too long" });
    return;
  }
  // Optional context fields — accept-and-clamp.
  const osVersion = asStr(b.os_version);
  const biosVt = asStr(b.bios_vt);
  const hostClass = asStr(b.host_class);
  const installerVersion = asStr(b.installer_version);
  const optCap = (v: string | null) =>
    v && v.length <= 128 ? v : null;

  // §3 rate-limit
  const sourceIp = req.ip || "0.0.0.0";
  const rl = _consumeInstallAttemptToken(installId, sourceIp);
  if (!rl.allowed) {
    res.setHeader("Retry-After", String(rl.retryAfterSec));
    res.status(429).json({ error: "rate limit exceeded" });
    return;
  }

  const sourceIpHash = createHash("sha256")
    .update(sourceIp)
    .digest("hex")
    .substring(0, 8);

  _db
    .prepare(
      `INSERT INTO install_attempt
         (received_at, musu_install_id, step, error_class, elapsed_ms,
          os_version, bios_vt, host_class, installer_version,
          source_ip_hash, schema_version)
       VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?)`,
    )
    .run(
      Date.now(),
      installId,
      step,
      errorClass,
      elapsedMs,
      optCap(osVersion),
      optCap(biosVt),
      optCap(hostClass),
      optCap(installerVersion),
      sourceIpHash,
      1, // schema_version (payload-level)
    );
  res.status(204).end();
});
```

### 4.4 Status code summary

- `204` — accepted; row inserted
- `400` — missing/malformed required field; body shape: `{ error: "<reason>" }`
- `429` — rate-limit hit; `Retry-After: <seconds>` header set; body shape: `{ error: "rate limit exceeded" }`
- `500` — DB exception bubbles up via Express default error handler (matches existing routes)

### 4.5 What the handler explicitly does NOT do

- No echo of `t`, server-now, or any oracle (matches `requireInstallHmac` discipline at telemetry.ts:382-387)
- No log of raw IP (only the 8-char hash lands in the DB; raw IP stays in Fly proxy logs only)
- No leak of `INSTALL_ID_RE` shape in the 400 message (says "malformed" without disclosing regex — minor defense-in-depth)
- No fallback to `requireTelemetrySecret` even when `MUSU_TELEMETRY_SHARED_SECRET` is set (this endpoint is unauth by design — NOT dual-accept like `requireInstallHmac` at telemetry.ts:362-367)

## §7 Installer caller integration

### 7.1 `Send-MusuInstallAttempt` helper (Musu-Common.psm1)

New exported function in Musu-Common.psm1, sibling to `Save-MusuFailureDump` (Musu-Common.psm1:133-193). The helper is **non-blocking** on network failure (matches Musu-Common.psm1:10-14 C7 MEDIUM precedent — install failures must not be masked by telemetry failures).

```powershell
function Send-MusuInstallAttempt {
    <#
    .SYNOPSIS
    Best-effort POST to musu-relay's /v1/telemetry/install_attempt
    endpoint. NEVER throws; NEVER logs the response body to console.
    Designed to wrap Save-MusuFailureDump call sites without changing
    their throw semantics.

    .DESCRIPTION
    Unauth POST per V23.3 B2 (wiki/390). Pre-bootstrap installer
    failure telemetry — runs even when the host has no account_key.
    Rate-limited server-side (20 POSTs/hr per install_id + source_ip).

    .PARAMETER InstallId
    The 32-hex musu_install_id from New-MusuInstallId.

    .PARAMETER Step
    Failure step name (matches Save-MusuFailureDump -Step values).

    .PARAMETER ErrorClass
    Failure class (matches Save-MusuFailureDump -ErrorClass values).

    .PARAMETER ElapsedMs
    Milliseconds elapsed before failure.

    .PARAMETER OsVersion
    Optional Windows OS version string.

    .PARAMETER BiosVt
    Optional 'yes'|'no'|'unknown'.

    .PARAMETER HostClass
    Optional host_class string.

    .PARAMETER InstallerVersion
    Optional installer version string (from $InstallerVersion constant).

    .PARAMETER SigningBase
    The signaling URL base (default https://signaling.musu.pro). Overridable
    for dev/test against a local musu-relay.

    .OUTPUTS
    None. Best-effort fire-and-forget.
    #>
    param(
        [Parameter(Mandatory = $true)][string]$InstallId,
        [Parameter(Mandatory = $true)][string]$Step,
        [Parameter(Mandatory = $true)][string]$ErrorClass,
        [int]$ElapsedMs = 0,
        [string]$OsVersion = "",
        [string]$BiosVt = "",
        [string]$HostClass = "",
        [string]$InstallerVersion = "",
        [string]$SigningBase = "https://signaling.musu.pro"
    )
    # Validate install_id shape locally; if malformed, do NOT POST
    # (server will 400 anyway and a malformed id usually means we
    # haven't loaded New-MusuInstallId yet — emitting would be a bug).
    if ($InstallId -notmatch '^[0-9a-f]{32}$') {
        return
    }
    $body = [ordered]@{
        musu_install_id = $InstallId
        step            = $Step
        error_class     = $ErrorClass
        elapsed_ms      = $ElapsedMs
    }
    if ($OsVersion)        { $body['os_version']        = $OsVersion }
    if ($BiosVt)           { $body['bios_vt']           = $BiosVt }
    if ($HostClass)        { $body['host_class']        = $HostClass }
    if ($InstallerVersion) { $body['installer_version'] = $InstallerVersion }
    $json = $body | ConvertTo-Json -Compress
    try {
        Invoke-RestMethod `
            -Uri "$SigningBase/v1/telemetry/install_attempt" `
            -Method POST `
            -ContentType "application/json" `
            -Body $json `
            -UseBasicParsing `
            -TimeoutSec 5 `
            -ErrorAction Stop | Out-Null
    } catch {
        # Best-effort. Network down, server 4xx/5xx, DNS fail — all
        # swallowed. Failure dump on local disk (Save-MusuFailureDump)
        # is the durable record.
    }
}
```

### 10.2 Schema rollback (only if Const III later reverses)

`install_attempt` is append-only telemetry; dropping it loses only collected rows. There is no FK, no application-side SELECT outside the route handler itself, and no downstream aggregation pipeline at V23.3 plan time (V23.5+ rollup work is forward-pointed).

Operator workflow on rollback:

```sql
-- Run inside `fly ssh console` against /data/telemetry.db on signaling.musu.pro:
DROP TABLE IF EXISTS install_attempt;
DELETE FROM schema_version WHERE version = 42;
```

After SQL drop + `git revert` + `fly deploy`, the server boots at v41 with no v42 trace. Non-destructive to upstream telemetry tables. Rollback estimated at <2min wall-clock once operator decides.

### 10.3 Helper rollback (PowerShell side)

`Send-MusuInstallAttempt` is best-effort and non-blocking by design (§7.4). Reverting just the `Send-MusuInstallAttempt` call sites in install-wsl2.ps1 (keeping the helper definition in Musu-Common.psm1) is the lightest-touch partial rollback. The helper unused in installer paths costs nothing — module load adds <1ms.

### 10.4 Disable without revert

If a future operator wants to mute install_attempt POSTs without a revert:
- **Server side**: `fly secrets set MUSU_INSTALL_ATTEMPT_DISABLED=1` (and Builder adds a one-line short-circuit `return res.status(204).end()` at the top of the route handler — V23.4 follow-on if needed; not in V23.3 B2 scope).
