MUSU Architecture Overview

wiki/458 2026-05-18 Active Companion: wiki/457 · wiki/456 · wiki/455

TL;DR — musu가 무엇이고 어떻게 구성되어 있나

  1. musu = 여러 기계에서 AI 회사를 운영하는 multi-machine runtime. 한 PC가 아닌 여러 PC/laptop/server에서 동시에 agent 회사들이 일함.
  2. 4-layer 구조: ① musu-bridge 집사 (CoS) · ② 기계 (machines) · ③ 회사 (companies — musu native OR Paperclip observed) · ④ 일꾼 (agents — Claude/Gemini/Codex/Hermes adapters).
  3. 12개 musu-* modules가 active, 시스템 레벨에 6개 agent (CEO/CTO/Engineer/CoS=집사/QA/Worker)가 install 시 자동 시드, 5개 회사 template 제공.
  4. V23.5 계획: Paperclip external runtime observer 추가 (read-only, 30s poll) + LLM wiki를 HTML로 격상해 영구 기억 장치 완성.

§1 — 4-Layer 구조

사용자가 직접 musu와 대화하는 진입점은 CoS (Chief of Staff = 참모장 = 집사). CoS가 그 아래 layer들을 다 종합해서 briefing을 줌 ("Chairman Principle").

§3 — System-level 6 Agents (install 시 자동 시드)

musu install 후 seed_agents.py가 자동으로 6개 agent를 musu DB에 시드. 이건 musu 자체의 agents (개별 company 안의 agents가 아님). 사용자가 직접 대화하는 대상은 CoS (집사).

CEO — Chief Executive Officer

musu 차원 전략 결정. Frontier model (Claude Sonnet 4.6 / Gemini 2.5 Pro). $5 budget per task.

CTO — Chief Technology Officer

musu 차원 기술 결정. Frontier model. $5 budget.

CoS — Chief of Staff (참모장 = 집사) ⭐

사용자가 직접 대화하는 대상. 회사들 진행 상황 종합 briefing. 한국어 기본 (instructions/cos.md). budget model ($0.5).

Engineer

Software engineering task 처리. Frontier model. $2 budget.

QA

Quality assurance · 테스트 · 검증. Frontier model. $2 budget.

Worker

단순 task · 빠른 처리. Budget model (Claude Haiku / Gemini Flash). $0.5 budget.

CoS instruction 핵심 (한국어)

당신은 MUSU의 참모장 에이전트입니다. MUSU는 여러 대의 컴퓨터에서 AI 팀이 협력하는 업무 메신저입니다. 팀 조율, 일정 관리, 진행 상황 추적을 담당합니다.

역할: 작업 우선순위 정리 · 팀 내 커뮤니케이션 조율 · 진행 중인 작업 상태 추적 및 보고 · 병목 지점 파악 및 에스컬레이션.

원칙: 명확한 정리 · 선제적 파악 · 중립적 조율 · 한국어로 소통한다.

§4 — 5개 회사 Templates

POST /api/companies에 template_key를 넘기면 자동으로 agent roster가 시드됨. 각 template은 다른 도메인용:

dev-team — 소프트웨어 프로젝트 (4 agents)
  • Team Lead — 회사 안 일감 우선순위 + 위임
  • Planner — 작업 분해 + 일정
  • Engineer — 실구현
  • QA — 검증

가장 일반적인 template. M0 백엔드 + auth 같은 소프트웨어 build 작업에 적합.

content-team — 문서/아티클 (4 agents)
  • Team Lead
  • Researcher — 소스 조사
  • Writer — 초안 작성
  • Editor — 편집/검수
writer-studio — 장편 픽션 (5 agents)
  • Studio Lead
  • Project Manager
  • Researcher
  • Writer
  • Editor
research-team — 심층 조사 (4 agents)
  • Team Lead
  • Analyst
  • Researcher
  • Summarizer
marketing-team — 마케팅 운영 (7 agents)
  • Marketing Lead
  • Strategist
  • Content Creator
  • Social Manager
  • Analytics Lead
  • Marketing Editor
  • Content Radar

§5 — External Runtime Observer (V23.5 — Paperclip first)

사용자가 Paperclip(또는 다른 외부 OSS agent orchestrator)을 이미 쓰고 있다면, musu가 그 회사들을 read-only로 인식해서 멀티 기기에서 통합 조회 가능. 복사 안 함, 인식만.

외부 OSSLayer현재 statusmusu integration plan
Paperclip (paperclipai/paperclip) 회사 운영 (Layer 2) External OSS 66.3k stars, MIT, localhost:3100 V23.5 Pull polling (30s), read-only
Hermes Agent CLI Agent 운영 (Layer 3) Adapter 존재 musu-core/adapters/hermes.py 이미 동작 추가 작업 불필요
OpenClaw Agent 운영 (Layer 3) External (paperclip 안에서 사용) V24+ Paperclip 통해 간접 인식 우선

설정 방법 (V23.5)

# ~/.musu/runtimes.toml
[[runtime]]
type = "paperclip"
url = "http://localhost:3100"
token = ""                       # optional
poll_interval_seconds = 30

또는 musu-bee UI의 /app/runtimes 페이지에서 form으로 같은 row 추가. 자동 스캔 안 함 — 사용자 명시 등록만.

UI 통합

musu-bee 메인 페이지의 회사 목록이 두 섹션으로 분리됨:

Musu native companies

사용자가 musu에서 직접 만든 회사들. CEO/Team Lead 통한 직접 운영.

Paperclip companies (via localhost:3100)

외부 paperclip이 운영 중인 회사들. status / progress만 표시. 편집은 paperclip UI에서.

§8 — FAQ

musu 깔면 paperclip을 대체하나? 아니면 같이 쓰나?
같이 쓰는 게 default. musu는 paperclip을 *대체*하는 게 아니라 *멀티 기기 도구*로 추가됨. Paperclip 사용자는 musu 깔면:
  • Paperclip은 그대로 씀 (workflow 안 바뀜)
  • 4대 기기에서 paperclip 회사 진행 상황을 한 화면에 통합 조회 가능
  • musu의 CoS가 paperclip 회사 status도 briefing에 포함
"집사 agent" 가 뭐고 어디서 시작되나?
CoS (Chief of Staff = 참모장). seed_agents.py:88에서 musu install 시 자동 시드됨. 한국어 instruction (instructions/cos.md). 사용자가 "지금 어떤 일들이 돌고 있어?" 같은 질문하면 CoS가 모든 회사 (native + paperclip)를 종합해서 briefing.
musu native company vs Paperclip company — 어떻게 다른가?
책임 위치가 다름.
  • musu native: musu가 회사 정의 + task 실행 + 모든 상태 소유. POST /api/companies로 생성, musu의 CEO/Team Lead가 일감 위임.
  • Paperclip: paperclip이 회사 정의 + task 실행 다 소유. musu는 status만 30초마다 polling해서 보기만. 편집은 paperclip UI(localhost:3100)에서.
Paperclip이 다운되면 musu도 다운되나?
아니요. musu native companies는 paperclip과 완전 독립. musu-bee UI의 "Paperclip companies" 섹션이 unreachable 표시되고, native companies 섹션은 그대로 작동.
왜 자동 scan 안 함? 깔자마자 paperclip 인식하면 더 편할 텐데.
보안 + 명시적 동의. localhost에 다른 서비스가 같은 포트를 쓸 수도 있음. 또 사용자가 의도하지 않은 외부 연결이 자동 활성화되면 안 됨. 사용자가 musu-bee UI의 "Add runtime" 버튼을 명시적으로 누르거나 runtimes.toml에 직접 추가해야 함.
HTML wiki memory가 "영구 기억 장치"인 이유는?
musu agent들은 매 invocation이 fresh start (conversation memory 없음). 단 wiki는 영속. agent가 write_wiki_page()로 발견 사항을 저장하면 다음 turn / 다른 agent / 다른 기기에서도 search_wiki()로 찾을 수 있음. 즉 wiki = agent 간 / turn 간 공유 메모리. HTML로 격상하면 사용자도 같은 wiki를 시각적으로 즉시 흡수 가능 (Tariq use case #14 패턴).
musu-bridge 외에 다른 module들 (musu-port, musu-supervisor)은 뭐 함?
각각 다른 layer의 인프라:
  • musu-port (Rust): 네트워크 ingress · QUIC/WebSocket 브로커. mesh 안에서 다른 기기로 task 라우팅 시 사용.
  • musu-supervisor (Rust): 프로세스 supervisor. systemd 통합. agent subprocess가 crash 시 재시작.
  • musu-worker: agent subprocess 실행 wrapper. cgroup/process isolation.
  • musu-indexer: 코드베이스 검색 (FTS5). agent가 "이 함수 어디 정의됨?" 질문 시 사용.
  • musu-control: MCP server. 50+ tools를 agent에게 노출 (wiki/task/code/governance).
사용자는 이 모듈들을 직접 다루지 않음. install 후 background 서비스로 자동 운영.
왜 SQLite? Postgres가 더 안 좋나?
self-contained product 원칙. musu가 SQLite를 쓰면 사용자 PC에서 추가 dependency 없이 작동. Postgres 요구하면 "Postgres 설치하세요"가 install 첫 단계가 됨 — 사용자 onboarding 마찰. 단 V24+에서 enterprise multi-machine deploy가 demand되면 Postgres adapter 추가 후보 (현재 YAGNI).

§10 — Honesty Notes

  • "7.2/10 backend"는 V23.4 hardening이 musu-relay에만 적용된 상태 기준. V23.5에서 musu-bridge에도 같은 패턴 적용하면 8.5+ 가능.
  • 이 HTML 문서 자체가 Tariq HTML Effectiveness 패턴의 first prototype. wiki/455에서 제안한 §9 Step 2를 실제로 처음 실행. 자주 다시 읽힐 master plan + qual eval + closure doc 후보들이 같은 패턴 따라갈 예정.
  • Paperclip 통합은 V23.5 계획이지 현재 상태 아님. 이 문서의 §5는 plan, §1-4는 현재 상태.
  • module 12개 중 일부는 stub-tier (musu-ai-detector는 실제로 writer-studio editor에만 사용; cross-cutting infra 아님). "Active"는 코드 존재 + import 됨을 의미, 모든 module이 critical path 아님.
