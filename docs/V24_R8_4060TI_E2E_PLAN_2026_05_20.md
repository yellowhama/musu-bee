## §3 Step-by-step operator runbook

### Step 3.1 — Generate bridge token + start dual-port

Open a Git Bash / WSL terminal (the wrapper script is bash):

```bash
cd F:\workspace\musu-bee

# Generate token (≥32 chars)
export MUSU_BRIDGE_TOKEN=$(head -c 24 /dev/urandom | base64)
echo "Token: $MUSU_BRIDGE_TOKEN"  # save this — you'll need it for curl

# Start dual-port (Python :8071 + Rust :8070)
bash scripts/v24-rfast-dual-start.sh
```

**Expected output**:
- Python bridge boots silently on `127.0.0.1:8071`
- "musu-rs bridge starting" tracing line appears
- `Const III` schema-apply notice prints (first boot only — applies SQLite schema v1)
- "musu-rs bridge listening" line appears with `:8070`
- Process stays in foreground; Ctrl+C to stop both

**If errors here, paste them back**:
- "port already in use" → kill stale process via netstat output
- "MUSU_BRIDGE_TOKEN required" → export it as shown above
- Python startup error mentioning `BRIDGE_HOST must be 127.0.0.1` → that's the §6.1 guard working correctly; ensure no other env vars override it

### Step 3.2 — Verify /health

In a SECOND terminal:

```bash
# Replace TOKEN with the value you exported in step 3.1
curl http://127.0.0.1:8070/health

# Expected JSON shape:
# {
#   "status": "ok",
#   "version": "1.14.0-dev",
#   "auth_mode": "production",
#   "schema_applied": true,
#   "audit_schema_applied": true,
#   "audit_degraded": false,
#   "db_size_mb": <small number>,
#   "disk_free_pct": 0.0,
#   "relay": null
# }
```

**Verifies**:
- §9.2 acceptance: /health returns 200
- R2 §11 acceptance: schema_applied=true (after dual-start fired Const III)
- R1 N-1 audit-fix: audit_schema_applied=true

**If /health returns 404 or connection refused**: Rust bridge didn't start — go back to step 3.1.

### Step 3.3 — Verify /health/ready

```bash
curl -H "Authorization: Bearer $MUSU_BRIDGE_TOKEN" http://127.0.0.1:8070/health/ready

# Expected:
# {"ready":true,"schema_applied":true,"audit_schema_applied":true,...}
```

If `ready:false`, schema didn't apply. Paste error output.

**Then verify**:

## §4 Failure paths + debug

| Symptom | Likely cause | Debug |
|---|---|---|
| `port 8070 already in use` | Stale process | `netstat -ano | findstr :8070` → kill PID |
| `MUSU_BRIDGE_TOKEN required` | Token env not set | Re-export per step 3.1 |
| `Python startup: BRIDGE_HOST must be 127.0.0.1` | C-SEC-3 guard fired correctly; some env override is wrong | Check `BRIDGE_HOST` env; unset or set to 127.0.0.1 |
| `cargo build` fails on `to_ipv4_mapped` | rustc <1.80 | Upgrade toolchain: `rustup update stable` |
| `/health` returns 401 | Auth middleware running on bypass path? No — `/health` is bypassed. Bridge not running. | Check Rust bridge terminal |
| `POST /api/companies` returns 500 "schema_not_applied" | Const III didn't fire | Check step 3.1 output for Const III notice; may need `cargo run -- core` to apply explicitly |
| `cat ~/.musu/companies/...yaml` not found | YAML write hook didn't fire | Check Rust bridge stderr for errors; verify R2 integration commit `5416463` is in your git log |
| audit_log shows `actor_ip = '0.0.0.0'` | N-1 audit-fix not applied | Verify commit `e775702` is in your git log |
| audit_log empty | audit_schema_applied=false in /health | Schema didn't apply for audit_log table; check Const III notice |

**For any unexpected error**: paste the terminal output (Rust bridge stderr + curl response + tracing logs) back to orchestrator. Don't try to fix beyond the table above.
