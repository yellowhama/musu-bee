## §1 Scope

Add `setInterval`-based 30-day retention sweeper on the `install_attempt` table introduced by V23.3 B2. Without retention, single-IP attacker rotating `install_id` (free `[Guid]::NewGuid().ToString("N")`) fills 1GB Fly `/data/telemetry.db` volume in <1 day → SQLite write failure → HMAC-authed `/install`, `/nat_pierce`, `/agent_spawn` routes also break (shared volume). Cross-route DoS.

Per Critic C8 resolution in wiki/425 §2, this sub-WS is **dual-audit**: data deletion is one-way blast radius (canonical dual-audit trigger).

## §2 Implementation (full outline in wiki/425 §5.1; reproduced here for self-containment)

### 2.1 Constants (telemetry.ts, near rate-limit constants at telemetry.ts:308-317)

```typescript
const INSTALL_ATTEMPT_RETENTION_MS = 30 * 24 * 60 * 60 * 1000; // 30 days
const INSTALL_ATTEMPT_SWEEPER_INTERVAL_MS = 60 * 60 * 1000;    // 1 hour cadence
const INSTALL_ATTEMPT_SWEEPER_BATCH = 1000;                    // C9: bound event-loop block per tick
```

### 2.2 Functions

```typescript
let _installAttemptSweeperTimer: ReturnType<typeof setInterval> | null = null;

export function _runInstallAttemptSweeperOnce(): number {
  const cutoff = Date.now() - INSTALL_ATTEMPT_RETENTION_MS;
  // C9 per-tick LIMIT — SQLite DELETE...LIMIT requires compile-flag; use IN-SELECT-LIMIT.
  const info = _db
    .prepare(
      `DELETE FROM install_attempt
       WHERE id IN (
         SELECT id FROM install_attempt WHERE received_at < ? LIMIT ?
       )`,
    )
    .run(cutoff, INSTALL_ATTEMPT_SWEEPER_BATCH);
  return info.changes;
}

export function _maybeStartInstallAttemptSweeper(): void {
  if (process.env.MUSU_INSTALL_ATTEMPT_SWEEPER_DISABLED === "1") return;  // C12 safety hatch
  if (process.env.NODE_ENV !== "production") return;  // test-env exempt (jest hangs on dangling intervals)
  if (_installAttemptSweeperTimer) return;  // re-entrancy guard
  _installAttemptSweeperTimer = setInterval(
    _runInstallAttemptSweeperOnce,
    INSTALL_ATTEMPT_SWEEPER_INTERVAL_MS,
  );
}

export function _stopInstallAttemptSweeper(): void {
  if (_installAttemptSweeperTimer) {
    clearInterval(_installAttemptSweeperTimer);
    _installAttemptSweeperTimer = null;
  }
}
```

## §7 Auditor scope (per wiki/425 §11.2)

When Phase 5 dual-audit reviews landed code, MUST verify explicitly:

1. **Quality-engineer seed (deletion-correctness)**:
   - DELETE SQL targets only rows with `received_at < cutoff` (no false positives on recent rows)
   - LIMIT 1000 actually bounds the delete (verified via T22 with 1001 stale rows asserting exactly 1000 deleted in one call)
   - `_runInstallAttemptSweeperOnce()` returns accurate `info.changes` count

2. **Security-engineer seed (data-retention semantics + concurrent-write isolation)**:
   - 30-day retention vs telemetry.ts:9 header consistency (post-update)
   - Concurrent POST during sweep: better-sqlite3 sync semantics + WAL mode mean the sweep DELETE acquires write lock; concurrent INSERT waits. Verify no deadlock or starvation.
   - First-run blast radius: if production has accumulated >30-day rows, first sweep deletes them. Safety hatch documented + accessible to operator.
