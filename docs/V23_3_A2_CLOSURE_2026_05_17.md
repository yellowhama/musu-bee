## 3. Behavior change — boot-time fail-fast

A2 introduces an intentional semantics shift that warrants explicit documentation (Auditor MEDIUM #1).

`makeWrtcFactory()` at `musu-relay/src/gateway/wrtc-factory.ts:184-193` calls `loadWrtc()` **eagerly** at line 185 — i.e., at the moment the `GatewayConfig` object literal is constructed inside `main()`. Pre-A2, the throw-stub fired only when the gateway received its first WebRTC OFFER, which meant a gateway could boot, register with signaling, and look healthy from the outside while being completely unable to serve any peer.

Post-A2: if the `wrtc` native module fails to load (wrong arch, missing native deps, etc.), the gateway process exits at boot with the `loadWrtc()` error. This is the desired production semantics: a gateway that can connect to signaling but cannot serve any peer is worse than one that refuses to boot — the latter is loud and operator-actionable; the former silently presents itself as a working node and absorbs visitor connections that will then fail at WebRTC negotiation.

No flag is added to defer `loadWrtc()`; deferred-load is explicitly **not** wanted in production.

Reference: `musu-relay/src/gateway/wrtc-factory.ts:184-193`.
