## What shipped

### CEOReconciler (`controllers/ceo_reconciler.py`)

One instance per `ControllerManager`. Keyed reconciles by company id.
Pass:
1. Load company row by `req.key`.
2. List active agents scoped to the company.
3. For each agent with no in-flight `resource_requests`
   (`status IN ('pending','bound','running')`), post a new pending
   request with `Requires` derived from `agent.adapter_config.requires`.
4. The scheduler (21.C `SchedulerReconciler`) picks it up.

Idempotent: a second reconcile pass on the same company posts zero
duplicates because of the in-flight guard.

Tests (7 pass):
- happy: posts for all active agents
- skip: agent with in-flight request stays at 1 row
- repost: completes ‚Üí re-reconcile ‚Üí second request posted
- noop: missing company / idle company / paused-only company
- config: `requires` extracted from adapter_config JSON
- robust: garbage adapter_config falls back to empty `Requires`

### MachineReconciler (`controllers/machine_reconciler.py`)

One instance per `ControllerManager`. Keyed reconciles by machine id.
Status-triggered behavior:
- `online`  ‚Üí no-op
- `draining` ‚Üí no-op (filter excludes for new binds; in-flight finish)
- `offline` ‚Üí atomic reclaim: every request with
  `bound_machine_id = $1 AND status IN ('bound','running')` ‚Üí set
  `status='pending', bound_machine_id=NULL, bound_at=NULL`.

Atomicity: single `UPDATE` under `db._lock` using `cur.rowcount`
(same audit-fixed pattern as 21.C `binder.try_bind`). Idempotent ‚Äî
re-run on already-reclaimed rows touches nothing.

Tests (9 pass):
- 5 happy: online noop / draining noop / offline-bound reclaim /
  offline-running reclaim / completed unchanged
- 4 edge: missing machine / offline idempotent / only reclaims this
  machine's bindings / unknown status logged + skipped

### Bridge integration (`musu-bridge/server.py` lifespan)

In the FastAPI lifespan, before `yield`:

```python
controller_manager = ControllerManager(db=_get_backend()._db)
controller_manager.add(CEOReconciler controller for "companies")
controller_manager.add(MachineReconciler controller for "machines")
controller_manager.add(SchedulerReconciler controller for "resource_requests")
await controller_manager.start()
```

All three controllers run inside the bridge's asyncio loop, fed by
`KindSource` polling the respective tables (2s default) and woken by
the v21.B `WatchDispatcher.notify` calls already wired in to the
binder + various endpoints. Graceful `stop(timeout=10)` on shutdown.

17 bridge tests using `TestClient(app)` pass through the new
lifespan, including the v21.B watch routes.

## Why this matters (frame v9 alignment)

After 21.E the two axes from frame v9 ¬ß3 + ¬ß4 are both live as
reconciled control loops:

- **Company axis** ‚Äî CEOReconciler watches `companies`, decides what
  agents need work, drops it on the queue.
- **Machine axis** ‚Äî MachineReconciler watches `machines`, reclaims
  bindings on offline transitions (the failover story).
- **Scheduler** between them (21.C) does the matching.

Together with the v21.B `WatchDispatcher` (in-proc + cross-machine
notify), this gives us K8s's "decoupled control planes communicating
through shared state" model on a single SQLite file. No leader
election, no etcd ‚Äî the SQLite atomic UPDATE is the consensus point.
