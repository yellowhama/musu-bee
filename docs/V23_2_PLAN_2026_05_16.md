## 0. Scope in one sentence

Ship the Windows WSL2 installer + Alpine `musu-backend.tar` import path so a non-technical Windows user can go from "downloaded the .exe" to "K3s running locally + signaling reachable" in under 5 minutes, while closing the three V23.1 security/correctness HIGHs from the audit before any real-traffic deploy.

## 1. Why now

V23 master plan §10.4 names V23.2 as the WSL2-installer spike. V23.1 closure adds an additional driver: the audit surfaced three HIGH findings (authentication / fail-open / user_id squatting) that all sit on the **signaling server** between musu.pro and the user's PC. We cannot deploy `signaling.musu.pro` for the T1.14 manual demo without addressing them; without `signaling.musu.pro`, V23.1's §10.5 criterion 1 is unverifiable.

So V23.2's first half is "make signaling deploy-safe"; the second half is the master plan's "WSL2 spike."

## 2. Two workstreams in parallel

### Workstream A — V23.1 audit remediation (must precede deploy)

| ID | Severity | File:line | Fix |
|---|---|---|---|
| T2.AUTH.1 | HIGH | `signaling/server.ts:109` | Replace fail-open path with fail-closed-with-cached-grace. Network error → return cached `valid:true` if present (degraded mode for active sessions). Outside the cache window, reject. Keep circuit breaker as a logger-only signal. |
| T2.AUTH.2 | HIGH | `signaling/telemetry.ts:113-197` | Per-install HMAC: at install time, gateway POSTs to `/v1/telemetry/install` with a `musu-installer-secret` header (env-injected at installer build time). Server validates the secret and issues an `install_key` (HMAC base). All subsequent telemetry POSTs sign their JSON body with `install_key`. Schema: add `telemetry_install_keys` table. |
| T2.AUTH.3 | HIGH | `signaling/server.ts:60` | `validateToken` must accept (token) and return canonical `userId`, not (token, userId). Drop the `${token}:${userId}` cache key — key by `token` alone, derive userId from the response. HELLO message stops carrying user_id; server learns it from the validation API. |
| T2.SEC.1 | MED | `gateway/bridge.ts:100-110` | Replace `path.startsWith(prefix)` with: parse via `new URL("http://x" + path).pathname`, normalize, reject any `..` segment, then prefix-check. Add traversal tests covering `/api/../admin`, `/api/%2e%2e/admin`, `/api//admin`. |
| T2.PROTO.1 | MED | `signaling/server.ts:330-343` | `PEER_LEFT` payload includes `peer_id`. Bump signaling protocol version to v2 (HELLO carries `protocol_version: 2`, server replies WELCOME with `accepted_protocol: 2`). Backward-compatible: v1 clients still receive payload-less PEER_LEFT. Gateway uses `peer_id` to close the specific PeerSession + native handle. |
| T2.OBS.1 | MED | `gateway/wrtc-factory.ts:92-100` | `addRemoteIceCandidate`: explicit log on JSON parse failure; bound `candidate` string at 4KB; raise to telemetry as `fail_cause: "ice_invalid"` after >10 bad candidates from same peer. |
| T2.LOW.* | LOW | misc | LOWs from audit: error-message path leak (#8), missing `dc.send` throw test (#9), wall-clock test sleeps (#10), telemetry-db ephemeral default (#11), `as any` getDataChannel (#12). Bundle into a single cleanup commit at end of workstream A. |

Acceptance: a third-party can't pollute telemetry; an offline musu.pro validation API doesn't grant free auth; bridge path-traversal tests pass on 5 new vectors. Full suite still green.
