## 1. Scope

A3 extracts the HMAC sign+POST body-identity pattern that V23.2 B1 introduced as two-call-site discipline into a single helper, then swaps both call sites onto it. Both call sites — `recordOutcome` in `musu-relay/src/gateway/client.ts` and `emitInstallCompleted` in `musu-relay/src/gateway/main.ts` — now delegate to `signAndPost`. The body-identity invariant — that the same `rawBody` bytes are used for both the HMAC signing input AND the POST body — is now structural rather than per-call-site discipline. Pure refactor: byte-for-byte preservation of on-wire HMAC behavior.

A3 was split into two sub-workstreams per wiki/379 §5.1.1 to keep the helper landing reviewable on its own before the swap touches production paths.

## 3. The body-identity invariant

The invariant inherited from V23.2 B1: the client sends

```
x-musu-telemetry-signature: t=<sec>,v1=<hmac-sha256(accountKey, `${t}.${rawBody}`)>
```

AND a POST body of exactly the bytes `rawBody`. The server (telemetry endpoint) reads the raw request body **before** `express.json` parses it, recomputes the HMAC over `${t}.${rawBody}` using the account key, and compares. If body bytes differ from what the client signed even by one byte (whitespace, key ordering, anything) → 401.

**Pre-V23.3**: this was Builder discipline. Each call site had to compute `rawBody = JSON.stringify(record)` exactly **once**, then reuse the same variable for both signing input AND POST body. Two duplicated call sites; V23.2 qual eval (wiki/377 §7 lesson 5) flagged this as fragile — "structure > discipline."

**Post-A3**: the invariant is structural. `signAndPost(input)` takes `input.rawBody: string` and uses the same reference for `signedString = \`${t}.${input.rawBody}\`` (HMAC input, telemetry-hmac.ts:60) **and** `body: input.rawBody` (fetch body, telemetry-hmac.ts:76). Auditor B verified this bytewise: same reference, same string, same UTF-8 bytes both into `createHmac.update(...)` and into `fetch({ body })`.

Server-side `req.rawBody` is a Node Buffer holding the UTF-8 bytes of the client's `input.rawBody` string. Node's default UTF-8 encoding for `createHmac.update(string)` is the implicit bytewise-parity contract on the client side; this matches the server's Buffer-mode `createHmac.update(req.rawBody)` byte-for-byte.

## 5. Builder-during-test catch

Recording this for transparency about the development process.

The first A3.swap pass moved the fetch call **inside** the HMAC-only branch in `recordOutcome`. Pre-swap, the call site's structure was: assemble headers conditionally based on auth mode, then POST unconditionally. The first swap accidentally lifted the POST itself into the HMAC branch, which silently dropped POSTs in the shared-secret legacy path AND the unauth fallthrough path.

5 tests in the existing `telemetry-emit.test.ts` failed immediately:
- T1.12 success-path
- T1.12 timeout-path
- T2.AUTH.2 omits-secret
- T2.AUTH.2-final no-auth
- T2.AUTH.2-final bootstrap-503

**Fix**: restored an explicit else-branch fetch in `recordOutcome` that covers both the shared-secret legacy path and the unauth fallthrough. The HMAC branch goes through `signAndPost`; the else branch retains the original inline fetch. Re-verified: 195/195 passing.

This is exactly the regression dual-audit was designed to catch. In this case the existing `telemetry-emit.test.ts` canary caught it pre-audit, which is even better — Critic C3 (tsc-green at helper-only commit) + existing canary test (no-auth and legacy paths preserved) = belt-and-suspenders, and the audit found the post-fix codepath clean.
