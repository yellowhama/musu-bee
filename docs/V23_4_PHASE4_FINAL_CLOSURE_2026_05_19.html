§1 · Phase 4 thesis recap

Phase 4 ships personal multi-PC AI workflow execution. Backend: an asyncio workflow runner with SQLite state inside musu-bridge. Frontend: a Fleet view that lists the user's PCs and a form-based workflow builder under /c/[id]/workflows. Self-contained-product positioning preserved: fly.io retired from the critical path; the first installed PC plays signaling rendezvous; STUN-only fallback when direct WebRTC fails.

Plan v1 was RED at the Phase −1 strategic gate. The original wiki/431 v1 prescribed K3s + Argo + CRD + a Go operator. Phase −1 (business-panel-experts debate mode: Christensen + Taleb + Kim & Mauborgne + Drucker) rejected the thesis on two independent grounds: (a) wrong product shape — K8s vocabulary inside a personal-productivity tool; (b) fly.io retained in the critical path despite a four-phase-old user objection. The reshape eliminated ~2500 LOC of platform-engineering code (Argo install + Go operator + CRD plumbing), retired fly.io entirely, and introduced T2-A' (asyncio + SQLite) and T2-F (self-hosted signaling). See §5 Phase −1 retrospective.

§5 · Phase −1 strategic gate retrospective

wiki/431 v1 was the first plan ever submitted to the Phase −1 strategic gate after the 2026-05-18 update to MODE_Agent_Team.md. The verdict was RED on first run. The panel (Christensen + Taleb + Kim & Mauborgne + Drucker, debate mode) returned eight findings on the v1 thesis itself:

The eight Phase -1 findings (SG-1 ... SG-8)
  1. SG-1 HIGH (Christensen + Kim ERRC) — Wrong product shape. K8s + Argo + CRD + operator is platform-engineering infra inside a personal-productivity-tool positioning. Job-execution chart does not intersect solution-feature chart.
  2. SG-2 HIGH (all four frameworks convergent) — fly.io paid SaaS retained in the critical-path signaling despite a user objection four phases ago. Every Phase 4 code path that ships with fly grows the removal cost.
  3. SG-3 HIGH (Taleb + Drucker) — Five compounding new fragilities in one phase: Argo upstream, Argo EOL clock, Go toolchain, controller-runtime, and AdmissionWebhook with failurePolicy: Fail (downtime blocks CRD apply by design).
  4. SG-4 HIGH (Christensen + Kim) — Industry mis-import. K8s + Argo + operator is correct for orchestration-platform industry (Kubeflow, Flyte, Prefect). musu is not in that industry.
  5. SG-5 HIGH (Drucker) — Effectiveness vs efficiency: the 11-week plan was internally efficient but moved further from "5 working users by V23.5" than a 4-6 week lightweight version would.
  6. SG-6 MED (Drucker + Taleb) — Plan explicitly schedules future damage-management work ("plan v4.0 upgrade workstream") and ships anyway.
  7. SG-7 MED (all four) — All 17 prior Critic findings lived within the unquestioned thesis frame. Tech-only Critic structurally cannot ask "should this exist".
  8. SG-8 LOW (Kim) — K8s-vocabulary lint treats symptom, not cause; can't rename "300MB Argo OCI failed to pull" in user log files.

§8 · Process learnings

  1. Phase −1 strategic gate is load-bearing. First invocation caught two RED-level findings (wrong product shape, fly.io critical-path violation) and seven supporting HIGHs. Permanent adoption confirmed; will run on every master plan and every sub-WS that extends a master's thesis.
  2. Plan-as-spec Auditor zero-overlap with Critic. Validated across 3-4 invocations in Phase 4. Critic catches plan-shape; Auditor catches cross-section invariants. Both gates are necessary; neither subsumes the other. 500 LOC threshold confirmed via T2-D-mini calibration skip.
  3. Runtime smoke-tests beat AST parse. The T2-Z audit caught two HIGHs that AST + bash -n could not see: bench-windows.ps1 $Host parameter shadow (PowerShell automatic variable conflict) and K3s airgap-trim regex missing rancher/mirrored-* upstream prefix. Process doc PLAN_TEMPLATE_HEALTH_VERIFICATION_2026_05_19.md (FO-A1a-6) is the formal response — every new file Builder ships gets a runtime smoke alongside syntax check.
  4. Form-heavy editor LOC multiplier. T2-D-mini plan budget was ~580 FE; actual ~1170 FE (1.7x). For form-heavy editors in V23.5 estimates, bake in 1.7x multiplier; for pure form-list pages, 1.3x suffices.
  5. HTML closure docs work. The wiki/439 T2-D-mini HTML closure (per feedback-scribe-html-only) feels more navigable than Markdown for closure context. Single self-contained file, light+dark theming, expandable sections, inline SVG diagrams. Adopting HTML format for this final closure as well.
