## §9 The cross-layer dependency lesson

This is the headline finding for this closure. The F12 build-pipeline HIGH discovered by post-build Auditor surfaces a defect class that no plan-stage phase could catch: **cross-layer integration drift across 6 independent layers, each internally correct but globally incoherent**.

### §9.1 The 6 layers that had to agree

For the rendezvous role to actually boot on a user PC, six independent layers had to agree on file paths and contracts:

1. **SOURCE**: `musu-relay/src/signaling/user-server.ts` — Builder wrote this; entrypoint for rendezvous mode. ✅ correct in bf1c1a7.
2. **TYPESCRIPT BUILD**: `musu-relay/tsconfig.user-server.json` — Builder configured `include` = `["src/signaling/shared.ts", "src/signaling/user-server.ts"]`. ✅ correct in bf1c1a7.
3. **BUILD SCRIPT**: `musu-relay/installer/build-musu-backend.sh` — should compile (2) + stage outputs to install layout. ❌ **MISSING Step 3.f in bf1c1a7**. This is F12.
4. **SERVICE CONFIG**: `musu-relay/installer/openrc-musu-signaling.conf` — Builder wrote `command_args=/usr/local/lib/musu-signaling/dist/signaling/user-server.js` (references the path layer 3 should produce). ✅ correct in bf1c1a7 (but referencing a path that doesn't exist).
5. **INIT SHIM**: `musu-relay/installer/musu-init` — Builder wrote `rc-service musu-signaling start \|\| echo "WARNING ..."` per NEW-A4/OQ-A5. ⚠️ Failure-swallow worked exactly as designed — too well. Silenced the integration failure rather than surfacing it.
6. **INSTALLER ORCHESTRATION**: `musu-relay/installer/install-wsl2.ps1` Step 7.5 — atomic OpenRC service install. ✅ correct in bf1c1a7 (but installing a service file that points to a nonexistent path).

Each layer was internally correct. The defect was **cross-layer**: layers 3 and 4 had to agree on file path layout, and the build script was scoped to "only stage gateway dist" pre-T2F. Without explicit cross-layer verification, the bug was invisible to:

- **Critic** (Phase 1.5): reads plan text. Plan §6 line 537 listed the build-script requirement but Critic's job was correctness of the plan text, not implementation enforcement.
- **Plan-as-spec Auditor**: reads spec text. Could have flagged "checklist item unverified by spec acceptance criteria" but the spec acceptance #5 (`tsc --noEmit` green) tests compilation, not install layout. The miss was in the spec, not the Auditor's reading of it.
- **Builder** (Phase 3): wrote each layer correctly in isolation. Did not run a full install dry-run because the spec did not require one as a Builder MUST-do (acceptance #10 covered the LAN smoke test which requires real hardware — deferred).
- **signaling.test.ts**: tests wire protocol, not install layout.

Only the **post-build Auditor** with full repo state in hand could grep across all 6 layers and observe that the path `openrc-musu-signaling.conf` references is not produced by `build-musu-backend.sh`. This is exactly the defect class `[[feedback-plan-stage-auditor]]` memory anticipates — and validates a 4th time now: plan-stage phases catch text-level + spec-structural defects; post-build phase catches integration-reality defects.

### §9.2 Why the spec checklist was not enough

wiki/433 §6 line 537 reads:

> - [ ] Build script for user-PC tar uses `npm run build:user-server` + `npm ci --omit=optional` (F12)

This is a Builder MUST-do checklist item. The spec listed it. Builder shipped without implementing it. The unchecked box did not block bf1c1a7 because:

1. There is no automated gate that fails when a MUST-do is unimplemented — checklist enforcement is Auditor-by-inspection only.
2. The closest acceptance criterion (#5 `tsc --noEmit` green) passes WITHOUT the build script change — `tsc` doesn't care about install layout.
3. T1 require.cache contamination gate (acceptance #8) tests user-server.js loadability but Builder ran it against `./dist/signaling/user-server.js` (the dev build path), not against the install path `/usr/local/lib/musu-signaling/dist/signaling/user-server.js`.

The gap: **spec checklist items without explicit verification gates are advisory, not enforced**. Future plans crossing source + installer + build-pipeline layers should add an explicit "install-layout dry-run" acceptance criterion that runs `build-musu-backend.sh` under a scratch directory and validates the OpenRC config's `command_args` path actually resolves to a file.

### §9.3 The `|| echo` footgun

NEW-A4/OQ-A5's `\|\| echo` failure-swallow was specifically designed to keep the gateway up if signaling fails to start. The reasoning was sound: rendezvous failure should not boot-loop the gateway. But the implementation chose the **silent** failure mode (single echo line lost in journal noise) instead of the **visible** failure mode (multi-line diagnostic with status + ls outputs).

Phase 6 audit-fix retrofit the diagnostic block (§8.2) to retain non-fatal semantics while making failure observable. **Pattern lesson**: failure-swallow handlers should default to verbose diagnostics, not bare echos. A `WARNING` line in `/var/log/musu-init.log` next to 200 lines of normal boot output is invisible; an indented multi-line block with rc-service status + directory listings is not.

This sub-WS adds a 4th independent validation of the plan-stage Auditor memory, with a NEW twist: the plan-stage Auditor itself was insufficient to catch F12. The 5-phase architecture (Critic → plan-as-spec Auditor → Builder → post-build Auditor → audit-fix Builder) caught defects across distinct classes:

## §14 Lessons learned

- **5-phase agent-team chain produced zero shipped-code HIGH post-c59499e.** Plan-stage Critic caught 5 HIGH on plan text; plan-as-spec Auditor caught 3 HIGH on cross-section invariants (orthogonal to Critic); Builder shipped a plan-internally-correct first commit; post-build Auditor caught 1 NEW HIGH (F12) + 2 MED on cross-layer integration drift (orthogonal to all upstream phases); audit-fix Builder closed in single iteration. Each phase's HIGH set is structurally invisible to the others — strongest argument yet for keeping all 5 phases mandatory on sub-WS crossing source + installer + build-pipeline boundaries.

- **Plan-as-spec Auditor validated 4× now** but does NOT catch cross-layer integration drift. T2-F adds the new insight: plan-as-spec Auditor reads SPEC, not REPO. F12 was a repo-level invariant (`build-musu-backend.sh` does not stage the path `openrc-musu-signaling.conf` references) invisible to text-level review. Plan-as-spec Auditor remains valuable for what it catches (signature drift, naming mismatch, cross-section invariants in spec text) but is NOT a substitute for post-build Auditor.

- **Cross-layer integration drift is the post-build Auditor's reason-to-exist.** Six independent files (source + 2 tsconfigs + service config + build script + init shim + installer ps1) had to agree on install layout. Each was internally correct. Only `git ls-tree -r HEAD` + grep across `command_args` vs `cp` destinations could surface the gap. Post-build Auditor is non-negotiable.

- **Spec checklist items without explicit verification gates are advisory, not enforced.** wiki/433 §6 line 537 listed the F12 build-script requirement; Builder shipped without implementing it; no automated gate failed. Future plans crossing source + installer + build-pipeline layers should add explicit "install-layout dry-run" acceptance criterion (run build script under scratch dir; validate every service config's `command_args` path resolves to a real file in staged output).

- **The `|| echo` failure-swallow is a footgun pattern.** NEW-A4/OQ-A5 chose silent failure for non-fatal semantics — by design. But the implementation hid F12 from users (and from any Auditor running smoke tests against the build pipeline). Phase 6 retrofit a verbose diagnostic block; future failure-swallow handlers should default to verbose diagnostics from spec-time.

- **Single audit-fix iteration is the expected steady-state.** Phase 5 Auditor produced an actionable diagnostic (F12 named the missing build-script step + the silent failure mode); Phase 6 Builder shipped both fixes in one commit (c59499e) with no second-loop ambiguity. When the Auditor's HIGH includes file:line + the resolution path is unambiguous, the fix-loop is one commit. Multiple loops indicate Auditor diagnostic quality drift or Builder type mismatch — both warrant escalation per `MODE_Agent_Team.md` "Builder loops twice on same audit finding".

- **`signaling.test.ts` byte-identical pre/post-refactor is a real C-T2F-H5 win.** The handler lift from server.ts:409-516 to shared.ts as `_wssConnectionHandler` is exactly the kind of refactor that historically breaks tests in subtle ways (closure-captured state, import-cycle introductions, type-narrowing regressions). The mechanical gate of "tests pass UNMODIFIED" caught zero issues — confirming the lift was indeed byte-for-byte. Worth repeating as a pattern for future server.ts-class refactors.

- **OQ3 cloud /metrics regression was a `signaling.test.ts` coverage blind spot.** The post-refactor `peers_total: 0, rooms_total: 0` hard-coding slipped through C-T2F-H5 because signaling.test.ts does not exercise `/metrics`. Plan-as-spec Auditor INFO-A2 noted `_totalPeers`/`_totalRooms` may be dead exports — close but no flag. Post-build Auditor caught it. Lesson: invariants on routes not covered by existing tests should get explicit Builder MUST-do hand-verification gates (per NEW-A1 §6 line 545 which DID call for this — Builder shipped without running `curl /metrics`).
