## 2. Gate-conditional branching

### 2.1 If `gate_decision = alpha` (≥70% success on 5 hosts)

The α-path continues. V23.3 work focuses on hardening + replacing v21 surfaces inside the WSL2 substrate. See §3 below.

### 2.2 If `gate_decision = beta` (<70% success)

α-path retreat. V23.3 plan rewrites the install substrate. See §4 below. Most of §3 becomes moot.

### 2.3 If `gate_decision = abort_payload_inconsistent` OR `insufficient_diversity`

Re-run B4c with fixes. No V23.3 work scheduled until B4c records alpha or beta.

## 3. V23.3 task list — α-path

Candidates ranked by dependency-blocking impact + scope. The bracketed estimate is engineering-effort, not wall-clock.

### 3.1 Tier-A (must ship before V23.3 closure)

#### A1. musu-bridge as K3s Pod inside WSL distro (~1-2 weeks)
Replaces v21's native musu-bridge process with a K3s Deployment running inside the imported tar. v21 pathway stays alive in parallel until V23.5 cuts the native path. Blocks: V23.4 Tauri UI assumes bridge is K8s-managed.
- Const III: probable (new CRD or ConfigMap schema for bridge config)
- Const VI: probable (capacity benchmark — does K3s-managed bridge match v21 latency?)
- Const VII: feature-branch + main-merge gate

#### A2. T1.9 wrtc factory wiring (~3-5 days)
Replaces `src/gateway/main.ts` `pcFactory.create()` TODO stub with the actual `@roamhq/wrtc` `RTCPeerConnection` constructor. Currently throws on first WebRTC offer; gateway can connect to signaling but cannot accept peer sessions. This was master-plan-deferred but unblocks B4c success criteria (gateway "connects + accepts a visitor connection" is part of α-path validation in V23.3+).
- Const III: NO
- Const VI: NO
- Const VII: feature-branch gate

#### A3. Body-identity HMAC structural refactor (~2-3 days)
Per wiki/377 §7 lesson 5: extract a single `signAndPost(body)` helper used by both `recordOutcome` and `emitInstallCompleted` so the rawBody-shared-for-signing-and-POST invariant is structural, not Builder-discipline. Refactor target: musu-relay/src/gateway/client.ts + main.ts.

## 4. V23.3 task list — β-path (if `gate_decision = beta`)

If B4c lands β, V23.3 is a substrate pivot. Tier-A above becomes mostly moot (musu-bridge as K3s Pod assumes K3s works).

### 4.1 β-path decision sub-options (from wiki/375 §5.5)

#### β-a. Fix specific failure + re-run B4c
If dominant failure is one mode (e.g., K3s `--snapshotter=native` was wrong flag), patch B4a's openrc-k3s.conf + rebuild + re-run B4c. Cost: ~1 week elapsed.

#### β-b. Debian-slim base pivot
Replace Alpine base in `build-musu-backend.sh` with `debian-slim`. Rebuild tar, re-run B4c. K3s on Debian-WSL2 is more battle-tested than on Alpine-WSL2.
- Cost: ~2-3 weeks
- Risk: image size grows (~50-80 MB)

#### β-c. Drop α-path, rebase on β (Docker-Desktop-hard-require OR drop K3s entirely)
If WSL2 itself is unsupportable for V23.2 prosumer audience:
- β-c1. Docker-Desktop-hard-require (re-introduces V23.0 rejected dependency; needs license-cost messaging for non-personal use)
- β-c2. Drop K3s entirely; ship simpler container substrate (containerd-direct, podman-WSL, or native-Python like v21)
- Cost: ~6-8 weeks. Effectively V23.5 work pulled forward.

### 4.2 Decision matrix

| Failure pattern | Recommended β-sub |
|---|---|
| K3s never readies on 3+ hosts of same Win11 build | β-a (K3s flag retry) |
| K3s never readies on mixed Win11 builds | β-b (Debian-slim) |
| wsl --import fails on ≥2 host classes | β-c (drop α-path) |
| /issue_install_key 401s on all hosts (HMAC broken in prod) | NOT a β trigger — it's a B1 regression; fix forward |

## 7. Parallelization map

Once V23.3 plan-mode starts, these can run as parallel sub-workstreams:

```
Tier-A1 (musu-bridge K3s Pod) ─ Tier-A2 (wrtc factory) ─ Tier-A3 (HMAC refactor)
        │                              │                          │
        └── all 3 can run in parallel (no overlap in modified files)
             │
             ▼
        Tier-B candidates (most are independent)
             │
             ▼
        V23.3 closure
```

A1 + A2 + A3 are file-disjoint (A1 is bridge code + K3s manifest; A2 is wrtc-factory.ts; A3 is client.ts + main.ts). All three can spawn parallel agent-team sub-workstreams.

B1 (SOURCE_DATE_EPOCH) blocks on A1 finishing (need a working tar baseline). B2 (install_attempt) blocks on Const III review. B3-B6 are independent.
