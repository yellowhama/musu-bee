§7 Lessons learned (panel reshape validation)

§7.1 Phase -1 strategic gate was load-bearing

Operator's initial draft was "Big-bang Rust rewrite — Python 25k 통째로 Rust로. 4-6주. 동안 musu = down." Panel YELLOW with 4 HIGH reshaped this to R-fast (~2 weeks → JTBD ships) → R-cleanup (~2-4 weeks parallel to operator dogfood). Without the panel gate, V24 would have shipped a multi-week down-window and missed acceptance items because operator's daily workflows would have been blocked.

§7.2 LOC estimates undershot by 2-5×

Every sub-WS LOC estimate was low: R5 200→990, R6 600→4,180, R3 estimated low but landed ~1,720, R4 1,100→1,899. Master plan §5 LOC budgets should default to 3× the initial estimate; better yet, abandon LOC budgets and bound by test-count + acceptance items.

§7.3 Researcher F1 frame corrections are critical

R3 task brief originally said "replaces /api/control/*" — wrong frame. Researcher F1 caught this BEFORE Planner committed to wrong design. Same pattern in R4 where Researcher F2 found ~1,500 LOC of Python musu-indexer was unrelated session/spy/PTY tooling. Frame corrections at Phase 0 save Phase 3 rebuild.

§7.4 Dual-audit catches what single-audit misses

R6 dual-Critic returned 4+3=7 HIGH; dual-Auditor returned 6 HIGH. Single-Critic/Auditor on R3+R4+R5 returned 3 + 1 + 1 NEW HIGH respectively. Dual-audit on auth-touching / one-way blast radius ranges (R6) is mandatory and pays off; single is sufficient for MED-risk non-auth.

§7.5 Operator gates work

R8 (4060Ti install/E2E) was operator-bound and PASSED with 7 audit_log non-testclient rows. §9.12 ungameable metric will operator-attest within 7 days post-R10. The orchestrator's role is to ship code; operator's role is to attest acceptance. This separation is the load-bearing Goodhart firewall.
