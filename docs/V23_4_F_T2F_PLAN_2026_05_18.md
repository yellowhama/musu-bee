## §2 Implementation

### §2.1 `user-server.ts` entrypoint (NEW)

```typescript
// musu-relay/src/signaling/user-server.ts
//
// V23.4 T2-F (wiki/433): user-PC self-hosted signaling rendezvous.
// Differs from src/signaling/server.ts (cloud variant on signaling.musu.pro):
//   - NO telemetry router mount
//   - NO checkTelemetryAuthBootConfig
//   - Binds 0.0.0.0:MUSU_SIGNALING_PORT (default 9900) for LAN reach
//   - Same WS protocol so peers interoperate with cloud server.ts

import http from "http";
import { WebSocketServer } from "ws";
import express from "express";
import {
  validateToken, rooms, _resetAuthState, _validationCache,
  _wssConnectionHandler,
} from "./server";

const PORT = parseInt(
  process.env.PORT || process.env.MUSU_SIGNALING_PORT || "9900", 10,
);

const app = express();
app.set("trust proxy", 1);

const _startTime = Date.now();
app.get("/health", (_req, res) => {
  res.json({ status: "ok", rooms_active: rooms.size, mode: "user-server" });
});

app.get("/metrics", (_req, res) => {
  res.json({
    uptime_seconds: Math.floor((Date.now() - _startTime) / 1000),
    rooms_active: rooms.size,
    peers_active: Array.from(rooms.values()).reduce((n, r) => n + r.size, 0),
    mode: "user-server",
  });
});

const server = http.createServer(app);
const wss = new WebSocketServer({ noServer: true });

server.on("upgrade", (req, socket, head) => {
  if (req.url !== "/signaling") { socket.destroy(); return; }
  wss.handleUpgrade(req, socket, head, (ws) => wss.emit("connection", ws, req));
});

wss.on("connection", _wssConnectionHandler);

if (require.main === module) {
  server.listen(PORT, "0.0.0.0", () => {
    console.log(`[user-server] listening on 0.0.0.0:${PORT} (rendezvous mode)`);
  });
}

export { app, server, wss, validateToken, rooms, _resetAuthState };
```

**Prerequisite refactor in server.ts**: extract `wss.on("connection", async (ws) => {...})` at `:409-516` to module-level `function _wssConnectionHandler(ws: WebSocket)` and add to existing `export { ... }` block. Zero behavior change.

## §9 Open questions for Critic

Most OQs resolved by orchestrator. Remaining items for Phase 1.5 `system-architect` Critic:

1. **Q1 (architecture)**: `user-server.ts` separate file vs `MUSU_HOST_MODE=user` env-flag conditional inside server.ts. Plan picks separate-file for clarity (no shared mutable state across host modes; telemetry router import becomes conditional/dead-code-strippable). Critic may challenge — if env-flag wins, scope drops ~30 LOC but couples futures changes to one file and increases risk of accidental telemetry mount on user PC.

2. **Q2 (idempotency of `-MakeMeRendezvous` takeover)**: When user's PC #1 dies and they run `-MakeMeRendezvous` on PC #2 (previously a peer), plan documents: (a) re-install OpenRC service idempotently via tmp+mv, (b) rewrite `MUSU_SIGNALING_URL=http://localhost:9900`, (c) restart musu-gateway, (d) peers NOT auto-notified (operator manually re-runs with `-RendezvousUrl http://<PC-2>:9900`). Critic may push for mDNS auto-discovery (V23.5 scope per O4-b). Also: drain semantics — plan assumes PC #2's `rooms: Map` is empty (was peer; peer's signaling server never ran) so flip is no-op for state. Critic should validate this invariant holds.

3. **Q3 (server.ts refactor risk)**: extracting `wss.on("connection", ...)` to named exported function — verify no closure-captured state moves silently. All references inside the handler are module-scope (validateToken, rooms, joinRoom, send, etc.) — confirmed safe to share between cloud and user server processes (each process has independent module state).
