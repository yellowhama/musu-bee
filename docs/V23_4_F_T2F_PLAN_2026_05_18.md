### §2.1 `user-server.ts` entrypoint (NEW)

```typescript
// musu-relay/src/signaling/user-server.ts
//
// V23.4 T2-F (wiki/433): user-PC self-hosted signaling rendezvous.
// Differs from src/signaling/server.ts (cloud variant on signaling.musu.pro):
//   - NO telemetry router mount
//   - NO checkTelemetryAuthBootConfig
//   - NO import of ./server (would transitively pull telemetry.ts → better-sqlite3) — C-T2F-H1 fix
//   - Imports ONLY from ./shared which has zero telemetry/better-sqlite3 imports
//   - Binds 0.0.0.0:MUSU_SIGNALING_PORT (default 9900) for LAN reach
//   - Same WS protocol so peers interoperate with cloud server.ts

import http from "http";
import { WebSocketServer } from "ws";
import express from "express";
import {
  rooms, _resetAuthState, _validationCache,
  _wssConnectionHandler,
} from "./shared";  // OQ-CRIT-1: NOT "./server"
// NOTE (NEW-A3 / OQ-A3): validateToken is NOT imported here. It is invoked
// from inside _wssConnectionHandler (HELLO message handler) which lives in
// shared.ts. user-server.ts therefore neither imports nor re-exports it.
// NOTE (post-Audit): user-server /metrics intentionally omits peers_total,
// rooms_total, circuit_breaker (those live in cloud server.ts /metrics).
// user-server is a lightweight rendezvous; full observability stays on cloud variant.

const PORT = parseInt(
  process.env.PORT || process.env.MUSU_SIGNALING_PORT || "9900", 10,
);

const app = express();
app.set("trust proxy", 1);

const _startTime = Date.now();
app.get("/health", (_req, res) => {
  res.json({ status: "ok", rooms_active: rooms.size, mode: "user-server" });
});

app.get("/metrics", (_req, res) => {
  res.json({
    uptime_seconds: Math.floor((Date.now() - _startTime) / 1000),
    rooms_active: rooms.size,
    peers_active: Array.from(rooms.values()).reduce((n, r) => n + r.size, 0),
    mode: "user-server",
  });
});

const server = http.createServer(app);
const wss = new WebSocketServer({ noServer: true });

server.on("upgrade", (req, socket, head) => {
  if (req.url !== "/signaling") { socket.destroy(); return; }
  wss.handleUpgrade(req, socket, head, (ws) => wss.emit("connection", ws, req));
});

wss.on("connection", _wssConnectionHandler);

if (require.main === module) {
  server.listen(PORT, "0.0.0.0", () => {
    console.log(`[user-server] listening on 0.0.0.0:${PORT} (rendezvous mode)`);
  });
}

export { app, server, wss, rooms, _resetAuthState };
// NEW-A3 / OQ-A3: validateToken dropped from exports (unused at this surface).
```

### §2.2 OpenRC service config (NEW)

```sh
#!/sbin/openrc-run
# musu-relay/installer/openrc-musu-signaling.conf
# V23.4 T2-F (wiki/433). Installed ONLY on rendezvous PC at /etc/init.d/musu-signaling.

name="musu-signaling"
description="musu self-hosted WebRTC signaling rendezvous (V23.4 T2-F)"

command="/usr/bin/node"
command_args="/usr/local/lib/musu-signaling/dist/signaling/user-server.js"

command_user="root"
command_background=true
pidfile="/run/${RC_SVCNAME}.pid"
output_log="/var/log/musu-signaling.log"
error_log="/var/log/musu-signaling.log"

depend() {
    need net localmount
}
```

**Boot model (per OQ-CRIT-4 resolving Critic M1)**: musu-signaling is **NOT** added to the default runlevel. Following the existing musu-gateway pattern, the `musu-init` script (already shipped in the `musu-backend` tar) starts musu-signaling AFTER `rc-service musu-gateway start` returns success AND `MUSU_IS_RENDEZVOUS=true` is set in `gateway.env`. This avoids the race where signaling tries to bind before networking is fully up and avoids stuck-boot on non-rendezvous peer PCs that have the service file installed but should not run it.

**§2.3.4 New step 7.5 — install musu-signaling OpenRC service file if rendezvous** (~10 LOC):

**§2.3.5 musu-init edit** (NEW; per OQ-CRIT-4) — ~5 LOC patch to `musu-relay/installer/musu-init` (source file; built into `musu-backend.tar` by `build-musu-backend.sh`; NEW-A7 wording fix). After successful `rc-service musu-gateway start`, read `MUSU_IS_RENDEZVOUS` from `/etc/musu/gateway.env`; if `true`, run `rc-service musu-signaling start`. Log result. Idempotent (already-running case returns 0).

```sh
# musu-init (after existing line 142 'rc-service musu-gateway start'):
# NEW-A4 / OQ-A5: musu-init runs with `set -e` (line 22). A bare
# `rc-service musu-signaling start` failure would propagate and stop the
# init script, leaving the user without the working gateway. The `|| echo`
# pattern explicitly swallows the failure with a WARNING log line so the
# gateway stays up even if signaling fails to bind.
if [ "${MUSU_IS_RENDEZVOUS:-false}" = "true" ]; then
    rc-service musu-signaling start || echo "musu-init: WARNING musu-signaling failed; gateway still up"
fi
```

### §2.7 `installer/test-signaling-smoke.ps1` (NEW; per OQ-CRIT-3 resolving C-T2F-H4)

2-PC LAN handshake validator. Uses **tcpdump** as the instrument (NOT dnsmasq logs — dnsmasq doesn't capture all DNS pathways; tcpdump on the WSL interface is the correct probe). Requires root in WSL distro. Approx 60 LOC.

```powershell
# musu-relay/installer/test-signaling-smoke.ps1
# V23.4 T2-F (wiki/433): 2-PC LAN handshake validator.
# Per OQ-CRIT-3 (resolves Critic C-T2F-H4).
#
# Prereq: 2 PCs already paired via musu installer.
#   - PC-A:  installed with -MakeMeRendezvous
#   - PC-B:  installed with -RendezvousUrl http://<PC-A>:9900

Param(
  [Parameter(Mandatory=$true)][string]$PeerAddr,   # PC-A IP:9900 from PC-B's perspective
  [int]$CaptureSeconds = 30
)

Import-Module "$PSScriptRoot/Musu-Common.psm1" -Force

function Step($n, $msg) { Write-MusuInfo "[Step $n] $msg" }

# (a) Both PCs paired? (operator-asserted; we just print the assumption)
Step 1 "Assuming PC-A + PC-B already paired via installer."

# (b) On PC-A: musu-signaling service running
Step 2 "Verifying musu-signaling service is running on PC-A (must be run on PC-A)."
$status = wsl.exe -d musu -- rc-service musu-signaling status 2>&1
if ($status -notmatch "started") {
  Write-MusuErr "musu-signaling not started on this PC. Run: wsl -d musu -- rc-service musu-signaling start"
  exit 1
}
Write-MusuOk "musu-signaling status: started"

# (c) On PC-B: export MUSU_SIGNALING_URL and restart musu-gateway
Step 3 "On PC-B run BEFORE this script: setx MUSU_SIGNALING_URL http://$PeerAddr (then restart musu-gateway)."

# (d) tcpdump for $CaptureSeconds — must show ZERO traffic to signaling.musu.pro
Step 4 "Running tcpdump for $CaptureSeconds seconds — capturing any traffic to signaling.musu.pro."
$out = wsl.exe -d musu -u root -- timeout $CaptureSeconds tcpdump -i any -nn host signaling.musu.pro 2>&1
$hits = ($out | Select-String -Pattern "signaling\.musu\.pro" -AllMatches).Matches.Count

# (e) PASS/FAIL summary
Step 5 "Result"
if ($hits -eq 0) {
  Write-MusuOk "PASS: zero packets to signaling.musu.pro during $CaptureSeconds s capture."
  exit 0
} else {
  Write-MusuErr "FAIL: $hits packets to signaling.musu.pro observed. Self-host not effective."
  Write-Host $out
  exit 1
}
```

**Why tcpdump and not dnsmasq**: dnsmasq logs only the DNS resolutions that go through dnsmasq itself. WSL2 distros may bypass dnsmasq for `/etc/hosts` entries, systemd-resolved stub paths, or Windows host DNS (mirrored mode). tcpdump on `-i any` observes the actual L3/L4 traffic and is the authoritative instrument. Root is required inside the musu WSL distro (already root for service management; acceptable).
