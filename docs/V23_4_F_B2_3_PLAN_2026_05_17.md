## §1 Scope

Wrap `_db.prepare(...).run(...)` in `try/catch` on all 4 telemetry routes. Currently DB-write exceptions bubble to Express default error handler (HTML stack trace in dev / generic 500 in prod). Replace with structured `{ error: "database write failed" }` JSON 500 + structured `console.error` log for Fly log aggregation.

Per Critic C3 (wiki/425 §11): F-B2-3 changes a (small) public API surface on legacy HMAC-authed routes. Builder MUST regression-scan existing 218 tests for any case asserting on the pre-F-B2-3 error shape BEFORE landing. Per Critic C11: the `return` after catch is load-bearing (avoids double-response "Cannot set headers after sent" error).

Single `quality-engineer` Auditor (per wiki/425 §2 — defensive-coding scope, no new auth surface or secret).

### 2.2 Pattern (applied uniformly to each route)

```typescript
try {
  _db.prepare(`INSERT INTO ... VALUES (?, ?, ...)`).run(/* params */);
} catch (err) {
  console.error(
    `[telemetry] /<route>: db write failed: ${err instanceof Error ? err.message : String(err)}`,
  );
  res.status(500).json({ error: "database write failed" });
  return; // C11: load-bearing — without this, falls through to res.status(204).end() and Express throws "Cannot set headers after they are sent"
}
res.status(204).end();
```

Each route gets:
- `try { ... } catch (err) { ... return; }` wrapping the existing `_db.prepare(...).run(...)` call (or `openDb()` if Builder follows F-B2-1 pattern).
- Route name hardcoded in the log line for grep-ability (`/install`, `/install_attempt`, `/nat_pierce`, `/agent_spawn`).
- `return` after the catch.

### 2.3 NO production code beyond the 4 try/catch wraps

Do not refactor handler structure, do not extract a shared helper, do not add new middleware. The fix is 4 localized wraps + return statement.

## §4 Test plan

`musu-relay/tests/telemetry-db-failure.test.ts` — NEW file, 4 cases, one per route. Pattern per wiki/425 §5.3 + Critic C11 double-response guard:

| # | Name | Setup | Assertions |
|---|---|---|---|
| TDF-1 | `/install` DB-write failure → 500 + log | Mock `openDb()` (or test-only swap via existing pattern) so `.prepare(...)` throws on the install INSERT. POST valid signed body to `/install`. | Status 500; `res.body.error === "database write failed"`; `console.error` called with `/install: db write failed: ...` substring (jest spy); NO "Cannot set headers after they are sent" error in test output |
| TDF-2 | `/install_attempt` DB-write failure → 500 + log | Same pattern but for install_attempt route (no auth). | Same assertions; route name in log is `/install_attempt` |
| TDF-3 | `/nat_pierce` DB-write failure → 500 + log | Same pattern, HMAC-authed | Same assertions; route name `/nat_pierce` |
| TDF-4 | `/agent_spawn` DB-write failure → 500 + log | Same pattern, HMAC-authed | Same assertions; route name `/agent_spawn` |

**Mock strategy options** (Builder picks):

(a) **Override `openDb()` per-test** via jest.spyOn or a test-only setter exported from telemetry.ts. Risk: pollutes module state.

(b) **Manual SQL constraint trigger**: insert a row that violates a UNIQUE constraint (e.g., if any route has a UNIQUE column, insert dup). But install_attempt doesn't have a meaningful UNIQUE constraint, so this is fragile.

(c) **better-sqlite3 readonly mode**: open DB read-only, then any write throws. Requires re-opening DB which conflicts with existing test fixtures.

(d) **Direct method mock**: jest.spyOn(_db, "prepare").mockImplementationOnce(() => { throw new Error("simulated DB failure") }). Cleanest; needs `_db` accessible from test (via `_getDbForTests()` existing export).

Builder defaults to **(d)** unless wiki/408 Builder phase finds a reason to deviate. Document choice in wiki/428 closure.

### 4.1 Console.error spy

Use `jest.spyOn(console, "error").mockImplementation(() => {})` in beforeEach; assert call shape; `mockRestore` in afterEach. Pattern mirrors install-attempt.test.ts:342-367 console.warn spy for T13.

### 4.2 Double-response guard (per Critic C11)

In each test, after asserting 500 + body, also assert that the test output does NOT contain `"Cannot set headers after they are sent"` (jest captures process stderr — check via spy on stderr or by ensuring the test response is exactly one response, not a stream of two). Simplest: a `console.error` spy will catch Express's error logging if double-response happens.

**Test count delta**: 224 → 228 (+4 net; supersedes wiki/425 §5.3 "+4 to 225" because F-B2-1 audit-fix1 already bumped baseline from 221 to 224).
