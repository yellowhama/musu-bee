SQLite 분산 합의(Split-Brain) 한계를 피하기 위해 ~/.musu/tasks/{id}.json에 상태를 직렬화하는 단일 진실원(SSOT) 모델을 수립했습니다. Go 크롤러 소스를 모노레포 내부의 musu-brainai로 이전하여 컴파일 및 REST API(tasks, graph, files)를 탑재하고, Rust 실행기의 이벤트 핸들러와 연동하여 실시간 동기화를 달성했습니다.

3대 아키텍처 성과 평가 (Architectural Breakdowns)

1. 분산 SQLite 스플릿 브레인 완벽 회피 (Database-Free State Tracking)

  • 문제점: 기기 간 SQLite 동기화 시 네트워크 불안정으로 인한 데이터 잠금(Locking) 및 이력 분기 충돌(Split-Brain) 발생 가능성 존재.
  • 해결책: 개별 태스크의 상태를 독립적인 JSON 파일(~/.musu/tasks/{id}.json)로 영구 저장하는 플랫 파일 SSOT 구조로의 전환. Originator 기기가 파일의 독점 쓰기 권한을 보유하고, 타겟 실행기는 단순 단방향 HTTP 콜백 및 JSON 병합 업데이트만 수행하여 복잡한 분산 분기 합의 알고리즘 없이 동기화 문제를 회피함.

2. 가상 네트워크 컨텍스트 전송 통합 (W14 Context Transport Baseline)

  • 문제점: 타 노드로의 태스크 위임 시 원격지 기기에 최신 소스 코드나 지식 파일 컨텍스트가 부재하여 발생하는 오동작 위험.
  • 해결책: musu-brainai 내부에 안전한 파일 관리 API(/api/files/*) 및 경로 검증(Path traversal guard)을 선결 도입하여, 향후 원격 위임 시 작업 공간을 경량 압축(ZIP) 전송하고 결과를 수집할 수 있는 인프라 기반 마련.

3. 단순 인덱서에서 "다이내믹 액터 성단"으로의 도약 (Galaxy V2 UI/UX)

  • 문제점: 이전 Galaxy V1 UI는 문서 및 키워드 연관성만 보여주어 실시간 Fleet 상태 파악이 불가능했음.
  • 해결책: D3.js 포스 레이아웃 대시보드를 확장하여 지식 문서 외에 실시간 태스크 노드(Green/Yellow/Red/Gray), 물리 PC 기기 노드(Purple), 회사/프로젝트 노드(Pink)를 실시간 링크 관계로 시각화.
