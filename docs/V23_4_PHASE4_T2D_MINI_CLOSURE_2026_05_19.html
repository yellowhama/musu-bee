§2 · C1 HIGH — the load-bearing Critic finding

Critic flagged that FormStep.step_id (used as React list key) was being conflated with agent_id (the identity backend uses to dispatch). The encoder would have shipped a workflow spec where every agent ID was "step-1" / "step-2" — none of which exist in any company's agent roster — and the executor would have silently failed every wake call.

// FormStep in v1 plan
type FormStep = {
  step_id: string,    // "step-1" — used as both React key AND agent dispatch id
  agent_id: string,
  prompt: string,
  depends_on: string[]
}

// Encoder emits:
agents: steps.map(s => ({ id: s.step_id, type: s.agent_id }))
                          // ↑ "step-1" leaks to backend

// workflow_executor.py:285
await enqueue_wake(agent_id=node.id)   # tries to wake "step-1" — no such agent
# silent dispatch failure, workflow stalls
// FormStep in v2 plan (Critic-corrected)
type FormStep = {
  reactKey: string,   // UUID — React list key ONLY, never sent to backend
  agent_id: string,   // e.g. "writer" — real dispatch identity
  prompt: string,
  depends_on: string[]  // references other agent_ids
}

// Encoder emits:
agents: steps.map(s => ({ id: s.agent_id, type: s.agent_id }))
                          // ↑ "writer" reaches backend correctly

// workflow_executor.py:285
await enqueue_wake(agent_id=node.id)   # wakes "writer" — agent exists, dispatched OK
Why this matters This is exactly what Phase 1.5 Critic + agent-team mode is for. A text-reading reviewer caught a runtime-only bug at plan stage, before Builder wrote a line. Cost: one Critic invocation. Saved: an entire audit-fix cycle plus a broken-workflow user report.

A1 detail — missing GET /workflows/{id}

Plan v2 specified the editor route loads existing workflow state via GET /api/companies/{cid}/workflows/{wfid}. Builder implemented the frontend fetch + state hydration, but did not add the backend route. Edit page hit 404 on first open. Fixed in e4ff17a:

# musu-bridge/app/workflow_routes.py
@router.get("/companies/{company_id}/workflows/{workflow_id}")
async def get_workflow(company_id: str, workflow_id: str):
    spec = await workflow_store.fetch(company_id, workflow_id)
    if spec is None:
        raise HTTPException(404, "workflow not found")
    return spec

Plus test t14b in test_workflow_routes.py covering 200 + 404 paths.

A2 detail — Playwright mock URL pattern

e2e spec used page.route('**/workflows', ...) but the actual frontend call hits /api/companies/[cid]/workflows. The mock never intercepted; the test relied on real backend (intermittent flake). Fixed pattern:

await page.route('**/api/companies/*/workflows*', route => route.fulfill({
  status: 200,
  contentType: 'application/json',
  body: JSON.stringify(mockWorkflow)
}));

Playwright run now stable at 18.3s, 1/1 pass.
