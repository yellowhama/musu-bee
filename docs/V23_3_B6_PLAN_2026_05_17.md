## §1 Scope

B6 makes `build-musu-backend.sh` produce a **byte-identical** `musu-backend.tar` when invoked twice from the same git SHA on the same Alpine WSL2 build host. The Const VI experiment in wiki/379 §4 phrases the bar as "sha256(tar) identical across 2 clean builds 1 week apart", but Phase 0 research (see §11.F4) shows the 1-week criterion conflates two distinct determinism layers:

- **Layer A — recipe determinism** (SDE + tar flags + native-compile path fixes): achievable in V23.3 B6 within scope.
- **Layer B — registry-content determinism** (apk repo snapshot + npm registry cache replay): out of scope for V23.3 B6; partial mitigation via exact-version apk pinning, full mitigation deferred to V23.5 reproducible-build attestation per wiki/379 §1.3.

B6 therefore commits to a **layered Const VI verdict** (§7.3): intra-hour same-host byte-identity is the V23.3 B6 acceptance bar; 1-week-apart byte-identity is recorded as a follow-on tracked under V23.5.

### 1.1 Why this scope split

Phase 0 Researcher Finding F4 (apk repo content drift) is genuinely outside the scope of SOURCE_DATE_EPOCH:

- Alpine 3.19 main/community repos receive rolling security backports. Two `apk add nodejs openrc alpine-base` runs 1 week apart can resolve to different rebuilds of the same Alpine 3.19 line.
- SDE controls timestamps inside the build process; it does not control what bytes the upstream Alpine mirror serves on a given day.
- The fix is either (a) exact-version pinning via `apk add nodejs=20.10.0-r0` syntax (partial — catches version drift but not same-version rebuilds), or (b) a self-hosted apk snapshot mirror (full — costs ~50MB in repo or LFS).

B6 commits to (a) as the V23.3 deliverable; (b) is flagged as a V23.5 follow-on with cost estimate captured in §10.

### 1.2 What this plan does NOT cover (forward-pointers)

- **V23.5**: full apk snapshot mirror or vendored `*.apk` cache; K3s artifact vendoring; reproducible-build attestation (in-toto / SLSA).
- **V23.4+**: rebuilding the `manifest.yaml` k3s sha256 placeholders into a more elaborate vendoring scheme (B6 lands the simple checksum-enforcement here; full vendoring deferred).
- **NOT changed**: musu-bridge K3s migration (A1.*), gateway WebRTC (A2 — already landed), HMAC body-identity (A3 — already landed), install_attempt telemetry (B2 — separate).

#### S0.a — Add GNU tar + binutils to build-host sanity checks (F6, F3)

```bash
#   - GNU tar required for --sort=name --mtime=@SDE --format=pax flags.
#     BusyBox tar lacks these; the canonical reproducible-builds.org
#     recipe needs GNU. Run: apk add tar

if ! tar --version 2>/dev/null | head -1 | grep -q "GNU tar"; then
  echo "FATAL: 'tar' is not GNU tar (need GNU for --sort/--format=pax). Run: apk add tar" >&2
  exit 1
fi
```

```bash
# manifest.yaml. Falls back to floating versions only if manifest is missing
```

#### S3.b — CFLAGS for node-gyp path normalization (F3)

Insert immediately before line 174 (the first `npm ci`):

```bash
# V23.3 B6 (wiki/392 F3): node-gyp invokes gcc to compile better-sqlite3.
# gcc embeds the build directory's absolute path into DWARF debug info
# (.debug_info section) and into __FILE__ macro expansions. -ffile-prefix-map
# rewrites those at compile time so the resulting .node binary is path-
# independent. SOURCE_DATE_EPOCH alone does NOT fix this.
#
# Critic C-B6-M3: cover all THREE directories npm/node-gyp run in across
# the build (in order of encounter):
#   1. $SOURCE = $SCRIPT_DIR/.. (musu-relay source dir) — first npm ci at line 174
#   2. $STAGE_NM_PARENT (per-build temp under STAGING) — second npm ci at line 184
#   3. $STAGING = $SCRIPT_DIR/../.build-stage — where .node binaries land
SOURCE_DIR="$(cd "$SCRIPT_DIR/.." && pwd)"
STAGE_NM_PARENT_PATH="$STAGING/usr/local/lib/musu-gateway/node_modules-staging"
PREFIX_MAPS="-ffile-prefix-map=${SOURCE_DIR}=/src -ffile-prefix-map=${STAGE_NM_PARENT_PATH}=/stage -ffile-prefix-map=${STAGING}=/build -fdebug-prefix-map=${SOURCE_DIR}=/src -fdebug-prefix-map=${STAGE_NM_PARENT_PATH}=/stage -fdebug-prefix-map=${STAGING}=/build"
export CFLAGS="${PREFIX_MAPS} ${CFLAGS:-}"
export CXXFLAGS="${PREFIX_MAPS} ${CXXFLAGS:-}"
```

#### S3.c — Strip native binaries after install (F3)

Insert immediately after line 185 (`mv "$STAGE_NM_PARENT/node_modules" ...`):

```bash
# V23.3 B6 (wiki/392 F3): strip gcc-embedded metadata from compiled .node
# binaries. -ffile-prefix-map above handles paths inside DWARF; strip + objcopy
# remove the .comment section (gcc version string) and unneeded symbols
# (which would otherwise contain absolute paths from the link line).
echo "[3.b/10] Strip native ELF metadata from compiled .node binaries"
find "$STAGING/usr/local/lib/musu-gateway/node_modules" -name "*.node" -type f -print0 | \
  while IFS= read -r -d '' nodebin; do

    # intermediate sha audits clean.

  done
```

#### S8 — Reproducible tar invocation (F1, F10, F11)

Replace line 268 with the reproducible-builds.org canonical recipe:

```bash
# V23.3 B6 (wiki/392 F1): reproducible tar. Citations:
#   - reproducible-builds.org/docs/source-date-epoch/
#   - reproducible-builds.org/docs/archives/
# Flags:
#   --sort=name         deterministic entry order (replaces readdir order)
#   --mtime=@SDE        every member's mtime = SDE (eliminates wall-clock leakage)
#   --owner=0/--group=0/--numeric-owner   uid/gid = 0 regardless of build user
#   --format=pax        portable extended headers (required to fit large mtimes)
#   --pax-option=...    strip atime/ctime from PaxHeaders and remove the
#                       random "archive build time" name in exthdr.name=
tar --sort=name \
    --mtime="@${SOURCE_DATE_EPOCH}" \
    --owner=0 --group=0 --numeric-owner \
    --format=pax \
    --pax-option=exthdr.name=%d/PaxHeaders/%f,delete=atime,delete=ctime \
    -cf "$OUTPUT" -C "$STAGING" .
```

## §7 Const VI experiment — layered determinism verification

Per Phase 0 §4 recommendation, run a **three-step ladder** rather than a single 1-week-apart test. This isolates which finding fails at which time scale and lets the Auditor make a precise verdict.

### 7.1 Step 1 — intra-hour same-host (V23.3 B6 acceptance bar)

**When**: within 1 hour of the Builder commit landing on `v22/gap-analysis`.
**How**:

1. Run build A: `bash build-musu-backend.sh --arch amd64 --k3s-version v1.30.4 --output /tmp/musu-backend-A.tar`
2. Wait 1-5 minutes (just enough that wall-clock changes are real)
3. Run build B with identical args: `... --output /tmp/musu-backend-B.tar`
4. Compare: `sha256sum /tmp/musu-backend-A.tar /tmp/musu-backend-B.tar` — the two hashes MUST match.

**Pass criterion**: byte-identical sha256.

**On failure**: Auditor blocks the sub-WS. Most likely cause is F3 (node-gyp embedded paths leaked through despite -ffile-prefix-map) — Builder probes with `objdump -s -j .comment node_modules/better-sqlite3/build/Release/better_sqlite3.node` to see what's still leaking.

### 7.2 Step 2 — same-host ~24h later (informational, not gating)
