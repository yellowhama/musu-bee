# Product Monetization

## Core principle

MUSU monetizes `remote access + packaged operating scenarios + marketplace`, not the local runtime itself.

The free core remains useful on its own:
- local install
- CLI / MCP usage
- localhost / LAN web UI
- local multi-device orchestration inside the user's own network

## Product tiers

### MUSU Core (Free)

Scope:
- local install on user-owned devices
- `localhost` access
- LAN access inside the same trusted network
- CLI / MCP usage
- local orchestration across user-owned devices in the same network
- a small set of free starter packs

Promise:
- "Turn my own machines into one local AI work system."

### MUSU Connect (Paid)

Scope:
- internet-reachable remote access from outside the user's local network
- secure relay / tunnel
- account-linked remote web GUI
- session resume / remote notifications / remote device access
- authenticated remote control-plane access

Promise:
- "Access and operate my MUSU system from anywhere."

### MUSU Packs (Paid)

Scope:
- packaged operating scenarios
- domain-specific workflow bundles
- installable and updateable execution packs

Examples:
- product listing pack
- card-news factory pack
- blog/newsletter pack
- research collection pack
- back-office automation pack

Promise:
- "Buy a working operating scenario, not just a runtime."

### MUSU Shop (Marketplace)

Scope:
- official packs
- third-party packs
- ranking / review / verification
- revenue share / commission

Promise:
- "MUSU becomes a distribution platform for operating scenarios."

## Boundary summary

- Free boundary: `my network`
- Paid boundary: `outside my network`

This must remain explicit in product copy and architecture:
- LAN / localhost stays free
- internet remote access is a paid capability

## Why this model fits MUSU

- It preserves MUSU's identity as a local/on-prem control plane.
- It gives users a clear and fair upgrade trigger.
- It allows scenario monetization without weakening the free runtime.
- It creates a path from product to platform.
