# MUSU SSOT (1-page) — 2026-04-09

## 한 줄 정의

**MUSU는 여러 개인 컴퓨터를 하나의 보호된 on‑prem AI operation으로 묶는 control plane이다.**  
기본 인터페이스는 `CLI/MCP`이고, 채팅/웹 GUI는 `musu.pro`에서 **어디서든 접속 가능한 원격 운영면(surface)**이다.

---

## 코어(지금 기준)

현재 MUSU를 “코어”로 읽어야 하는 축:
- `musu-port` (local control-plane baseline)
- `musu-connects` (peer/trust transport + discovery/route + proof artifacts)
- `MUSU-CRT` (remote session/operator surface baseline; 아직 제품 의미로 fully integrated는 아님)
- `MUSU-WORKS` (company runtime contract / governance seed + safety profile)

**비코어(상위 surface)**:
- Chat/Web GUI: 원격 운영면(필수 아님)
- Mattermost: 도그푸딩 UI(유용하지만 제품 코어 UI로 고정하지 않음)

## 제품 구조(제품 언어)

**Core concerns**
- Control: 상태를 읽고(queue/lane/run/report), 실행을 지시하고(scheduler/executor/watchdog/supervisor), 위험은 승인으로 분리한다(approval).
- Connectivity/Trust: 멀티노드에서 “신뢰 가능한 연결/증거”를 남긴다(trust level, evidence, artifacts).
- Contract/Governance: 회사/프로젝트 운영 규칙이 domain owner로 수렴한다(`MUSU-WORKS`).

**Interfaces**
- Default: `Codex CLI` / `Claude Code` / `MCP`
- Pro: `musu.pro` Web GUI(원격 관측/지시/이어받기; 필요 시 채팅형 UX)

---

## 지금 가장 큰 갭(핵심)

기능 부족이 아니라 **blocked/hard-decision을 “증거 패킷(evidence packet)”으로 바꾸는 체계**가 최대 갭.
- 예: Paddle credential evidence, 5070Ti status proof, run-linkage repair + QA gate verification 등

---

## 운영 안정성(보조 프로그램 관점)

`musu-worker` 중심으로 “터지지 않게” 3종 가드레일을 추가/정리했다:
- 동시성 cap (process explosion 방지)
- systemd/cgroup 제한 (CPUQuota/MemoryMax/TasksMax)
- 디스크 hygiene (TTL/size cap cleanup + optional timer)

## 다음 7일 우선순위(권장)

1) **Ownership 경계 고정**: root control layer / runtime capability / runtime contract 분리(문서가 말하는 구조를 코드 구조로 고정)
2) **Evidence packet 표준화**: blocked/high를 “결정 1–2 + env/deploy 체크리스트 + 검증 커맨드 + 아티팩트”로 통일
3) **CLI/MCP UX 완성**: chat 없는 경로로도 “관측→지시→실행→증거” 루프가 매끄럽게 닫히게
4) **Web GUI는 Pro 운영면으로 제한**: 코어 기능을 먼저 닫고, 원격 접근성/이어받기 가치로 유료화
