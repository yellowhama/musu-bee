§3 Schema v1 freeze

Schema v1 is the authoritative DDL for V24. Four tables, 8 indexes, 1 FK. Column sets were frozen at plan-time (wiki/492 §4) via byte-exact comparison to R1's INSERT/SELECT statements; Auditor VERIFIED-IN-CODE at core/schema.rs:21-78 against R1's existing handlers — no column drift. All future schema changes require a new schema version + migration step; v1 itself is sealed.

companies

cols 10idx 2FK 0
Source-of-truth for company registration. Mirror to companies/<slug>.yaml on every INSERT/UPDATE via R1 integration delta. slug is unique-indexed; active drives /v1/companies?active=true filter.

route_executions

cols 7idx 3FK 1
FK company_id → companies.id ON DELETE SET NULL (preserve audit trail when company removed). Indexed by (idempotency_key), (created_at), (company_id, created_at).

audit_log

cols 9idx 2FK 0
NO FK on company_id per Critic H-1 — audit rows must survive company deletion at the row level, not via SET NULL. Indexed by (ts), (actor_id, ts). v6-mapped columns aligned with R1 audit writer.

machines

cols 8idx 1FK 0
Forward-investment: R3 (control) reads, R9 (cross-machine) writes. Ships in schema v1 to avoid an early schema-v2 churn. Critic M-3 noted mild prefer-defer; trade-off accepted (one DDL freeze beats two).
Column freeze verification: Auditor confirmed byte-exact alignment between schema.rs:21-78 and R1's INSERT/SELECT column lists across all 4 tables. VERIFIED-IN-CODE No column drift; no type-affinity surprise; no NULL-vs-DEFAULT inconsistency. The audit_log no-FK invariant was verified positive (insert succeeds with non-existent company_id) and negative (FK violation would be caught by sqlx error type) in schema.rs:141-176 tests per Critic H-1 requirement.

§5 R1 integration delta

R2 lands a minimal, surgical edit to R1 to wire core/ into the boot path and the companies write surface. Per Critic H-3, the delta is small enough to enumerate exactly. All four amendments + the N-1 audit-fix:

  1. bridge/mod.rs:71 — core::apply(&pool) insertion. Between init_pool (line 66) and AuditState::new (line 73), R2 inserts an idempotent schema apply call. On first boot this creates v1; on subsequent boots it no-ops based on schema_version read. Const III banner emits here on apply.
  2. handlers/companies.rs:187 — post-INSERT write_yaml. After successful company creation (sqlx INSERT returning row), call core::companies::write_yaml(&company) to mirror the row into companies/<slug>.yaml atomically. Errors propagate as 500 (write_yaml failure means DB and yaml are inconsistent — fail loud).
  3. handlers/companies.rs:261 — post-UPDATE write_yaml (after re-SELECT). After UPDATE on /v1/companies/{id}/activate and similar, re-SELECT the row to get current state, then call write_yaml. The re-SELECT step is non-optional: it ensures yaml mirror matches DB after row-trigger or default-column updates.
  4. config.rs:33 — #[allow(dead_code)] retention. Struct-level allow retained while schema_version_required field awaits R3 consumption. Per Critic H-3, R-cleanup tightens to field-level allow.
  5. N-1 audit-fix (commit e775702): 3 handlers received axum::extract::ConnectInfo<SocketAddr> extractor in their function signature, replacing placeholder "127.0.0.1" string. Files: handlers/companies.rs (signature + 1 audit call site), handlers/run.rs (signature + 1 audit call site), handlers/tasks.rs (signature + 1 audit call site). 23 LOC delta total.

R1 unit tests (43) continue to pass after the integration delta — verified by cargo test --workspace on v24/rust-cleanup @ e775702. No R1 behavior changed except: (a) /health now requires schema v1 present, (b) actor_ip in audit_log is real for the three N-1 handlers.

§10 R3 unblocking — what R2 provides for R3

R3 (control module) starts here. R2 hands R3 the following stable surfaces:

  • core::templates::resolve(key) — accepts non-default keys. R3 can register templates by calling into this resolver; current single-default ships in v1 but the contract supports expansion without schema change.
  • core::companies::load_yaml(slug) + load_dir() — file-system discovery surface for R3's indexer. load_dir returns vec of companies with orphan .tmp recovery applied; safe to call from indexer cold-start.
  • machines table — R3 control plane queries this for online nodes. Already in schema v1 (Critic M-3 ship-now trade-off paid off here): R3 doesn't need a schema-v2 migration just to start writing rows.
  • core::probe::is_ready(pool) — boolean readiness check. R3 control loops should gate on this before issuing any cross-cutting query; prevents R3 from racing the migration on cold boot.
  • core::error::CoreError — unified error type R3 should use as its dependency error shape via thiserror #[from]; preserves sqlx + io chain through R3's call sites.
  • musu core check CLI — R3 development docs should reference this as the first-line health check operators run when troubleshooting R3 issues; it's already shipped.
R3 kickoff guidance: Phase 0 Researcher should focus on the R3 control loop surface: which operations are long-poll (need timeout slack like R1 MED-A1 deferred), which are write-heavy (need to batch into route_executions), and which need cross-machine awareness (need machines queries). R2 has frozen the schema; R3 is a pure consumer of schema v1 + a writer to route_executions and machines. No new tables in R3.
