## §4 Process learnings (V23.6+ inputs)

1. **Compressed chain works for low-risk** — 11 of 13 risk-bearing sub-WS shipped SHIP-OK on Critic + Builder without a separate per-WS Auditor. Pattern: risk classification done at impl-plan time (wiki/460 §4), full chain reserved for "extends master thesis" (C-3) or "new XSS surface" (W-2). V23.6: classify sub-WS risk in impl plan, default to compressed chain except where flagged.

2. **Plan-as-spec Auditor pulls its weight at master plan layer, not sub-WS layer** — 4 NEW HIGH (A1..A4) zero overlap with Critic's 9 at the master plan level (wiki/459 ~1540 LOC plan body). No sub-WS plan body exceeded ~110 LOC; running plan-as-spec Auditor on each would have been zero-yield. wiki/439 calibration (500 LOC threshold) holds — V23.5 confirms.

3. **Risk ≠ LOC** — C-3 was a 110-LOC plan but the highest-risk sub-WS (extends master thesis, adds external SaaS dependency path). It received Phase −1 mini-gate + 4-constraint lock independent of its LOC size. V23.6 should classify sub-WS risk on a 2D matrix (LOC × thesis-extension) rather than LOC alone.

4. **HTML closure 60 KB ceiling underutilized** — wiki/470 final shipped at ~49 KB (~82% of ceiling). Room for richer SVG diagrams or expanded retrospective callouts in future closures. wiki/469 W-8 closure was ~23 KB.

5. **Compressed chain trade-off: Builder becomes self-auditor** — risk that single-pass Builder skips edge cases that a separate Auditor would catch. V23.5 W-2 + H-2 caught the issue in-build (Builder's own pre-push review), but the safety margin is thin. V23.6: keep separate Auditor for sub-WS that (a) extend master thesis, (b) touch the XSS / auth boundary, or (c) carry forward a Critic HIGH from master Critic round.

6. **Test count expectation tracking holds** — wiki/460 §6 baseline (510 musu-bee + 95 musu-bridge test files) was the right granularity. Actual test additions (~217 new cases across all four runners) match what wiki/460 §4 acceptance criteria mandated. No surprises, no missing test coverage at close.

7. **Re-grounding (wiki/459 v4 → wiki/460 v0) caught 7 deltas before any Builder spawned** — branch decision, line numbers, scope param strictness, fail-open site count, H-4a HTTP-only scope, C-1 filesystem-vs-DB, test baseline. All 7 would have surfaced as audit-fix commits if Builder had spawned directly from wiki/459 v4. V23.6 should keep the master-plan → impl-plan re-grounding step as standard.

## §5 What worked exceptionally

- **Wave-based parallelization** — 3 sub-WS spawned in parallel within Wave 2 (W-1 + H-3 + H-4a), Wave 3c (W-5 + H-2 + H-5), and Wave 5 (W-6 + C-2). All disjoint file sets; zero merge conflict; 2 wave-iterations saved vs sequential execution.
- **Builder branch-shift handling** — W-3 spawned at HEAD `35951e9` (W-1) but rebased against `605395c` (H-1) at land time. Same for W-6, W-7, C-4. Builders detected the shift in PRIOR ARTIFACTS commit SHAs and rebased without orchestrator intervention.
- **Phase −1 mini-gate as lightweight pattern** — C-3 ran 4-constraint validation in ~5 minutes orchestrator time vs full-panel re-spawn (~20-30 min). Pattern available for V23.6: thesis-extension sub-WS get mini-gate at impl-plan time, full panel only at master plan.
- **HTML closure format consolidated** — three closures (wiki/447, wiki/469, wiki/470) now ship the same Tariq #14 + CSS-vars + 60 KB-cap template. V23.6+ Scribe convention set.

## §6 What needs improvement

- **Test LOC unbudgeted in master plan** — drives the headline "+23% overrun" that is in fact just unbudgeted test code. V23.6 master plan should add a separate test-LOC budget line per sub-WS (rule of thumb: 1.5x production LOC for risk-bearing sub-WS, 0.5x for low-risk).
- **Risk classification belongs in impl plan, not master plan** — wiki/460 §4 had it; wiki/459 §2 implicitly treated all sub-WS as equivalent. V23.6 master plan should declare risk class per sub-WS up front so compressed-vs-full chain decision is made at master plan time.
- **HTML closure SVG diagram space underused** — wiki/470 ships one wave diagram. Could carry per-track timing diagrams (1 per W/H/C), pipeline depth view, or Phase −1 reshape visualization. Future closures should aim for 2-3 SVG diagrams to use the 60 KB ceiling productively.
- **`MUSU_USER_LLM_API_KEY` operator-side documentation thin** — C-3 ships the constraint but the operator-facing setup doc is one-line in `wiki/479`. V23.6 should produce a per-operator setup guide if dogfood shows ≥3 users attempting opt-in.
- **W-1 npm lockfile noise dominates LOC stats** — 2252 lines of `package-lock.json` for 7 deps. Future closures should report production LOC + test LOC + dep-lock LOC separately to make the headline number truthful.
