## §4 Process learnings (V23.6+ inputs)

1. **Compressed chain works for low-risk** — 11 of 13 risk-bearing sub-WS shipped SHIP-OK on Critic + Builder without a separate per-WS Auditor. Pattern: risk classification done at impl-plan time (wiki/460 §4), full chain reserved for "extends master thesis" (C-3) or "new XSS surface" (W-2). V23.6: classify sub-WS risk in impl plan, default to compressed chain except where flagged.

2. **Plan-as-spec Auditor pulls its weight at master plan layer, not sub-WS layer** — 4 NEW HIGH (A1..A4) zero overlap with Critic's 9 at the master plan level (wiki/459 ~1540 LOC plan body). No sub-WS plan body exceeded ~110 LOC; running plan-as-spec Auditor on each would have been zero-yield. wiki/439 calibration (500 LOC threshold) holds — V23.5 confirms.

3. **Risk ≠ LOC** — C-3 was a 110-LOC plan but the highest-risk sub-WS (extends master thesis, adds external SaaS dependency path). It received Phase −1 mini-gate + 4-constraint lock independent of its LOC size. V23.6 should classify sub-WS risk on a 2D matrix (LOC × thesis-extension) rather than LOC alone.

4. **HTML closure 60 KB ceiling underutilized** — wiki/470 final shipped at ~49 KB (~82% of ceiling). Room for richer SVG diagrams or expanded retrospective callouts in future closures. wiki/469 W-8 closure was ~23 KB.

5. **Compressed chain trade-off: Builder becomes self-auditor** — risk that single-pass Builder skips edge cases that a separate Auditor would catch. V23.5 W-2 + H-2 caught the issue in-build (Builder's own pre-push review), but the safety margin is thin. V23.6: keep separate Auditor for sub-WS that (a) extend master thesis, (b) touch the XSS / auth boundary, or (c) carry forward a Critic HIGH from master Critic round.

6. **Test count expectation tracking holds** — wiki/460 §6 baseline (510 musu-bee + 95 musu-bridge test files) was the right granularity. Actual test additions (~217 new cases across all four runners) match what wiki/460 §4 acceptance criteria mandated. No surprises, no missing test coverage at close.

7. **Re-grounding (wiki/459 v4 → wiki/460 v0) caught 7 deltas before any Builder spawned** — branch decision, line numbers, scope param strictness, fail-open site count, H-4a HTTP-only scope, C-1 filesystem-vs-DB, test baseline. All 7 would have surfaced as audit-fix commits if Builder had spawned directly from wiki/459 v4. V23.6 should keep the master-plan → impl-plan re-grounding step as standard.

## §5 What worked exceptionally

- **Wave-based parallelization** — 3 sub-WS spawned in parallel within Wave 2 (W-1 + H-3 + H-4a), Wave 3c (W-5 + H-2 + H-5), and Wave 5 (W-6 + C-2). All disjoint file sets; zero merge conflict; 2 wave-iterations saved vs sequential execution.
- **Builder branch-shift handling** — W-3 spawned at HEAD `35951e9` (W-1) but rebased against `605395c` (H-1) at land time. Same for W-6, W-7, C-4. Builders detected the shift in PRIOR ARTIFACTS commit SHAs and rebased without orchestrator intervention.
- **Phase −1 mini-gate as lightweight pattern** — C-3 ran 4-constraint validation in ~5 minutes orchestrator time vs full-panel re-spawn (~20-30 min). Pattern available for V23.6: thesis-extension sub-WS get mini-gate at impl-plan time, full panel only at master plan.
- **HTML closure format consolidated** — three closures (wiki/447, wiki/469, wiki/470) now ship the same Tariq #14 + CSS-vars + 60 KB-cap template. V23.6+ Scribe convention set.

## §6 What needs improvement

**Section v2 amend (2026-05-19)** — items #1..#6 added at top per `business-panel-experts` adversarial critique of wiki/471 v1. Panel verdict: AMEND (4-expert unanimous). 5 HIGH + 6 MED findings; the items below are the load-bearing ones. Original cosmetic items moved to #7..#11.

### Honest failure modes (panel-driven, 2026-05-19 amend)

1. **Doc evaluates orchestrator process, not product. Product outcome unmeasured.** §1-§5 are exclusively agent-team-chain metrics (commits, LOC, phase chain effectiveness, estimation accuracy). The original §4-§6 16 items contained ZERO product-outcome questions: did the agent-facing wiki HTML render get read by an agent? Did `recent_wiki_pages` in `api_company_briefing` change downstream behavior? Did anyone opt into `MUSU_USER_LLM_API_KEY`? **Unknown — V23.5 ships no usage metering.** This is the load-bearing failure mode: V23.5 may have shipped genuine value; this doc cannot tell us whether it did. (Christensen H2.)

2. **Inline-folded audit-fixes destroy compressed-chain ROI measurement.** §1 line "0 audit-fix-as-separate-commit (folded inline)" was originally framed as a process win. It is opacity: the pre-fix Builder intermediate state was rebased away, so the "11 of 13 risk-bearing sub-WS shipped SHIP-OK first pass" claim in §2 is structurally unfalsifiable. V23.4 had separate audit-fix commits (`27f9a0a`, `b6a1548`); V23.5 does not. V23.6 must either restore separate audit-fix commits OR record pre-fix vs post-fix Builder diff in the plan doc HANDOFF NOTES, or the compressed-chain effectiveness claim cannot survive. (Taleb H4.)

3. **Parallel sub-WS spawn outran operator review bandwidth; #436 batched-merge backlog is 4 cycles deep.** §5 "Wave-based parallelization saved 2 wave-iterations" frames parallelism as throughput. For a solo operator, parallelism that ships 3 sub-WS in the wall-clock of 1 piles up 3x unreviewed backlog. Verifiable fact: `git log` confirms 18 Builder commits in 2h39m wall-clock. The operator-merge gate (#436) bundles V23.3 + V23.4 Tier-1 + V23.4 Phase 4 + V23.5 — four full cycles unreviewed. By Little's Law, throughput is bounded by the slowest stage (operator review), not Builder elapsed time. V23.6 must default to operator-attended cadence (sequential sub-WS where operator decision is pending) OR explicitly tag autonomous sub-WS chains and report chain performance per category. (Kim&Mauborgne H5 + Drucker M5.)

4. **Sub-WS risk classification treats dependency additions as low-risk; this is the wrong frame.** W-1 added 7 npm dependencies (react-markdown + dompurify + rehype-sanitize + transitive) and was classified low-risk on the LOC axis alone. Per `[[feedback-self-contained-product]]` and `[[feedback-no-yagni-architecture]]`, dep additions are irreversible product-positioning commitments — every dep is a future security-patch obligation + supply-chain attack surface + long-tail breaking-change vector. V23.6 risk classifier must add an **irreversibility axis** orthogonal to LOC and thesis-extension. (Christensen M6.)

5. **"All tests green" is unfalsifiable when tests were not pre-specified.** §3 acknowledges test LOC was unbudgeted (`~7400 LOC test code unbudgeted in master plan`). When the master plan does not enumerate the test surface and the Builder writes its own tests, "all green" is a tautology — the Builder validated whatever the Builder produced. V23.6 master plan MUST enumerate per-sub-WS test obligations (test count + key fixtures + acceptance criteria) at master-plan time, and treat tests as acceptance criteria the Builder must satisfy, not artifacts the Builder produces. (Taleb M3.)

6. **#436 is currently treated as procedural; in reality it is musu's velocity bottleneck.** §7 v1 marked V23.6 readiness GREEN with #436 pending; the doc made the bottleneck invisible by framing it as a one-line readiness check. v2 §7 below downgrades to YELLOW until operator confirms whether #436 is a deliberate batching policy or accidental backlog. (Drucker M5.)

### Cosmetic items (v1, retained but lower-priority)

7. **Test LOC unbudgeted in master plan** — drives the headline "+23% overrun" that is in fact just unbudgeted test code. V23.6 master plan should add a separate test-LOC budget line per sub-WS (rule of thumb: 1.5x production LOC for risk-bearing sub-WS, 0.5x for low-risk). **Superseded by #5 above** — the deeper issue is that tests must be pre-specified as acceptance criteria, not just budgeted.
8. **Risk classification belongs in impl plan, not master plan** — wiki/460 §4 had it; wiki/459 §2 implicitly treated all sub-WS as equivalent. V23.6 master plan should declare risk class per sub-WS up front so compressed-vs-full chain decision is made at master plan time. **Superseded by #4 above** — risk axis must include irreversibility, not just LOC.
9. **HTML closure SVG diagram space underused** — wiki/470 ships one wave diagram. Could carry per-track timing diagrams (1 per W/H/C), pipeline depth view, or Phase −1 reshape visualization. Future closures should aim for 2-3 SVG diagrams to use the 60 KB ceiling productively.
10. **`MUSU_USER_LLM_API_KEY` operator-side documentation thin** — C-3 ships the constraint but the operator-facing setup doc is one-line in `wiki/479`. V23.6 should produce a per-operator setup guide if dogfood shows ≥3 users attempting opt-in. **Conditional on #1 above** — until we know whether anyone is opting in, prioritizing operator-setup-doc work is premature.
11. **W-1 npm lockfile noise dominates LOC stats** — 2252 lines of `package-lock.json` for 7 deps. Future closures should report production LOC + test LOC + dep-lock LOC separately to make the headline number truthful.
