## §1 Scope + gate precondition

A1.b bakes three K3s manifest objects + (optionally) one runtime-generated Secret for the `musu-bridge` Pod into the imported WSL2 distro, so that K3s auto-deploys the bridge on first boot via the K3s manifest controller:

- **Deployment** — single-replica Pod running the OCI image baked by A1.a (`musu-bridge:v0.1.0` @ digest `sha256:ad3097cd0966fe34a0872ac9c3c30386d6fc59d59c737e791d53c692aabb4626`).
- **Service** — `ClusterIP` only in A1.b (Pod-internal reachability). A1.c will add the host-port surface (NodePort or hostPort).
- **ConfigMap** — non-secret env vars (`MUSU_ENV=production`, `BRIDGE_HOST=0.0.0.0`, `BRIDGE_PORT=8070`).
- **Secret** (runtime-generated by `musu-init`, NOT baked) — `musu-bridge-token` Opaque Secret carrying `MUSU_BRIDGE_TOKEN` (≥32 chars, generated via `openssl rand -hex 32` on first boot, idempotent across distro restarts).

Manifest file lands at `var/lib/rancher/k3s/server/manifests/musu-bridge.yaml` inside `musu-backend.tar`; K3s's auto-deploy manifest controller picks it up on first start and applies it to the cluster.

### 1.1 Image reference

The Deployment `image:` field uses **manifest digest pinning** (not tag):

```yaml
image: musu-bridge@sha256:ad3097cd0966fe34a0872ac9c3c30386d6fc59d59c737e791d53c692aabb4626
```

Digest source: A1.a closure wiki/381 §5 + manifest.yaml:58 (`bridge_image_digest`). containerd resolves this digest against the airgap-loaded image at `/var/lib/rancher/k3s/agent/images/musu-bridge-v0.1.0.tar` on first import. Digest is immutable; tag-based pinning would not protect against airgap-cache rebuild drift.

### 1.2 What this plan does NOT cover (forward-pointers)

- **A1.c** (wiki/384): host-port surface (NodePort/hostPort), `--prefer k3s|systemd` CLI flag in `cli.py`, Const VI capacity bench (cold-start latency, P99 RPC latency, sustained CPU/throughput, oom recovery, rollback ≤30s).
- **PersistentVolumeClaim via local-path-provisioner**: deferred to V23.5 polish. A1.b uses hostPath for `~/.musu/` state.
- **Image registry pull**: never. K3s containerd resolves the digest against the local airgap-loaded image.
- **Multi-replica HA**: bridge is operator-local and stateful; single replica matches the host-systemd model.
- **NetworkPolicy / PodSecurityPolicy / OPA**: deferred to V23.4 hardening (musu-bee installer's first cut is single-tenant).

### 1.3 Acceptance bar (formal)

1. K3s Pod `musu-bridge` becomes `Running` on first boot of the imported tar.
2. `kubectl logs -n musu deploy/musu-bridge` shows uvicorn-started without FATAL.
3. `curl http://<pod-cluster-ip>:8070/health` returns HTTP 200 with body `{"status":"ok","worker":false,...}` (worker:false is steady state; NOT a defect — see §3 Finding H3 + §7).
4. `curl http://<pod-cluster-ip>:8070/health/ready` returns HTTP 200 with body `{"status":"ready","db":"ok",...}` once the DB-init has completed (~3-5s after Pod Ready).
5. After `wsl --terminate musu && wsl -d musu` (distro restart), Pod returns to Running with the same Secret value (`MUSU_BRIDGE_TOKEN` persists in K3s sqlite datastore at `/var/lib/rancher/k3s/server/db/state.db`). **Critic C-A1b-H5 disclosure**: during restart, the `Recreate` strategy + `terminationGracePeriodSeconds: 45` produces a ≤45s service gap (P50 likely ~10s with no active tasks; P99 up to 45s with active tasks). musu-bee callers see connection-refused during this window; A1.c plan owns mitigation.
6. **B6 byte-identity NOT regressed**: two intra-hour clean builds of `musu-backend.tar` with the A1.b changes produce identical sha256 (matches B6 §7.1 Step 1 contract; matches A1.a §5 Const VI re-verification at sha256 `b71b225d6079782354ae1c73a030583a768418e70be187b63b69692806073b75`).

### 4.4 File 3: `musu-relay/installer/musu-init` (extend ~20 LOC)

Insert a new step **3.5** between the existing step 3 (account_key wait, currently lines 55-76) and step 4 (gateway start, line 78). The new step generates the `musu-bridge-token` Secret idempotently using kubectl-server-side-apply semantics:

```sh
# ── 3.5. Generate musu-bridge-token Secret (V23.3 A1.b / wiki/382) ─────────
# Idempotent: kubectl create --dry-run=client -o yaml | kubectl apply -f -
# preserves the existing Secret value across distro restarts (sqlite datastore
# at /var/lib/rancher/k3s/server/db/state.db survives wsl --terminate).
# Regenerates only on a fresh `wsl --unregister`/import cycle.
#
# Token spec: 64 hex chars (openssl rand -hex 32) satisfies the ≥32-char
# enforcement at musu-bridge/server.py:208-217 (post-S-A1a-H2 fix).
echo "musu-init: ensuring musu-bridge-token Secret exists in musu namespace"
KUBECTL="kubectl --kubeconfig /etc/rancher/k3s/k3s.yaml"
if ! $KUBECTL -n musu get secret musu-bridge-token >/dev/null 2>&1; then
  # Critic C-A1b-H2 resolution: use kernel CSPRNG via /dev/urandom + busybox od
  # (no openssl binary dependency, no xxd dependency). 64 hex chars satisfies
  # the >=32-char gate at musu-bridge/server.py:208-217.
  TOKEN="$(head -c 32 /dev/urandom | od -An -tx1 | tr -d ' \n')"
  # Namespace already created by the K3s manifest controller from
  # musu-bridge.yaml; this `apply` is idempotent belt-and-suspenders.
  $KUBECTL create namespace musu --dry-run=client -o yaml | $KUBECTL apply -f -
  $KUBECTL -n musu create secret generic musu-bridge-token \
    --from-literal=MUSU_BRIDGE_TOKEN="$TOKEN" \
    --dry-run=client -o yaml | $KUBECTL apply -f -
  echo "musu-init: musu-bridge-token Secret created (64-char token)"
else
  echo "musu-init: musu-bridge-token Secret already present (preserved across restart)"
fi

# Critic C-A1b-H4 resolution: chmod /etc/musu/account_key to 0644 root:root
# (single-tenant WSL2 appliance accepts world-readable inside the distro
# filesystem; uid-1000 pod must read it via the hostPath mount). The original
# 0600 root:root host ACL was sized for the host-systemd bridge running as
# root; the K3s pod model invalidates that assumption.
if [ -f /etc/musu/account_key ]; then
  chmod 0644 /etc/musu/account_key
fi

# Critic C-A1b-H3 resolution: scale Deployment from 0 → 1 once both Secret
# and account_key are present. Eliminates CrashLoopBackOff entirely.
echo "musu-init: scaling musu-bridge Deployment to 1 replica"
$KUBECTL -n musu scale deploy musu-bridge --replicas=1 || true
```

**Critic C-A1b-H2 resolution (no openssl dependency)**: step 3.5 uses busybox `od -An -tx1` against `/dev/urandom`. No new apk pin needed. `od` is available in `alpine-base`.

**Critic C-A1b-H4 resolution (account_key host ACL)**: step 3.5 also chmods `/etc/musu/account_key` to `0644 root:root` after the wait-loop fires. World-readable inside the WSL2 distro filesystem is acceptable for a single-tenant operator appliance; the host file is already isolated to the Windows operator's user profile (`%LOCALAPPDATA%\Packages\…`). **Plan §4.5 amendment**: `musu-write-key` is NOT touched (we chmod at musu-init time instead, which is in scope for `musu-relay/installer/` modifications).

**Note on `/etc/musu/account_key` wait-loop interaction**: the existing wait-loop at musu-init:63-75 covers the host file (`/etc/musu/account_key`). Per H3 resolution above, the Deployment starts at `replicas: 0` and step 3.5 scales to 1 only AFTER both the Secret is created AND the chmod has run. No CrashLoopBackOff cascade.

### 4.6 build-musu-backend.sh step 6 amendment (Critic C-A1b-H1)

In addition to the new step 3.e (manifest bake), step 6 (reserve directories) must be extended to:
1. Add `musu` group with gid 1000 in the host rootfs: `chroot $STAGING addgroup -g 1000 musu` (or sed/echo into /etc/group).
2. Add `musu` user with uid 1000 in the host rootfs: `chroot $STAGING adduser -D -u 1000 -G musu -H -s /sbin/nologin musu`.

**Alternative (safer for B6)**: skip the chroot adduser at build time; have musu-init step 3.b (NEW, before step 3) do `addgroup -g 1000 musu 2>/dev/null; adduser -D -u 1000 -G musu -H -s /sbin/nologin musu 2>/dev/null; mkdir -m 0770 -p /var/lib/musu/state; chown 1000:1000 /var/lib/musu/state`. This keeps the tar deterministic and pushes the group/user creation to first-boot. Builder picks this path.

## §10 Forward-pointers for A1.c (wiki/384)

A1.c is the cutover plan. A1.b leaves the following explicit gaps for A1.c to close:

1. **Host-port surface (H4 deferral)** — A1.c MUST add either `hostPort: 8070` on the Pod container, or a `NodePort` Service on the existing `Service` object, or a sidecar reverse proxy. A1.c MUST also address the collision case where a developer host runs both host-systemd musu-bridge AND K3s Pod (port-8070 contention). Recommended: NodePort 30870 or hostPort 8071 + reverse proxy from 8070.

2. **A1.c bench reachability** — Without host-port, A1.c's Const VI capacity bench cannot hit the Pod from outside the cluster. Options: (a) `kubectl port-forward` from a Windows-host benchmark harness (high overhead, may distort latency measurements), or (b) bench-pod inside the cluster (`kubectl run bench --image=busybox:latest -- wrk ...`). Recommendation: option (b) for clean measurements, with a smaller (a) variant to validate operator-facing latency.

3. **Rollback semantic** — Plan A1.a §10.3 + plan A1.b §1.2 originally said "fall back to systemd-on-host." For WSL2 operator hosts, **there is no host-side systemd musu-bridge**: the imported distro IS the only execution surface. Rollback options A1.c must consider: (a) `kubectl scale deploy musu-bridge --replicas=0` + manual `python -m uvicorn` run inside the distro (operator-painful), (b) fall back to the previous tar version (re-import), (c) document "no rollback; fix-forward only" as the explicit A1.c stance. The musu-bee frontend's `MUSU_BRIDGE_URL` default of `http://localhost:8070` (per `src/app/api/agent-route/route.ts` + `src/app/api/bridge/[...path]/route.ts` + `src/app/api/mcp/config.ts`) requires the bridge to be reachable at `localhost:8070` on the operator host regardless of substrate — A1.c's host-port decision must preserve this contract.

4. **`--prefer k3s|systemd` flag scope** — On WSL2-only hosts, this flag has no `systemd` branch to prefer. A1.c may need to refactor to `--prefer k3s|host-python` or drop the flag entirely in favor of substrate-detection.

5. **Cold-start latency in bench** — A1.b's `startupProbe failureThreshold:30` gives a 60s cold-start budget. A1.c's bench should report actual cold-start time (P50, P99) and refine probe timings if observed cold start is consistently <10s.

6. **Recreate-strategy 45s gap (Critic C-A1b-H5)** — A1.b's `Recreate` + `terminationGracePeriodSeconds: 45` produces up to 45s service unavailability on every Pod update. A1.c MUST either: (a) document acceptance of the gap in A1.c plan + operator-facing release notes, (b) reduce `terminationGracePeriodSeconds` to 15 (accepting in-flight task termination as the trade-off), or (c) introduce a hostPath-state-coordination mechanism that allows brief 2-replica overlap during update.
