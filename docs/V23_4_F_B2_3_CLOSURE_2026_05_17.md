## §1 What shipped

F-B2-3 closes the V23.3 B2 forward-pointer NEW-MED-3 by wrapping all four
telemetry route DB-write call sites in a uniform `try/catch` and converging
the failure path on a single structured 500 contract. Before F-B2-3, a thrown
exception from `_db.prepare(...).run(...)` (disk-full on the shared Fly
`/data/telemetry.db` volume, schema-mismatch from a hot redeploy, sqlite-busy
under write contention) escaped the route handler and Express returned its
default error response — an HTML stack trace in dev, a bare unstructured 500
in prod. F-B2-3 normalizes this to `{ error: "database write failed" }` JSON
across all four routes (`/install`, `/install_attempt`, `/nat_pierce`,
`/agent_spawn`) so clients see a stable contract regardless of which write
fails.

### 1.1 Server-side surface (Builder commit `5b7814b`)

- **`/install` DB-write wrap** at
  `musu-relay/src/signaling/telemetry.ts:851-880` — single
  `_db.prepare(INSERT_TELEMETRY_INSTALL).run(...)` call site wrapped in
  `try { ... } catch (err) { console.error("[telemetry] /install: db write
  failed: " + msg); res.status(500).json({ error: "database write failed" });
  return; }`. The trailing `return;` at telemetry.ts:879 is load-bearing per
  Critic C11 — without it, control falls through to the route's terminal
  `res.status(204).end()` and Express throws "Cannot set headers after they
  are sent".
- **`/install_attempt` DB-write wrap** at telemetry.ts:988-1016 — same
  structure; `return;` at telemetry.ts:1015.
- **`/nat_pierce` DB-write wrap** at telemetry.ts:1030-1053 — same structure;
  `return;` at telemetry.ts:1051.
- **`/agent_spawn` DB-write wrap** at telemetry.ts:1066-1087 — same
  structure; `return;` at telemetry.ts:1086.
- **500 contract uniformity**: all four catch blocks emit identical wire-level
  shape (`{ error: "database write failed" }`) and identical console.error
  format (`"[telemetry] /<route>: db write failed: <err.message>"`). The
  per-route token in the log line is the only inter-route variance — both for
  operator triage and so future grep-based test patterns can disambiguate
  which handler caught.

### 1.2 Test surface

- **NEW** test module
  `musu-relay/tests/telemetry-db-failure.test.ts` (242 lines), 4 tests
  (TDF-1..TDF-4 covering all four routes 1:1):
  - **TDF-1** — `/install` returns 500 + JSON contract + log line + no
    "Cannot set headers" leakage when the INSERT throws.
  - **TDF-2** — `/install_attempt` same set of assertions.
  - **TDF-3** — `/nat_pierce` same set of assertions.
  - **TDF-4** — `/agent_spawn` same set of assertions.
- **Mock strategy**: `jest.spyOn(_db, "prepare")` with a selective predicate
  that throws only on the INSERT statement for the route under test
  (matches by SQL text fragment) and falls through to the real prepare for
  all other statements. Per wiki/408 §4 strategy (d): selective predicate
  chosen over `mockImplementationOnce` because each route handler's
  pre-INSERT prepare-call order varies (HMAC auth path validates a session
  row via its own prepare → unconditional `mockImplementationOnce` would
  trip on the auth check, not the route INSERT). Selective predicate is
  resilient to pre-INSERT prepare-call order across handlers.
- **C11 regression assertion**: every TDF-N asserts the absence of "Cannot
  set headers after they are sent" in `console.error` output, so any future
  regression that drops the `return;` after a catch surfaces immediately.
- **Test count**: 224 → 228 (+4).
- **Final**: **228/228 jest green, `npx tsc --noEmit` clean**.
