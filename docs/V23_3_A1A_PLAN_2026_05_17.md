## §1 Scope

A1.a containerizes the musu-bridge daemon as an OCI image, bakes the resulting image artifact into `musu-backend.tar` (parallel to the K3s airgap-images vendoring pattern), and pins both the image tag + digest in `manifest.yaml`. Per wiki/379 §2 A1.a row and Critic C10: **bridge-only scope** — no gateway-service artifacts, no worker/connectsd/VNC sidecar absorption.

### 1.1 Acceptance bar

- An OCI image `musu-bridge:vX.Y.Z` exists, baked into the rootfs tar at `var/lib/rancher/k3s/agent/images/musu-bridge-vX.Y.Z.tar`
- The image runs `uvicorn server:app --host 0.0.0.0 --port 8070` as USER `musu` (UID 1000) under `tini` as PID 1
- The image's `/health` and `/health/ready` endpoints are reachable on port 8070 inside the container (verified via local `docker run` smoke from outside the K3s context)
- **B6 byte-identity preserved**: two intra-hour clean builds on the same Alpine WSL2 host produce identical `sha256(musu-backend.tar)` (the B6 acceptance bar must not regress when the image is added)
- `manifest.yaml` pins `bridge_image_tag` + `bridge_image_oci_archive_sha256_amd64` + `bridge_image_digest`
- The Dockerfile installs ONLY runtime essentials (`tini`, optionally `curl`); NOT `x11vnc`/`Xvfb`/`openssl`/`lsof`/`sqlite3`/`musu-worker`/`musu-connectsd`

### 1.2 What this plan does NOT cover (forward-pointers)

- **A1.b** (wiki/382): K3s Deployment + Service + Secret + ConfigMap manifest; Pod-level SecurityContext + Secret mount semantics for `/etc/musu/account_key`; PVC/emptyDir for `~/.musu/db/`
- **A1.c** (wiki/384): `--prefer k3s|systemd` CLI flag in `musu-bridge/cli.py`; parallel-path cutover orchestration; Const VI capacity bench (cold-start latency, memory RSS, oom recovery, P99 RPC latency)
- **musu-worker / musu-connectsd / VNC subsystem containerization**: out of V23.3 scope (sibling-pod / V23.4+ work)
- **Image registry push, cosign signing, in-toto attestation, multi-arch (arm64)**: V23.5 reproducible-build attestation rollforward (per wiki/393 §10.2)
- **`MUSU_DB_PATH` env parameterization** across the bridge codebase: deferred to A1.b mount-semantics decision (A1.a's `/health` references `~/.musu/db/musu.db` and degrades gracefully if missing — Phase 0 Researcher F2 verified)
- **musu-bridge source-code edits** (server.py, config.py, cli.py): explicitly excluded — A1.a is Dockerfile + lockfile + build-pipeline + manifest-only. Per Critic C10 bridge-only scope, source-code changes are A1.b/A1.c concerns.

## §4 Implementation

### 4.1 New files (Builder creates)

#### `musu-bridge/Dockerfile`

```dockerfile
# syntax=docker/dockerfile:1.7
# V23.3 A1.a (wiki/380): musu-bridge OCI image.
#
# BUILD CONTEXT: repo root (sibling to musu-core/).
#   Run: buildah build -f musu-bridge/Dockerfile <repo-root>
#   NOT: buildah build musu-bridge/    ← `COPY musu-core/...` would fail
# (Critic M3: Dockerfile in isolation does not signal context requirement.)
#
# Built via `buildah build --timestamp ${SOURCE_DATE_EPOCH}` inside Alpine WSL2.
# Baked into musu-backend.tar as
#   var/lib/rancher/k3s/agent/images/<bridge_image_name>-<bridge_image_version>.tar
# K3s containerd auto-imports on first boot. The OCI archive is re-packed
# with the B6 reproducible tar recipe by build-musu-backend.sh step 3.d
# (Critic H1) — do NOT rely on buildah's native oci-archive layout.

FROM python:3.11-slim-bookworm

# Provenance labels (V23.3 A1.a F16). Values passed via --build-arg.
ARG GIT_SHA=unknown
ARG GIT_DESC=unknown
ARG BUILD_TS=unknown
ARG BRIDGE_VERSION=0.1.0
ARG SOURCE_DATE_EPOCH=0
LABEL org.opencontainers.image.source="https://github.com/yellowhama/musu-bee" \
      org.opencontainers.image.revision="${GIT_SHA}" \
      org.opencontainers.image.created="${BUILD_TS}" \
      org.opencontainers.image.title="musu-bridge" \
      org.opencontainers.image.version="${BRIDGE_VERSION}" \
      org.opencontainers.image.description="musu-bridge daemon (FastAPI+uvicorn watch-dispatcher)"

# tini for clean PID 1 signal handling (F9). Pinned version (Critic L1) for
# byte-identity across Debian point-releases. Verify with `apt-cache madison tini`
# inside the build distro and bump in lockstep with Bookworm minor bumps.
RUN apt-get update && \
    apt-get install -y --no-install-recommends tini=0.19.0-1 && \
    rm -rf /var/lib/apt/lists/* && \
    find /usr/bin/tini /usr/share/doc/tini -exec touch -h -d "@${SOURCE_DATE_EPOCH}" {} + 2>/dev/null || true

# Non-root user for A1.b Secret mount compat (F10). UID 1000 matches
# K3s SecurityContext pinning convention. HOME = /home/musu; A1.b mounts
# Secret at /home/musu/.musu/account_key (Critic H2 — matches host
# convention scripts/start-bridge.sh:288).
RUN groupadd -g 1000 musu && \
    useradd -u 1000 -g musu -m -s /sbin/nologin musu

# Lockfile-driven, hash-verified pip install (F11). Preserves B6 byte-identity.
COPY --chown=musu:musu musu-bridge/requirements-lock.txt /tmp/
RUN pip install --no-cache-dir --no-compile --no-deps --require-hashes \
        -r /tmp/requirements-lock.txt && \
    rm /tmp/requirements-lock.txt

# Source layout (F1, F3): wholesale copy of musu-bridge + musu_core sibling.
# Critic M5: PYTHONPATH=/app only (musu_core is importable as /app/musu_core;
# the second prior entry /app/musu-core/src pointed to a non-existent dir).
COPY --chown=musu:musu musu-bridge/ /app/
COPY --chown=musu:musu musu-core/src/musu_core /app/musu_core
ENV PYTHONPATH=/app

# Scrub .pyc/.pyo timestamps for B6 byte-identity (F4 belt-and-suspenders).
RUN find /app -name '__pycache__' -type d -exec rm -rf {} + 2>/dev/null; \
    find /app -name '*.pyc' -delete; \
    find /app -name '*.pyo' -delete

USER musu
WORKDIR /app

EXPOSE 8070

# Critic L4: BRIDGE_HOST/BRIDGE_PORT ENV are NOT set here because the CMD
# below hardcodes --host/--port; config.py defaults are only consulted by
# `python -m server` (the __main__ path), which CMD does not invoke.
# Keeping it minimal avoids dead-code env vars that mislead future readers.
ENV PYTHONUNBUFFERED=1 \
    PYTHONDONTWRITEBYTECODE=1

# MUSU_BRIDGE_TOKEN is REQUIRED at runtime (server.py:204-207 FATAL-exits
# without it). The image deliberately does NOT set a default — A1.b
# ConfigMap/Secret must inject. See Critic H3 + plan §10.3.

ENTRYPOINT ["/usr/bin/tini", "--"]
CMD ["python", "-m", "uvicorn", "server:app", "--host", "0.0.0.0", "--port", "8070"]
```

**S3.5.b — bridge image build** (new step, between `[3.c/10]` and `[4/10]`):

```bash
# ── Step 3.d: Build musu-bridge OCI image (V23.3 A1.a / wiki/380) ─────────
echo "[3.d/10] Build musu-bridge OCI image (buildah --timestamp=SDE)"

# Critic H5 resolution: split bridge_image_tag into name + version to avoid
# yaml_get parser fragility on values containing ':' (the awk parser at :87
# was only ever tested with colonless values).
BRIDGE_IMAGE_NAME="$(yaml_get bridge_image_name)"
BRIDGE_IMAGE_VERSION="$(yaml_get bridge_image_version)"
if [ -z "$BRIDGE_IMAGE_NAME" ] || [ -z "$BRIDGE_IMAGE_VERSION" ]; then
  echo "FATAL: manifest.yaml missing bridge_image_name or bridge_image_version (V23.3 A1.a / wiki/380 H5)" >&2
  exit 1
fi
BRIDGE_TAG="${BRIDGE_IMAGE_NAME}:${BRIDGE_IMAGE_VERSION}"
BRIDGE_IMAGE_TAR="$STAGING/var/lib/rancher/k3s/agent/images/${BRIDGE_IMAGE_NAME}-${BRIDGE_IMAGE_VERSION}.tar"
mkdir -p "$(dirname "$BRIDGE_IMAGE_TAR")"

# buildah needs repo root as context (Dockerfile COPYs both musu-bridge/ and musu-core/).
BUILD_CONTEXT="$(cd "$SCRIPT_DIR/../.." && pwd)"

# Critic H1 step 1: build into a temporary OCI directory (NOT directly to tar)
# so we control the tar layout in step 2 below.
BRIDGE_OCI_DIR="$STAGING/.bridge-oci-tmp"
rm -rf "$BRIDGE_OCI_DIR"
mkdir -p "$BRIDGE_OCI_DIR"

buildah build \
    --timestamp "${SOURCE_DATE_EPOCH}" \
    --build-arg SOURCE_DATE_EPOCH="${SOURCE_DATE_EPOCH}" \
    --build-arg GIT_SHA="${GIT_SHA:-unknown}" \
    --build-arg GIT_DESC="${GIT_DESC:-unknown}" \
    --build-arg BUILD_TS="${BUILD_TS:-unknown}" \
    --build-arg BRIDGE_VERSION="${BRIDGE_IMAGE_VERSION}" \
    -f "$BUILD_CONTEXT/musu-bridge/Dockerfile" \
    -t "${BRIDGE_TAG}" \
    "$BUILD_CONTEXT"

# Push to OCI directory format (uncompressed, layout-preserved).
buildah push "${BRIDGE_TAG}" "oci:${BRIDGE_OCI_DIR}:${BRIDGE_IMAGE_VERSION}"

# Critic H1 step 2: re-pack the OCI directory with the SAME B6 reproducible
# tar recipe used by the outer musu-backend.tar. Buildah's native oci-archive
# output does NOT guarantee --sort=name --format=pax --pax-option=... layout,
# so we don't trust it — we re-pack ourselves with the proven recipe.
tar --sort=name \
    --mtime="@${SOURCE_DATE_EPOCH}" \
    --owner=0 --group=0 --numeric-owner \
    --format=pax \
    --pax-option=exthdr.name=%d/PaxHeaders/%f,delete=atime,delete=ctime \
    -cf "$BRIDGE_IMAGE_TAR" -C "$BRIDGE_OCI_DIR" .
rm -rf "$BRIDGE_OCI_DIR"

# Verify against manifest.yaml-pinned sha256 (or populate-on-first-build).
EXPECTED_SHA="$(yaml_get bridge_image_oci_archive_sha256_${ARCH})"
ACTUAL_SHA="$(sha256sum "$BRIDGE_IMAGE_TAR" | awk '{print $1}')"
if [ -n "$EXPECTED_SHA" ] && [ "$EXPECTED_SHA" != "TODO-populate" ] && [ "$EXPECTED_SHA" != "$ACTUAL_SHA" ]; then
  echo "FATAL: bridge image sha256 mismatch. expected=$EXPECTED_SHA actual=$ACTUAL_SHA" >&2
  exit 1
fi
echo "       bridge image: $BRIDGE_TAG"
echo "       oci archive:  $BRIDGE_IMAGE_TAR"
echo "       sha256:       $ACTUAL_SHA"

# Bake digest into /etc/musu-version (extends step 6.c).
BRIDGE_IMAGE_OCI_SHA="$ACTUAL_SHA"
```

**S3.5.c — Image-size guard** (insert before existing step 10 size check at lines 437-444):

# V23.3 A1.a (wiki/380 F8 + Critic H4): image hard-budget 150MB OCI archive.
# Critic H4: distinct flag --allow-image-oversize (NOT --allow-oversize, which
# is reserved for the outer 500MB tar). A single flag conflation would let an
