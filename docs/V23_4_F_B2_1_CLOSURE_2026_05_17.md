## §1 What shipped

F-B2-1 closes the V23.3 B2 cross-route DoS forward-pointer by adding a
`setInterval`-based 30-day retention sweeper on the `install_attempt` table
(schema v42, introduced by wiki/391). Without retention, a single-IP attacker
rotating `install_id` (free `[Guid]::NewGuid().ToString("N")` PowerShell call)
fills the 1GB Fly `/data/telemetry.db` volume in <1 day → SQLite write failure
→ HMAC-authed `/install`, `/nat_pierce`, `/agent_spawn` routes also break
(shared volume). F-B2-1 bounds row growth at the table level so the cross-route
blast radius from B2 cannot materialize.

### 1.1 Server-side surface (Builder commit `809015a`)

- **Module header update** at `musu-relay/src/signaling/telemetry.ts:9-14` —
  rewritten from single-line `// Retention: 90 days raw; aggregated metrics
  retained indefinitely` to 2-class per-table retention documentation: 90-day
  raw for v40 tables (`telemetry_install` / `telemetry_nat_pierce` /
  `telemetry_agent_spawn`, V23.5 sweeper TODO) vs 30-day raw for
  `install_attempt` (v42, F-B2-1 enforced). Per Critic C4 in wiki/425 §11.2.
- **Sweeper constants** at telemetry.ts:419-428 — `INSTALL_ATTEMPT_RETENTION_MS
  = 30 * 24 * 60 * 60 * 1000`, `INSTALL_ATTEMPT_SWEEPER_INTERVAL_MS = 60 * 60 *
  1000` (1-hour cadence), `INSTALL_ATTEMPT_SWEEPER_BATCH = 1000` (per Critic
  C9 — bounds event-loop block per tick). Operator-safety-hatch preamble
  comment at telemetry.ts:419-425.
- **`_runInstallAttemptSweeperOnce()` export** at telemetry.ts:432-445 — uses
  `DELETE FROM install_attempt WHERE id IN (SELECT id FROM install_attempt
  WHERE received_at < ? LIMIT ?)` (IN-SELECT-LIMIT pattern; SQLite
  `DELETE...LIMIT` requires a compile-flag this build does not enable).
  Returns `info.changes` count for direct test access and observability.
- **`_maybeStartInstallAttemptSweeper()` export** at telemetry.ts:447-460 —
  three-gate guard: (1) hatch check `MUSU_INSTALL_ATTEMPT_SWEEPER_DISABLED ===
  "1"` at telemetry.ts:449 FIRST (per Critic C12), (2) `NODE_ENV !==
  "production"` short-circuit (test-env exempt; jest hangs on dangling
  intervals), (3) re-entrancy guard `if (_installAttemptSweeperTimer) return`.
  Registers `setInterval(_runInstallAttemptSweeperOnce,
  INSTALL_ATTEMPT_SWEEPER_INTERVAL_MS)`.
- **`_stopInstallAttemptSweeper()` export** at telemetry.ts:462+ — idempotent
  `clearInterval` + nulls the timer ref.
- **Wire-up** at telemetry.ts:834 inside `makeTelemetryRouter()` —
  `_maybeStartInstallAttemptSweeper()` invoked unconditionally; re-entrancy
  guard makes it safe to call on every router construction.

## §5 Operator first-run runbook (per Auditor A NEW-INFO-1)

Auditor A flagged that the first F-B2-1 deploy will trigger a backlog drain at
the per-tick LIMIT 1000 rate if production has accumulated >30-day rows since
B2 went live (wiki/391 deploy date). At 1000 rows/hour and a hypothetical
backlog of 50K rows, full drain takes ~50 hours of sweeper ticks. The data is
non-PII and the sweep is non-blocking, so this is not a correctness issue; it
is an operator-awareness issue. Three options:

- **(a) Accept multi-day drain** — do nothing. Data is non-PII, no operational
  urgency. Sweeper runs at 1-hour cadence and converges; meanwhile the volume
  continues to fill at the slow per-tuple-bucket-capped rate so the race is
  defensible.
- **(b) Temporary larger batch** — in a single-purpose commit before deploy,
  bump `INSTALL_ATTEMPT_SWEEPER_BATCH` from 1000 to e.g. 10000; deploy;
  monitor for one or two sweep ticks; revert in a follow-up commit. Reduces
  drain time roughly 10x. Trade-off: longer event-loop block per tick (Critic
  C9 sized the 1000 default for <100ms expected block).
- **(c) Out-of-band one-shot deletion** — set `fly secrets set
  MUSU_INSTALL_ATTEMPT_SWEEPER_DISABLED=1` to keep the in-process sweeper off;
  deploy F-B2-1 code; snapshot `/data/telemetry.db` for forensics; `fly ssh`
  in and issue a single-statement `DELETE FROM install_attempt WHERE
  received_at < (strftime('%s', 'now') - 30*86400) * 1000` to drain in one
  WAL-friendly checkpoint; `fly secrets unset
  MUSU_INSTALL_ATTEMPT_SWEEPER_DISABLED`; redeploy to re-enable the in-process
  sweeper for ongoing retention.

Per Auditor A explicit ops note: **before the first `fly deploy` carrying
commits `809015a` + `99e9c92`, decide whether to pre-snapshot
`/data/telemetry.db`**. Option (c) requires the snapshot; options (a) and (b)
do not but a defensive snapshot is cheap.
