§5 · Retrospective callouts

5.1 · 7th validation of [[feedback-strategic-critic-gate]]

V23.5 master plan went through four Phase -1 revisions (v1 → v2 → v3 → v4) before the Phase 1.5 Critic was invited in. The pattern is the same one the memory predicts: tech-only Critics catch tactical defects with high precision and structurally miss thesis-level questions about whether the work should exist in the proposed shape.

What the panel caught that the Critic could not. v1 thesis: "ship LLM CoS synthesis as the default path because agents need it". Christensen ERRC + Taleb fragility + Kim&Mauborgne value innovation + Drucker effectiveness debate produced a 3Z + 1X verdict against Option Y (default-on LLM), with the panel writing that Y would (a) add a recurring $1.4K/year/user cost to a self-contained product, (b) re-open the V23.6 firewall on six adjacent items (RAG, session memory, multi-tenant, etc.), and (c) couple V23.5 to an external API provider that feedback-self-contained-product explicitly rejects. The Critic, reading v3, never raised any of these — its job is plan-shape, not thesis. Reshape v4 ships Y as opt-in with the V23.7 default-on gate (wiki/479) defining the only legitimate promotion path.

5.2 · 7th–8th validation of [[feedback-plan-stage-auditor]] zero-overlap

After the Phase 1.5 Critic resolved 9 HIGH findings against the v3 plan, the plan-as-spec Auditor (quality-engineer reading the plan body as a spec) was spawned in v3 → v4. It returned 4 NEW HIGH, none overlapping the Critic's 9:

  • A1: bridge_error field in the agent-route fallback was sometimes a string ("503") and sometimes a structured object across H-2 / W-3 / W-4 — inconsistency the Critic missed because each section was self-consistent.
  • A2: C-1's filesystem scan and W-4's page route both compute the wiki-page list, with no shared utility — drift risk (a fix in one site won't propagate).
  • A3: H-4a feature-flag default ON in early draft contradicted v4 "default OFF until V23.6 dogfood" — caught reading two sections (env defaults + rollout plan) against each other.
  • A4: W-2 client-side render default vs W-3 server-side render endpoint — both exist with no decision rule for which to call when. Resolved by v4 §"Render mode" defaulting client-side and pinning server-side as a V23.6 candidate (firewall item #10).

All 4 NEW HIGH were resolved in v4 before any Builder spawned. The pattern: Critic catches plan-shape (variable names, missing fields, contract drift within a section); plan-as-spec Auditor catches cross-section invariants (additive-extension symmetry, error-contract symmetry across H-track + W-track + C-track, default-state agreement across sections). Neither subsumes the other. 8th validation of the zero-overlap principle now on record.

5.3 · HTML closure format consolidated

Three closure HTMLs now ship the same Tariq #14 + CSS-vars + 60 KB-cap pattern: wiki/447 (V23.4 Phase 4 final), wiki/469 (V23.5 W-8 mid-track closure), and wiki/470 (this document). The pattern crystallises:

  • Self-contained. Inline CSS, inline SVG, no external font / image / script. Renders identically over HTTP, file://, and offline.
  • Theme. CSS variables + prefers-color-scheme auto-switch. No JS theme toggle.
  • Token-cheap inline SVG. Wave maps + pipeline diagrams render as ~150-300 LOC SVG, expressive without bloating the doc.
  • Markdown stays for everything else. Per feedback-scribe-html-only: only the Phase 7 final closure ships HTML; impl plans / master plans / qual evals / Critic findings / audit reports all stay Markdown to preserve LLM↔LLM handoff diff + line-citation.

V23.6+ Scribe convention is therefore set: produce one HTML per master plan close, Markdown for everything in between.
