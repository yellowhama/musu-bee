## What was validated locally

Built `musu-relay/Dockerfile` against the current `v22/gap-analysis` branch (commit df79004) and ran the resulting image with a real Docker volume to simulate Fly's volume mount. The qual-eval doc (wiki/359) flagged three failure modes as "unverified" — all three are now verified:

### ✅ HIGH #1 entrypoint chown + privilege drop

```
$ docker top musu-b0-pos
UID    PID    CMD
1001   941004 npm start
1001   941056 node dist/signaling/server.js
```

Primary process runs as uid 1001 (musu), not root. The entrypoint's `su-exec musu:musu "$@"` works correctly under Alpine.

```
$ docker exec musu-b0-pos sh -c "ls -la /data"
drwxr-xr-x 2 musu musu  4096 May 15 18:43 .
-rw-r--r-- 1 musu musu  4096 May 15 18:43 telemetry.db
-rw-r--r-- 1 musu musu 32768 May 15 18:44 telemetry.db-shm
```

`/data` ownership flipped from root to musu (Fly volumes mount as root by default), telemetry.db files write successfully. The qual-eval's "fix candidate" is now a **verified fix**.

### ✅ HIGH #3 boot-config refusal

Container exits non-zero. With the secret set, boots clean: `[signaling] listening on 9900`.

## One issue surfaced + decision

**Module-load ordering**: `app.use("/v1/telemetry", makeTelemetryRouter())` at `server.ts:332` runs at module top level, which triggers `openDb()` *before* `if (require.main === module) { checkTelemetryAuthBootConfig(...) }` at line 480. If `/data` does not exist (e.g. `MUSU_TELEMETRY_DB` points at a missing directory), the app crashes with a `better-sqlite3` `Cannot open database` error before the boot check fires.

**Decision: don't fix in B0.** Reasoning:
- Production environment (Fly with the mounted volume) does not hit this — `/data` always exists.
- The crash is loud and the error message is clear, so a misconfigured `MUSU_TELEMETRY_DB` is self-evident in `fly logs`.
- Refactoring to hoist the boot check above the router mount adds risk for zero production benefit.
- B1 (next workstream) touches this exact area; if anything, B1's audit pass can re-evaluate the ordering then.

Recorded as INFO, not HIGH. No code change in B0.

## What remains user-side (cannot be done from this session)

These are the steps documented in `wiki/360 §B0`. Each requires Fly.io operator credentials:

1. `fly auth login` (browser-based OAuth)
2. `fly launch --no-deploy --copy-config` from `musu-relay/` directory — consumes the committed `fly.toml`
3. `fly volumes create musu_telemetry --size 1 --region <r>`
4. `fly secrets set MUSU_VALIDATION_API=https://musu.pro/api/v1/nodes/validate MUSU_TELEMETRY_SHARED_SECRET=$(openssl rand -hex 32)` — **save the secret externally** for the eventual B4b installer build
5. `fly deploy` — expect `[signaling] listening on 9900` in `fly logs`
6. `curl https://<app>.fly.dev/health` — expect `200 {"status":"ok"}`
7. Negative test: `fly secrets unset MUSU_TELEMETRY_SHARED_SECRET` → next deploy refuses with the FATAL line; re-set to restore
8. Open `/spike-demo` on musu-bee with `NEXT_PUBLIC_MUSU_SIGNALING_URL=wss://<app>.fly.dev/signaling`, confirm phase `welcome-received`

Steps 1–7 are deterministic given the local validation above; step 8 is the only remaining unknown (browser-side cross-region WebSocket from a real client).
