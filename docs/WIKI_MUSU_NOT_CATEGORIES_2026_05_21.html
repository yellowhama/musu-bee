§3 4 expert reasoning 요약

🔨 Christensen — JTBD divergence

3 카테고리 모두 different JTBD:

  • HPC GPU mesh = "GPU compute aggregation" (8 GPU → 1 거대 GPU)
  • SSI cluster = "resource pooling for compute scarcity" (broken/older PC 합쳐 한 대 강한 머신)
  • VM clustering = "OS-level portability + HA" (live-migration 보장)
  • musu = "agent fleet orchestration for 1 operator across heterogeneous capability nodes" (4060Ti = LLM inference, 5070Ti = video render, laptop = mobility)

Christensen verdict: musu user 가 hire 하는 job 은 위 3 중 어느 것도 아님. specialized working nodes 의 fleet 통합 control, pooled commodity hardware 가 아님.

🎲 Taleb — antifragile via negativa

3 카테고리 모두 musu 도입 시 fragility 추가:

  • HPC: vLLM (Python ML stack 3GB), RDMA HW ($500+), Argo (200MB+) — [[feedback-no-python]] + [[feedback-self-contained-product]] 위반
  • SSI: K3s control plane 200MB+, Argo upgrade cycles, RBAC complexity — V23.4 Phase 4 panel 이 이미 reject
  • VM: SAN/NAS SPOF (storage fail = all VM dead), $$$ 10-100GbE switch, vendor CPU homogeneity 봉쇄

Taleb verdict: 3 카테고리 합쳐 13+ 기술 중 musu 도입 권장 = 0. musu 의 antifragile property = HTTP/JSON + Tailscale + sqlite + single binary + heterogeneous OK.

🧭 Drucker — scope discipline + 1 YELLOW

musu 가 3 카테고리 중 어느 것으로 expand 해도 dilute 되는 것:

  • V24 Rust 단일 binary thesis
  • V25-OPS self-contained property
  • "MUSU operates the machines" 의 "operates" 의미 (agent fleet routing 이지 OS lifecycle 관리 아님)

Drucker 의 1 YELLOW: mesh router 가 현재 static agent → node mapping — "4060Ti down 일 때 5070Ti fallback" 같은 graceful degradation 이 manual TOML 편집 필요. V25+ 에서 capability-label-based selector (예: capability=cuda_24gb) 추가는 SSI 의 resilience 만 흡수하되 node-aware identity (audit.db separation) 유지 가능. K3s 도입 아님, mesh router 의 label scheduler 만. narrow scope.

§10 V26 acceptance scenario: Qwen → ComfyUI → operator (3 PC chain)

본 시나리오가 단계 1+2+3 의 정확한 acceptance bar. user 가 정리한 4 component:

§10.1 Brain (Comp 1) — LLM serving API

  • 기술: Ollama / vLLM / LM Studio — OpenAI-compatible REST API
  • 역할: Comp 1 GPU 에 Qwen 올리고, 다른 PC 가 http://[comp1]:11434/v1/chat/completions 같은 endpoint 로 호출 가능하게 expose
  • musu integration: 새 adapter ollama_local (또는 openai_compat_local) 추가 — V24 Rust adapter chain 의 일관 패턴

§10.2 Action (Comp 2) — Application API 제어

  • 기술: ComfyUI 자체 API (REST + WebSocket) + Python/Node requests client
  • 역할: Comp 2 의 :8188/prompt endpoint 로 workflow JSON POST → execution → /history + /view 으로 결과
  • musu integration: ComfyUI 자체가 이미 HTTP server — musu 가 추가 install/wrap 안 함. operator 가 manual setup. musu 의 agent 가 backend HTTP 호출만

§10.3 Organization (Comp 3 → Comp 1 → Comp 2 chain) — Message broker

이 부분이 musu 의 진짜 핵심. user verbatim: "musu 핵심 (Step 2)". chain orchestration 가 어디서 일어나나:

  • option A — Redis Pub/Sub + Queue:
    • Comp 3 에서 "고양이 사진 그려줘" → Redis queue push
    • musu '매니저 에이전트' 가 queue 감시 → Comp 1 Qwen API 호출 ("ComfyUI workflow JSON 만들어줘")
    • Qwen 결과 → Comp 2 ComfyUI /prompt POST
    • 결과 polling → Comp 3 으로 return
  • option B — MQTT broker: Pub/Sub 패러다임 동일, IoT-friendly
  • option C — FastAPI 중앙 매니저 (HTTP/WebSocket): musu-bridge 가 이미 그 역할 (V23.4 T2-A' asyncio + SQLite workflow runner SHIP). 별도 broker install 불필요

musu 의 권장 path — option C (musu-bridge 가 이미 broker):

  • V23.4 T2-A' workflow runner 가 chain orchestration backbone
  • Redis 추가 install 0 = [[feedback-no-yagni-architecture]] + [[feedback-self-contained-product]] honor
  • broker 가 musu-bridge 안에 native → 외부 dep 없이 chain build 가능
  • scale 증가 시 (>10 회사 동시 + >1000 task/min) Redis 도입 검토 가능, 현재 scale 에서는 over-engineering

§10.4 Telemetry (모든 PC) — 상태 보고

  • 기술: psutil (CPU/RAM) + pynvml (NVIDIA GPU)
  • 역할: 각 PC 가 1 초 (또는 5 초) 간격으로 자기 hardware state + 현재 task 를 JSON 으로 packing → 중앙 (musu-bridge audit.db) 에 heartbeat
  • musu integration:
    • 현재 musu = /api/system/stats + /api/system/circuit-breakers + heartbeat (V21.B WatchDispatcher) 가 partial. pynvml 통한 NVIDIA GPU live state (VRAM 사용률, 온도) 가 gap
    • V26 sub-WS 후보: musu-bridge 의 system stats endpoint 에 GPU live metric 추가 (Rust + nvml-wrapper crate, ~200 LOC)
    • Comp 3 의 musu-bee UI 가 fleet GPU map view 표시 (현재 V21.F machine view 의 capability extension)

§11 "유기적으로 촥촥" — Process Management + Workflow Orchestration (2026-05-21 user 답 정리)

user 가 본 시나리오 build 시 실제 발생하는 고통을 정확히 짚음:

"파이톤으로 하면 엄청 중첩되서 돌아가고, 막 1도 켜고 2 켜고 3도 키고 난리나던대 (...) 코드가 스파게티처럼 꼬이고(중첩), 컴퓨터 하나 꺼지면 전체가 멈추고, 일일이 터미널 열어서 1번 켜고 2번 켜는 '수동 노가다 지옥'"

본 고통 = musu thesis 의 정확한 출발점. 산업 표준 (K3s/Proxmox) 의 답이 아니라 **소프트웨어 패턴** 의 답:

§11.1 Process management — "1도 켜고 2 켜고" 지옥 탈출

  • 고통: 각 PC 마다 사람이 터미널 열어 Ollama / ComfyUI / agent 수동 시작. 죽으면 모름. PC 재부팅 시 모두 다시.
  • 패턴: process supervisor 가 모든 worker daemon 을 백그라운드에서 띄우고 죽으면 auto-restart
  • 옵션:
    • PM2 — Node.js 도구지만 Python script 도 supervise OK. user 가 선호 인용
    • systemd — Linux native. service file 등록 후 enable
    • launchd — macOS native
    • Windows SCM — Windows service
  • musu 현재 상태: ship — V24-R6 installer 가 systemd / launchd / Windows SCM 3 backend 직접 service registration. musu-bridge / musu-bee 자체는 service 로 install. operator worker (Ollama daemon, ComfyUI server, custom agent script) 도 같은 mechanism 으로 등록 가능
  • V26 gap: musu install 이 operator 의 추가 worker (Ollama, ComfyUI) 도 함께 register 하는 helper (예: musu service register --name ollama --start "ollama serve"). PM2-equivalent 한 줄 명령으로 추가 가능. ~150 LOC Rust

§11.2 Task queue — "중첩되서 난리나는" 동기식 호출 지옥 탈출

  • 고통: Comp 1 이 Comp 2 에 requests.get() 으로 직접 호출 → Comp 2 가 끝날 때까지 Comp 1 blocking. 여러 사용자 호출 들어오면 중첩 + 멈춤
  • 패턴: 중앙 우편함 (queue) + worker 가 pull. Comp 2 바쁘면 queue 에 쌓임, 끝나면 다음 꺼냄. 절대 중첩 안 됨
  • 옵션:
    • Celery + Redis — Python 생태계 가장 정착. user 인용
    • BullMQ + Redis — Node.js 동일 패러다임
    • RabbitMQ — heavier, enterprise
    • NATS — modern, lightweight
    • musu-bridge 의 SQLite-backed asyncio queue — V23.4 T2-A' SHIP 한 자체 runner
  • musu 현재 상태: partial ship — V23.4 T2-A' workflow runner 가 정확히 이 역할 (asyncio + SQLite queue, single-process). single-machine OK, cross-machine workflow 는 file-based handshake 또는 HTTP push 필요
  • musu 의 trade-off:
    • Redis 도입 = no — [[feedback-self-contained-product]] + [[feedback-no-yagni-architecture]] 위반. 외부 daemon dep 추가 + airgap installer 부담
    • musu-bridge native queue 확장 = yes — 이미 있는 V23.4 T2-A' runner 를 cross-machine 으로 확장. task push 가 HTTP forward → 그 node 의 local queue 로 들어감. Redis 와 같은 semantic, 외부 dep 0
  • V26 gap: V23.4 T2-A' single-process runner 의 cross-machine task handoff. mesh router 가 task 의 capability requirement 보고 적합 node 로 forward, 그 node 의 local runner 가 pull. ~400 LOC Rust

§11.3 Workflow orchestration — DAG retry + 가시화

  • 고통: chain 중간 step 실패 시 처음부터 재실행 vs 그 step 부터 재시도 선택 불가. flow 가 코드에 박혀서 visibility 0
  • 패턴: DAG (Directed Acyclic Graph) 정의 + per-step retry policy + flow visualization
  • 옵션:
    • LangGraph — LLM workflow 전용. node = LLM call / tool call, edge = conditional
    • Prefect — general workflow, Python-first
    • Apache Airflow — heavy, enterprise scheduler
    • Temporal — durable execution, strong consistency
    • musu-bridge 의 asyncio + SQLite DAG runner — V23.4 T2-A' SHIP, V23.4 T2-D-mini form-based builder
  • musu 현재 상태: ship — V23.4 T2-A' workflow runner + V23.4 T2-C fleet view UI + T2-D-mini workflow builder. DAG 정의 + step retry + audit 모두 SHIP
  • V26 gap:
    • 현재 workflow = 단일 회사 안 step 들. cross-machine step (예: step 1 Comp 1 Qwen / step 2 Comp 2 ComfyUI) 은 §11.2 cross-machine task handoff 와 결합 필요
    • workflow template store (회사별 reusable workflow) — wiki/507 §10.7 Gap 3 와 동일
    • LLM-friendly DAG builder (Qwen 이 직접 workflow JSON 생성 가능하게) — V26 Domain B "에이전트 고도화" 의 한 형태
