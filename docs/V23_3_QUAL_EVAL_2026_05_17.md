## §3 What worked (carry forward to V23.4)

### §3.1 Dual-audit consistently yielded orthogonal HIGHs that single-audit would have missed

The agent-team master plan §5.1 paid out in three concrete cases:

**A1.a dual-audit (security-engineer + quality-engineer)**: Auditor A (`quality-engineer`) returned SHIP-OK with 2 MEDs (OCI label "unknown" + dev-debris in `/app/`). Auditor B (`security-engineer`) independently returned **HOLD with 2 NEW HIGHs**: S-A1a-H1 (info disclosure — `.dockerignore` at `musu-bridge/` wasn't read by buildah because context is repo root; `.musu/qa_result_*.json`, `_legacy_mattermost/`, `instructions/*.md`, `.env.example`, `Dockerfile` all shipped into the image) and S-A1a-H2 (production strict-token check was inside `if __name__ == "__main__":`, unreachable because container CMD is `python -m uvicorn server:app`). Auditor A's quality-domain scan would not have caught either: H1 needed an image-content-disclosure mindset; H2 needed the seemingly-trivial reading of "container CMD is uvicorn, not python -m server". Both became mandatory pre-push fixes (`.dockerignore` moved to repo root + `server.py:208-217` added top-level production-mode token check).

**A1.c dual-audit (`quality-engineer` × 2 with split seeds)**: Auditor A (bench-protocol-validity seed) caught NEW HIGH A-H1: bench-pod.yaml `secretKeyRef.key: token` did not match A1.b's actual Secret schema (`MUSU_BRIDGE_TOKEN`). Bench pod would have failed at `MountVolume.SetUp` with substrate-irrelevant cause, producing a false-negative Const VI verdict on the very first operator bench run. Auditor B (Const-VI-verdict-math seed) caught NEW HIGH B-H2: `success_rate ≥ 99.5%` gate-eligibility filter was missing from `bridge-bench.sh`; a bench run with 50% 5xx responses would have produced a verdict computed on the surviving 50%, silently violating Const VI gate eligibility. Both HIGHs were orthogonal — protocol-validity vs verdict-math — and would not have surfaced together under a single Auditor.

**B2 dual-audit (security-engineer + quality-engineer)**: Auditor A (`security-engineer`) caught NEW-MED-1 (`OS_VERSION_RE = /^[a-zA-Z0-9._\s()-]{1,128}$/` admits `\n`/`\r`/`\t` via `\s` class → CRLF injection / ANSI-escape smuggling in future admin log viewers) and NEW-MED-2 (unbounded `install_attempt` row growth → `/data` Fly-volume exhaustion that crosses out of B2's "isolated telemetry" frame into a DoS on the authed `/install` route via shared SQLite db). Auditor B (`quality-engineer`) caught NEW-MED-3 (DB-write absent try/catch) + resume-path state-file gap + 3 LOWs. The two Auditors' HIGH/MED findings were unioned per MODE_Agent_Team.md dual-audit contract; NEW-MED-1 became audit-fix1 (`ebf3445`), the rest became V23.4 forward-pointers F-B2-1 through F-B2-3.

**Pattern**: Each dual-audit produced ≥2 disjoint findings that neither single auditor would have surfaced. The cost (extra Auditor invocation per high-stakes sub-WS) is amortized against the cost of shipping the missed HIGH and discovering it in production.

### §3.2 Audit-fix authority extended to plan-spec defects, not just code defects (B2 precedent)

B2 audit-fix1 (`ebf3445`) is the V23.3 precedent for an audit-fix loop correcting **both** the implementation **and** the plan specification in the same commit. The Critic (Phase 1.5) specified `OS_VERSION_RE = /^[a-zA-Z0-9._\s()-]{1,128}$/` in wiki/390 §4.6's mandatory-acceptance-criteria table; the Builder (`backend-architect`) implemented the regex to spec; Auditor A (`security-engineer`) caught that the spec itself was wrong (`\s` admits `\n`/`\r`/`\t`); the audit-fix1 commit landed both (a) the corrected regex in `telemetry.ts:285` (literal SP instead of `\s`), (b) a plan-body amendment row in wiki/390 §4.6 documenting the spec correction, and (c) a regression test T13b proving the tightened regex still 204s the request (preserving non-blocking semantics) while NULL-ing the DB column and emitting `console.warn`.

This is the protocol that audit-fix loops can amend plan §-anchors when the Critic-time specification was itself defective, not just the Builder's translation of the spec. Without this protocol, the orchestrator would have had to either (i) revert the Builder commit, re-enter plan mode, re-spawn Critic, re-spawn Builder, or (ii) silently land the wrong regex with a closure-doc footnote. Neither is acceptable.

**Recorded for V23.4**: audit-fix commits MAY edit plan §-anchors in the same commit as the code fix, provided the closure doc explicitly records the plan-amendment row alongside the code-change row (B2 wiki/391 §4 NEW-MED-1 row demonstrates the pattern).

### §3.3 Phase 1.5 Critic gate caught real architectural issues before Builder spawned

B2 Critic (`security-engineer`) returned 2 HIGH + 5 MED + 3 LOW + 2 INFO on the detail plan (wiki/390 §11). The two HIGHs:

- **C-B2-H1**: `req.ip` is the Fly edge proxy IP, not the real client IP, unless `app.set('trust proxy', ...)` is configured. Without this, the per-IP rate-limit dimension is a no-op — an attacker can rotate `install_id` (which is free-form) to mint unlimited buckets while the per-IP key collapses to one bucket. Resolution: `server.ts:324` adds `app.set("trust proxy", 1)` immediately after `const app = express()`, with positive defensive side-effect that future code paths reading `req.ip` on `/install`, `/nat_pierce`, `/agent_spawn` also see real client IPs.
- **C-B2-H2**: Const III gate operator-workflow sequencing was inconsistent between plan §5.1 and §10.2 — push to `v22/gap-analysis` does NOT trigger Fly deploy; only main-merge does. Resolution: plan §5.1 rewritten to a 6-step workflow making the env-var-seed-before-deploy ordering explicit and surviving as the canonical Const III operator workflow.

Without Phase 1.5, H1 would have shipped as a silent rate-limit-collapse vulnerability discoverable only by an attacker; H2 would have shipped as an operator-time confusion that could have caused Fly to attempt the v42 migration without the env-var set, producing a boot-time throw + rolling-deploy abort (still safe but operator-disruptive).

A1.b Critic (`system-architect`) returned 6 HIGH on the K3s manifest plan. All six were architectural (fsGroup no-op on hostPath, busybox xxd lacks `-c`, CrashLoopBackOff before Secret seeding, single-tenant trade-off undocumented, terminationGracePeriodSeconds vs server.py timing mismatch, containerd resolver bare-ref bug). All six landed as plan edits before Builder spawn; the Builder's `2ec6c99` commit shipped clean.

**Pattern**: Phase 1.5 Critic raised an average of ~3 HIGHs per non-mechanical detail plan across V23.3. The orchestrator-time cost per HIGH resolution was ~5-10 minutes of plan edits (V23.2 qual eval §4.1 estimate held). The avoided Builder-time cost per HIGH (failed first build, confused Builder, regression debug) is conservatively 30-60 minutes plus risk of the HIGH slipping past Audit.

### §3.4 Master plan §9 #10 "silence on MEDIUM is blocking" policy forced explicit V23.4 forward-pointer registry

Per master plan §9 #10 (added during V23.2→V23.3 transition per V23.2 qual eval §3.2 lesson 2): "any audit MEDIUMs not fixed inside V23.3 are explicitly documented in wiki/397 as V23.4 follow-ons with target plan IDs reserved at wiki/406+. Silence on a MEDIUM is blocking." This policy forced the cross-cutting audit XC-M2 (described in §4.4 below) to require an aggregated V23.4 forward-pointer registry, which now lives in wiki/396 §5.

Without the policy:

**Pattern**: blocking-on-silence forces every Auditor's MED-and-above finding to either land as audit-fix or become a tracked forward-pointer with a reserved wiki ID. The cross-cutting audit + final closure §5 registry is the structural enforcement.

## §4 What didn't work / process slips

### §4.1 Master plan acceptance-language drift vs sub-WS plan-phase scope narrowing

Master plan §9 #4 originally read "A1.c Const VI bench RECORDED" as an acceptance criterion. The A1.c detail plan (wiki/384) and closure (wiki/385) correctly scoped A1.c to harness-only — the bench harness is READY but the bench execution itself is operator-side, with the verdict declaration RESERVED for the first operator run. The closure recorded this scope correctly. **But the master plan §9 #4 acceptance row was not edited in lockstep with the sub-WS plan-phase scope narrowing.** Cross-cutting audit XC-M1 (security-engineer cross-WS pass) caught the drift between master plan acceptance language and what A1.c actually shipped.

**Root cause**: When the A1.c Researcher Phase 0 (and subsequent Critic round 1) narrowed the scope from "ship bench results" to "ship bench harness; first operator run records verdict," the orchestrator edited the A1.c detail plan but did not back-propagate to master plan §9 #4 in the same edit pass. The two acceptance languages drifted.

**Lesson recorded for V23.4**: When a sub-WS scope narrows during Critic/plan-edit phase, the master plan acceptance row for that sub-WS MUST be edited in the same commit as the sub-WS plan edit. The cross-cutting audit's XC-M1 finding became the forcing function in V23.3; V23.4 should bake this into the orchestrator's plan-edit workflow rather than relying on post-hoc audit catch.

### §4.2 Bridge image hash drift across buildah cache state

A1.b §7 documented that the inner `musu-bridge` OCI image's sha256 is NOT stable across buildah cache state changes, even with byte-identical source. A1.c §6.3 carried the same observation forward, extending the drift table:

| Run | Bridge OCI sha256 | Manifest digest |
|---|---|---|
| A1.a-fix archive (00:57) | `7e807ea2…` | `sha256:ad3097cd…` |
| A1.b Builder Build A (02:04) | `d73c7bcb…` (drifted) | `sha256:760c1ed2…` |
| A1.b final canonical Build A==B (02:33) | `f0c79d96…` | `sha256:b484bde2…` |
| A1.c final canonical Build A==B (~12:00) | `f2d05ad9…` | `sha256:e740ddd8…` |

**Root cause**: `python:3.11-slim-bookworm` base layer's apt metadata + buildah's internal staging cache aren't fully byte-deterministic across cache clears. `musu-bridge/`+`musu-core/` source is byte-clean (commit HEAD `3830e0d`); the drift comes from base-layer state.

**Mitigation in V23.3**: Outer-tar reproducibility (the actual Const VI gate object) is preserved — bridge image is pinned to the actual baked sha256 at build-time, manifest.yaml + musu-bridge.yaml stay in lock-step. The outer tar is byte-identical Build A == Build B intra-hour at every sub-WS canonical point.

**Deferred to V23.4 (F-A1c-3)**: pin `python:3.11-slim-bookworm` to digest `sha256:70fc1e698a0ef0f0bfdd63a5370d76c598ee98cb297a5cf30871607e52663c6d` + explicit apt-archive-clear in Dockerfile to make the bridge image itself byte-identical across cache states.

**Lesson recorded**: "Outer-tar Const VI gate PASS does not imply inner-image hash stability." This is a substrate-quality concern that needs naming in V23.5 reproducible-build attestation scope (apk snapshot mirror + npm registry replay + in-toto attestation per wiki/392 §10.2). For V23.4, the digest pin + apt-archive-clear is the targeted fix.

### §4.3 Builder subagent silent hang in A1.c

A1.c Phase 3 first `devops-architect` subagent invocation hung silently — returned no envelope after the token-budget interval per MODE_Agent_Team.md token-budget guardrails. Per the escalation rule ("Subagent returns silent/off-topic → do NOT re-spawn the same subagent type. Escalate to user with the empty result attached"), the orchestrator escalated and the user authorized orchestrator-direct fallback. Commit `fc61524` was authored by the orchestrator following wiki/384 §3 implementation steps verbatim.

**Root cause** (probable): the subagent expected a notification from a job it had not launched. The subagent's instructions were to spawn the Builder work and report on completion; it appeared to wait on an external signal rather than execute the work directly.

**Lesson recorded for V23.4**: Builder subagent prompts SHOULD explicitly state "you are responsible for executing the work and returning the universal envelope; do NOT wait for external notifications." This was implicit in the universal envelope contract (MODE_Agent_Team.md "Every subagent MUST return this output envelope") but the A1.c failure shows the implicit assumption is not load-bearing. Add explicit no-wait-on-external-signal language to Phase 3 Builder spawns.

### §4.4 Cross-cutting audit XC-M1 and XC-M2 (V23.3 introductions)

The V23.3 closure-time cross-cutting audit introduced two NEW MEDs that V23.2 did not surface:

- **XC-M1**: master plan §9 #4 vs A1.c closure scope drift (described §4.1 above) — process-level finding, not code-level.
- **XC-M2**: aggregated V23.4 forward-pointer registry was missing from V23.3 final closure draft. Resolution: wiki/396 §5 added as a single canonical list of all V23.4 carries from every sub-WS, with reserved wiki IDs (wiki/406+) pre-allocated.

XC-M2's existence is itself a process improvement: V23.2 had no equivalent aggregated registry; V23.4 carries were scattered across sub-WS closure footnotes. The XC-M2 finding forced V23.3 to produce a single source of truth that V23.4 plan-mode can read directly.

**Lesson recorded**: cross-cutting audit at V23.3 closure is structurally valuable — it catches per-sub-WS-closure-blind-spots (no sub-WS Auditor was looking at "is the master plan still consistent" or "is there a single forward-pointer registry"). Continue the cross-cutting audit pattern in V23.4 final closure.

### §4.5 Wiki-ID collision risk during B2

Master plan §6 reservation table allocated wiki/390 to B2 detail plan and wiki/391 to B2 closure. During B2 execution, when the B2 Critic returned wiki/390 §11 Critic findings, the orchestrator had to double-check the reservation table because B2 shipped in a different chronological order than initially expected (B2 was master plan ID-row 7, but landed after A3 which was row 5; collision risk existed with A2/A3 closure IDs wiki/387/389 which had already shipped).

**Outcome**: no actual collision (the reservation table was respected), but the verification step took ~2 minutes of orchestrator time per B2-related commit.

**Lesson recorded for V23.4**: when sub-WSs ship in different orders than the master plan reservation table predicted, the orchestrator MUST verify the wiki-ID reservation table is still respected before each new plan or closure commit. A scripted check ("for each new doc, grep wiki/396 §5 + master plan §6 for the proposed wiki ID, error if already used") is a candidate for V23.4 process automation.
