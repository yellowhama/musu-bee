## What shipped

### Schema (Constitution III approved via batch "이거 끝까지 다 해")

- **v34_resource_requests** — agent run queue. PK + `company_id` FK +
  `agent_id` FK + `priority` + `requires_json` + `affinity_json` +
  `status` CHECK constraint + `bound_machine_id` FK + lifecycle
  timestamps. Three partial indexes: pending queue, by company, by
  machine.
- **v35_machine_capacity** — 1:1 with `machines`. Heartbeated by
  bridges so capacity churn doesn't bump `machines.updated_at`.

Both idempotent (`CREATE TABLE IF NOT EXISTS`); `_v34_down` /
`_v35_down` drop indexes then table. 10 migration tests pass.

### Scheduler module (`musu_core/scheduler/`)

| File | Role |
|------|------|
| `request.py` | `Requires` / `Affinity` / `ResourceRequest` dataclasses + DB row codec |
| `capacity.py` | `MachineCapacity` + `load_all_capacities` (JOIN `machines` for status) |
| `filter.py` | pure filter — status / avoid / gpu / cpu / mem / runtime_class |
| `score.py` | pure score — base 100, +50 affinity, +VRAM bonus, +CPU idle, −queue depth penalty |
| `binder.py` | `try_bind` atomic UPDATE (v19.F.2 pattern) + `pending_per_machine` |
| `loop.py` | `SchedulerReconciler` — load → filter → score → bind → notify |

`adapters/scheduled_process.py`: drop-in replacement for
`RemoteProcessAdapter`. Posts a `ResourceRequest`, polls for binding,
then posts `/execute/process` to the bridge URL of the bound machine.
Cancels the request row on bind timeout so the scheduler stops trying.

### Acceptance test (frame v9 §5)

`tests/test_scheduler_4company_1gpu.py` — 5 tests:

1. **`test_four_companies_share_one_gpu`** — Samsung / Hyundai / SK / LG
   each request `gpu_vram_gb=4 + claude_local`. Both `m-gpu` (RTX 4060,
   8 GB VRAM) and `m-cpu` (no GPU) are online. **All 4 bind to
   `m-gpu`.** `m-cpu` is filtered out for every request. Queue depth on
   `m-gpu` = 4.
2. **`test_cpu_only_machine_filtered_when_gpu_required`** — when only
   `m-cpu` is online and the request needs GPU, the request stays
   pending (no false bind).
3. **`test_priority_jumps_the_line`** — high-priority request still
   binds even when 3 lower-priority requests already queued on
   `m-gpu`.
4. **`test_affinity_prefer_machine_honored_when_viable`** — without
   `prefer_machine`, larger free VRAM wins. With `prefer_machine=m-gpu`,
   the +50 affinity bonus overrides the VRAM gap.
5. **`test_offline_machine_filtered`** — `status='offline'` excludes
   the machine even if capacity meets requirements.

### Const VI — Scheduler overhead

`tests/test_scheduler_overhead.py` (slow). N=200 requests, 1 machine,
per-bind wall-clock latency:

```
min=0.8  median=1.0  P95=1.2  P99=3.5  max=5.3  ms
```

**Budget**: median < 50 ms, P95 < 200 ms. **Result**: 50× under
median budget, 166× under P95 budget. SQLite UPDATE + dataclass +
JSON-codec path is essentially free.

## Atomic-bind invariant (carried from 21.B)

```sql
UPDATE resource_requests
   SET bound_machine_id=?, bound_at=…, status='bound', updated_at=…
 WHERE id=? AND bound_machine_id IS NULL AND status='pending';
```

`conn.total_changes` delta dispatches the winner. Two concurrent
schedulers binding the same request → one returns True, one returns
False, the row sticks to the first writer. No SELECT-then-UPDATE.
Test: `test_try_bind_second_call_returns_false`.

## What's NOT in 21.C (intentional)

- **CEO controller per company** — still in 21.E. Today, the
  `ScheduledProcessAdapter` is invoked from existing call sites; once
  `CEOReconciler` lands, it'll insert `ResourceRequest` rows directly
  and the adapter goes away in favor of a bridge-side worker.
- **Cross-machine notify** — the dispatcher calls `notify()` on bind,
  but cross-process wake still relies on `KindSource` poll (v21.B
  cross-proc P95 = 1544 ms). The bridge's `/api/watch/subscribe` SSE
  path is wired and ready; 21.E will plug it into the machine-side
  controller.
- **Rust supervisor isolation** — 21.D.

## Independent audit (pre-commit)

Quality-engineer subagent flagged 3 BLOCKERs + several minors before
the commit landed. All blockers fixed:

| ID | Severity | Issue | Fix |
|----|----------|-------|-----|
| B1 | BLOCKER | `binder.try_bind` used `conn.total_changes` delta — perturbable by other writers | switched to `cur.rowcount` (per-statement, SQLite-documented) |
| B2 | BLOCKER | `scheduled_process` cancel-on-timeout could race a late binder, leaving an orphan `bound` row | post-cancel re-read; if `bound_machine_id` is now set, honor the bind instead of returning timeout |
| B3 | BLOCKER | `scheduled_process` 'bound' → 'running' transition didn't check rowcount; could dispatch on a cancelled row | new `_try_transition_to_running` helper using `cur.rowcount`; abort dispatch on no-op |
| m1 | minor | `score.sort` was non-deterministic on ties | added `machine_id` as secondary sort key |
| m2 | minor | P95/P99 percentile indexing was off-by-one | switched to nearest-rank (ceil) |

New test: `test_try_bind_rejects_cancelled_row` pins the audit-fix
behavior for B1.

Carried minors (deferred; document only):

- `loop.pending_per_machine` snapshot is stale by the time
  `try_bind` runs. Soft signal only — actual binding is still
  serialized by the atomic UPDATE. Acceptable per scheduler
  literature (K8s scheduler has the same property).
- `filter.gpu_vram_total_gb` not `_free_gb` — intentional. The
  total tells us "can this machine ever satisfy this request";
  `score` uses `free` to bias toward less-loaded machines. Future
  21.D supervisor will reserve VRAM atomically.
- `machine_capacity.last_heartbeat_at` has no index. Cheap to add
  in 21.E when MachineReconciler queries it.
