§1 What R8 was

R8 is the first-of-two JTBD gates on the R-fast path: prove that on a clean 4060Ti host the Rust musu binary boots, applies its own SQLite schema, accepts authenticated requests, registers companies for the day-2 workflows, and lands real audit-log rows. R9 will then prove the same on the 5070Ti and one cross-machine task hop. wiki/490 §9 lists the seven acceptance items and §9.12 layers an operator-attested closure-metric on top to prevent Goodhart-style fakes.

§3 What had to happen first — the install-clean delta

R8 plan §3.1 prescribed a pre-flight check for stale port holders. On this machine the check found a much bigger condition: the legacy Python musu install was running in production, supervised, and self-healing. Walked it backward:

  • Port 8070 holder → wslrelay.exe PID 36348 (Windows side proxy) → Ubuntu-22.04 WSL.
  • Inside that WSL: python3 -m uvicorn server:app --host 0.0.0.0 --port 8070 PID 175534, child of systemd --user PID 494, supervised by musud PID 175930 (port-daemon), with auto-update.sh re-installing the worker via pip install -e in parallel.
  • Systemd user units: musud.service, musu-bridge.service, musu-bee.service, musu-tunnel.service (cloudflared), musu-forgejo.service (self-hosted git), musu-worker.service, musu-connectsd.service, musu-autoupdate.timer, musu-backup.timer, plus their .service halves, all grouped under musu.target.

Pkill alone could not free :8070 because musud respawned children, and musu.target wanted them up. The actual clean-down sequence:

# operator-authorized: "그 설치된 무수를 지워" (delete the installed musu)
systemctl --user stop <all 11 musu units>
systemctl --user disable musu.target musud.service musu-bridge.service \
  musu-bee.service musu-tunnel.service musu-autoupdate.timer musu-backup.timer
rm ~/.config/systemd/user/musu*.{service,target,timer}
systemctl --user daemon-reload
rm -rf ~/.musu                                   # runtime state
# preserved: ~/musu-functions/ (source tree), cloudflared binary
# Windows side also clean-installed: ~/.musu moved to backup
mv C:/Users/empty/.musu C:/Users/empty/.musu.pre-v24-r8-backup-20260520-085023
mkdir C:/Users/empty/.musu/{db,data,companies}

With both Ubuntu and Windows-side .musu dirs gone, Rust musu had a true greenfield. The Windows-side backup tag is .musu.pre-v24-r8-backup-20260520-085023 — reversible if something downstream needs the old musu.db.

§10 Reverse-out path (if needed)

  1. Stop the running Rust bridge: taskkill /F /IM musu.exe (or close the background bash).
  2. Restore Windows-side state: mv C:/Users/empty/.musu.pre-v24-r8-backup-20260520-085023 C:/Users/empty/.musu.
  3. Restore Ubuntu Python install: re-enable the systemd units inside ~/musu-functions/<installer>. Source tree is preserved at ~/musu-functions/; the runtime ~/.musu would need re-init, but the systemd unit files were also removed and would need re-installation. This is one-way for practical purposes — the source is preserved, but the operational install was deleted on operator request.
