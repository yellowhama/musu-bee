Executive summary

R1 ships musu-rs::bridge, the first Rust crate in the V24 backend reshape: a single-binary HTTP bridge that natively serves 8 endpoints (/health, /v1/health/aggregate, /v1/companies, /v1/companies/{id}, /v1/companies/{id}/activate, /v1/run, /v1/tasks, /v1/nodes/online) and reverse-proxies the remaining ~73 endpoints to the legacy Python musu-bridge via an axum facade. Authentication, rate limiting, request ID propagation, dedup, audit logging and the V23.2-B1 security invariants all execute Rust-side; the facade is invoked only after auth and middleware succeed.

R1 is phase 1 of 6 in the R-fast / R-cleanup reshape that Phase -1 (business-panel-experts debate) imposed on the original V24 “big bang” thesis: deliver the smallest correct vertical slice that proves Rust ⇄ Python coexistence, then iterate. Build order followed the V24 master plan (wiki/490) verbatim: Researcher (Explore-only) → Planner (wiki/491) → Dual Critic (system-architect + security-engineer) → Builder (backend-architect) → Dual Auditor (security-engineer + quality-engineer) → Audit-fix (backend-architect) → Scribe (this document).

Status: SHIP-OK. Eight commits on v24/rust-cleanup (5 Builder + 3 audit-fix), 44 tests passing, clippy clean, release build clean. Phase 5 dual audit returned SHIP-WITH-FOLLOWUPS; the 2 HIGH and 1 MED findings flagged by the auditors were closed in audit-fix commits bdc5577, d3a812e, ac1c994. Remaining 4 MED + 5 LOW are documented for R-cleanup with concrete file:line anchors. Master-plan §9.12 (operator-attested ungameable closure) is operator-bound and is not an R1 deliverable; §9.1 install path is deferred to R6 + R8 per plan.

§6 Deviations from plan

Three small deviations from wiki/491 plan body; none are functional regressions.

  1. Relay health block stub. Plan §5.4 listed /v1/health/aggregate as composing relay status from musu-relay. R1 ships this as a stub returning {"relay": null, "reason": "deferred to R5"} in the aggregate payload. Decision rationale: musu-relay is in the V24 reshape's R5 sub-WS; pulling its health surface into R1 would force a cross-crate dependency that R1 scope explicitly excludes. noted in handler docblock
  2. disk_free_percent stub. Plan §5.4 specifies /health returns disk-free metrics. R1 ships a placeholder that returns null on Windows hosts (Linux/macOS paths work). The proper cross-platform implementation requires sysinfo crate, which R1 dep audit deemed out-of-scope (16 deps cap). tracked for R-cleanup under follow-ups
  3. sqlx feature naming. Plan §2.3 listed sqlx with "sqlite,runtime-tokio,tls-rustls" as three separate features. The current sqlx 0.8.x bundles these as the single feature "runtime-tokio-rustls". Cargo.toml uses the bundled name; behavior is identical. no test impact

§8 R2 unblocking — what R2 must deliver

R1 ships the bridge code, but several R1 handlers depend on schema-v37 tables that R2 (sqlx migrations) is responsible for creating. R1 currently runs against an existing musu-bridge sqlite file that already has these tables; once R2 lands, R1 will create them from scratch in fresh installs. R2 must deliver:

  • audit_log with v6-mapped columns: id, ts, actor_kind, actor_id, peer_addr_hash, route, method, status, request_id, payload_redacted, note (see wiki/491 §7.1)
  • companies with: id, name, slug, active, created_at, activated_at, last_audit_at
  • route_executions with: id, idempotency_key, route, method, status, response_body, created_at, finished_at
  • tasks with: id, company_id, route, payload_hash, status, created_at, finished_at, dedup_until
  • nodes with: id, kind, last_seen_at, peer_addr, capabilities_json
  • health_aggregate_cache (write-through) with: component, last_status, last_ts, last_payload_json
  • idempotency_cache with: idempotency_key (PK), route, response_body, created_at, expires_at

R2 also owns: WAL pragmas in migration (R1 sets them at pool init but migration should be authoritative), foreign-key constraints on tasks.company_id → companies.id, and one-time backfill of last_audit_at from legacy v5 schema for upgraded installs.
