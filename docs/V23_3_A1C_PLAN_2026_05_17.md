## §1 Scope + Track A vs Track B decision

A1.c closes the V23.3 Tier-A1 trio (image → manifest → cutover-bench) by:

1. **Adding the host-port surface** so that the musu-bee frontend's hard-contracted `http://localhost:8070` URL (used by ~20 route.ts files; see §11 evidence) resolves to the K3s Pod through WSL2's automatic localhost-forward of distro-bound ports.
2. **Running the Const VI capacity bench** comparing the K3s-Pod path (this V23.3 ship) against host-systemd reference numbers (informational baseline from a V21 dev-Linux host), recording pass/fail against pinned thresholds (§4).
3. **Re-verifying B6 byte-identity** of `musu-backend.tar` after the one-line manifest change (expected delta ≈ 0 bytes).

### 1.1 Adopted scope: **Track A only**

Per Researcher H-A1c-1 + handoff note #1, A1.c adopts **Track A (WSL2-import) only**. Track B (V21 dev-Linux parallel coexistence with `--prefer k3s|systemd` toggling) is deferred to V23.4+ pending a Tauri dev-host parity requirement that does not exist today.

**Rationale**:
- V23.2 prosumer install flow (`install-wsl2.ps1` + `musu-backend.tar`) provisions K3s **inside** the imported distro. The Windows operator host has **no systemd** and **no native musu-bridge process** — the imported distro IS the only execution surface.
- The "parallel-path coexistence" wording in wiki/379 §2 A1.c row and wiki/382 §10.1 implicitly assumed a V21 dev-Linux host model where host-systemd musu-bridge runs alongside the K3s Pod on the same machine. That model is irrelevant to operator deployments and only matters for developer dogfooding.
- Track A delivery is unblocked, sized small (~120 LOC of new orchestration + 1 YAML line), and satisfies the master plan's V23.3 §9 acceptance criterion #4 ("A1.c Const VI bench recorded in wiki/385 with explicit verdict") without dragging in the heavier `--prefer` flag refactor.

### 1.2 Track B deferral (forward-pointer to V23.4+)

V23.4 may revisit Track B if any of:
- A Tauri dev-host parity build lands that requires both substrates running on the same dev machine.
- Researcher A1.c-extension finds a real operator persona running both the V21 native install **and** the V23.2 WSL2 install on the same machine.
- Operator escalation requests a `--prefer k3s|systemd` flag for staging/canary use.

Until any of these fire, Track B stays out of the V23 roadmap. The plan does NOT touch `scripts/systemd/musu-bridge.service` (V23.5 retirement) and does NOT add a `--prefer` CLI flag (per A1.b INFO1 — dropped permanently in favor of substrate detection by the install pipeline).

### 2.1 Why hostPort=8070 (not 8071 + iptables redirect, not NodePort 30870)

Per Researcher OQ3 + handoff note #2:

- **hostPort=8070 direct**: chosen. Track A only means no port collision with a host-systemd bridge (Windows operator has no systemd). WSL2's automatic localhost-forward picks up hostPort-bound traffic on the distro's eth0 and surfaces it as `localhost:8070` on Windows. Zero extra moving parts.
- **hostPort=8071 + iptables redirect**: rejected. Adds musu-init complexity (iptables nat PREROUTING rule + sysctl tuning) and an extra hop in the per-request path. Only beneficial under Track B which is deferred.
- **NodePort 30870**: rejected. Requires the musu-bee frontend to either hard-contract `localhost:30870` (breaks ~20 existing route.ts files contracting `localhost:8070`) or run a separate Windows-side reverse proxy from 8070 → 30870. Either way more invasive than the one-line hostPort change.

**Collision note**: hostPort claims a port on the K3s node (the distro eth0). On a Track A operator host the only other 8070 listener possibility would be a separate Windows process binding `0.0.0.0:8070` BEFORE WSL2 forwards. This is operator-controllable and will surface as `Invoke-WebRequest` failure during §3 step 4 smoke; not a plan-time risk.

### 2.2 Why a separate bench-pod (not `hey` baked into the tar)

Per Researcher OQ4 + handoff note #3:

- **aiohttp pod (`python:3.11-slim` already pulled by bridge image lineage)**: chosen. The `python:3.11-slim` base layer is already in the airgap image cache thanks to A1.a bridge build; bench pod reuses it. Zero tar size impact. ~500ms Python startup is amortized once per bench run (warm-up phase per §5 discards first 100 requests).
- **`hey` baked into outer tar**: rejected. Adds ~3MB to the 380MiB outer tar for a build-only/test-only dependency. B6 byte-identity preserved either way, but `hey` would need a separate apk pin or static binary copy step.
- **wrk in the bench pod**: rejected. wrk's Lua scripting is overkill for the simple /health/ready GET profile here; aiohttp's `asyncio.gather` loop is ~20 LOC and produces the same RPS at the load levels we care about.

### Step 2 — Write `musu-relay/installer/bench-pod.yaml`

NEW file. Multi-doc YAML containing a single `Pod` in namespace `musu` that runs an inline aiohttp script via `python:3.11-slim` `args[]`.

**Critic-resolved scope changes** (C-A1c-H1, H2, H3 + L7):
- Bench measures **`/health/ready` only**. `/api/route` is dropped from A1.c per C-A1c-H3 option (i): /api/route requires worker sidecar (worker:false on Pod path) AND auth, and worker sidecar containerization is V23.4 forward-pointer. Const VI gate verdict asserts on /health/ready + cold-start + RSS + rollback only.
- Bench Pod **mounts `musu-bridge-token` Secret** and injects `Authorization: Bearer $TOKEN` header (C-A1c-H2 option (b) — lowest-risk; production auth surface unchanged).
- Bench orchestrator (Step 3 `bridge-bench.sh`) **temporarily sets `MUSU_DISABLE_RATE_LIMIT=1`** on the bridge Deployment before bench + reverts after (C-A1c-H1). This is environment-controlled and does not require code change.
- Percentile indexing uses `statistics.quantiles(latencies, n=100, method='inclusive')` (C-A1c-L7 fix).

Sketch:

```yaml
---
apiVersion: v1
kind: Pod
metadata:
  name: musu-bridge-bench
  namespace: musu
  labels:
    app: musu-bridge-bench
    musu/sub-ws: v23.3-a1c
spec:
  restartPolicy: Never
  automountServiceAccountToken: false
  containers:
    - name: bench
      # python:3.11-slim is already in the airgap image cache via A1.a bridge build lineage.
      # No new image pull required at run time.
      image: docker.io/library/python:3.11-slim
      imagePullPolicy: IfNotPresent
      env:
        - name: TARGET_BASE
          value: "http://musu-bridge.musu.svc.cluster.local:8070"
        - name: WARMUP_REQUESTS
          value: "100"
        - name: MEASURE_REQUESTS
          value: "3000"     # ~5min at 10 RPS
        - name: TARGET_RPS
          value: "10"
        - name: MUSU_BRIDGE_TOKEN     # C-A1c-H2 resolution: bench auth via Secret
          valueFrom:
            secretKeyRef:
              name: musu-bridge-token
              key: token
      command: ["python", "-u", "-c"]
      args:
        - |
          import os, time, json, asyncio, statistics
          import urllib.request

          # Install aiohttp from a local wheel cache if present; else fall back to
          # urllib for the simple GET path (sufficient for /health/ready bench).
          try:
            import aiohttp
            HAS_AIOHTTP = True
          except ImportError:
            HAS_AIOHTTP = False

          BASE = os.environ["TARGET_BASE"]
          WARMUP = int(os.environ["WARMUP_REQUESTS"])
          MEASURE = int(os.environ["MEASURE_REQUESTS"])
          RPS = int(os.environ["TARGET_RPS"])
          TOKEN = os.environ["MUSU_BRIDGE_TOKEN"]  # C-A1c-H2 resolution
          AUTH = {"Authorization": f"Bearer {TOKEN}"}

          def pct(sorted_lats, p):
            # Correct percentile per C-A1c-L7 fix.
            if len(sorted_lats) < 2:
              return sorted_lats[0] if sorted_lats else None
            q = statistics.quantiles(sorted_lats, n=100, method='inclusive')
            return q[p - 1]  # p=50 -> q[49], p=99 -> q[98]

          async def bench_path(session, path, n, rps):
            latencies = []
            successes = 0
            failures = 0
            interval = 1.0 / rps
            for i in range(n):
              t0 = time.perf_counter()
              try:
                async with session.get(BASE + path, headers=AUTH,
                                       timeout=aiohttp.ClientTimeout(total=10)) as r:
                  await r.read()
                  if r.status == 200:
                    successes += 1
                  else:
                    failures += 1
                  latencies.append((time.perf_counter() - t0) * 1000.0)
              except Exception:
                failures += 1
              # Pace at target RPS (best-effort; no backoff drift).
              await asyncio.sleep(max(0, interval - (time.perf_counter() - t0)))
            latencies.sort()
            return {"path": path, "n": n, "success": successes, "fail": failures,
                    "p50_ms": pct(latencies, 50),
                    "p95_ms": pct(latencies, 95),
                    "p99_ms": pct(latencies, 99)}

          def bench_path_urllib(path, n, rps):
            latencies = []
            successes = 0
            failures = 0
            interval = 1.0 / rps
            for _ in range(n):
              req = urllib.request.Request(BASE + path, headers=AUTH)
              t0 = time.perf_counter()
              try:
                with urllib.request.urlopen(req, timeout=10) as r:
                  r.read()
                  if r.status == 200:
                    successes += 1
                  else:
                    failures += 1
                  latencies.append((time.perf_counter() - t0) * 1000.0)
              except Exception:
                failures += 1
              elapsed = time.perf_counter() - t0
              if elapsed < interval:
                time.sleep(interval - elapsed)
            latencies.sort()
            return {"path": path, "n": n, "success": successes, "fail": failures,
                    "p50_ms": pct(latencies, 50),
                    "p95_ms": pct(latencies, 95),
                    "p99_ms": pct(latencies, 99)}

          if HAS_AIOHTTP:
            async def main():
              async with aiohttp.ClientSession() as s:
                await bench_path(s, "/health/ready", WARMUP, RPS)  # discard
                hr = await bench_path(s, "/health/ready", MEASURE, RPS)
                # /api/route DROPPED per C-A1c-H3 (worker:false breaks it; V23.4 forward-pointer)
                print(json.dumps({"runs": [hr]}))
            asyncio.run(main())
          else:
            # urllib fallback (slim base lacks aiohttp; this is the verified-working path)
            bench_path_urllib("/health/ready", WARMUP, RPS)  # discard
            hr = bench_path_urllib("/health/ready", MEASURE, RPS)
            print(json.dumps({"runs": [hr]}))
      resources:
        requests:
          cpu: "100m"
          memory: "128Mi"
        limits:
          cpu: "500m"
          memory: "256Mi"
```

**aiohttp availability note**: `python:3.11-slim` ships with `pip` but not `aiohttp`. Two resolution options:
- (a) Use `urllib.request` only (synchronous, harder to pace at 10 RPS but feasible for /health/ready).
- (b) `pip install --no-index --find-links=/musu/wheels aiohttp` from a wheel cache baked into the bridge image (or a sidecar volume).
- (c) Use a pre-baked image variant `python:3.11-slim-aiohttp` built into the bridge image cache during A1.a.

**Planner decision**: option (a) for V23.3 A1.c — urllib synchronous loop is sufficient for the modest 10 RPS bench load; future V23.4 cgroup/runtime bench can upgrade to aiohttp if needed. The script above keeps both code paths and falls back to urllib if aiohttp import fails. Builder may simplify to urllib-only if Critic flags the dual path as needless surface.

### Step 3 — Write `musu-relay/installer/bridge-bench.sh`

NEW file. ≤80 LOC orchestrator. Shape:

```bash
#!/usr/bin/env bash
# bridge-bench.sh — V23.3 A1.c / wiki/384
# Const VI bench: K3s-Pod musu-bridge latency/throughput against
# pinned thresholds in wiki/384 §4.
#
# Runs entirely inside the cluster via an ephemeral bench Pod.
# Output: JSON to stdout following V23_1_SPIKE_RESULT_TEMPLATE §2 shape.
#
# Usage:
#   ./bridge-bench.sh [--runs N]
#
# Exit codes:
#   0 = bench completed (caller compares against thresholds)
#   1 = pod failed to start / timeout
#   2 = bench pod ran but produced no parseable result
set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
POD_YAML="$SCRIPT_DIR/bench-pod.yaml"
NAMESPACE="musu"
POD_NAME="musu-bridge-bench"
KUBECTL="kubectl --kubeconfig ${KUBECONFIG:-/etc/rancher/k3s/k3s.yaml}"
RUNS="${1:-3}"

echo "[bridge-bench] preflight: musu-bridge Pod Ready?"
$KUBECTL -n "$NAMESPACE" wait --for=condition=Ready pod \
  -l app=musu-bridge --timeout=60s

# C-A1c-H1 resolution: disable rate limit on the BRIDGE Deployment for the
# bench window (bench load = 10 RPS × 5min = 600 RPM, 10× over the 60/60s
# default). Revert via trap on exit so a bench abort does not leave the
# Deployment with rate limit disabled in production.
echo "[bridge-bench] disabling rate-limit on musu-bridge Deployment (bench window)"
$KUBECTL -n "$NAMESPACE" set env deploy/musu-bridge MUSU_DISABLE_RATE_LIMIT=1
trap '$KUBECTL -n '"$NAMESPACE"' set env deploy/musu-bridge MUSU_DISABLE_RATE_LIMIT-' EXIT
# Wait for rollout (env change triggers Deployment update)
$KUBECTL -n "$NAMESPACE" rollout status deploy/musu-bridge --timeout=120s
# Re-wait for new Pod Ready
$KUBECTL -n "$NAMESPACE" wait --for=condition=Ready pod \
  -l app=musu-bridge --timeout=60s

ALL_RESULTS="[]"
for run in $(seq 1 "$RUNS"); do
  echo "[bridge-bench] run ${run}/${RUNS}"
  # Idempotent cleanup
  $KUBECTL -n "$NAMESPACE" delete pod "$POD_NAME" --ignore-not-found --wait=true
  $KUBECTL apply -f "$POD_YAML"

  # Wait for pod completion (Never restartPolicy → Succeeded or Failed)
  for i in $(seq 1 60); do
    PHASE=$($KUBECTL -n "$NAMESPACE" get pod "$POD_NAME" -o jsonpath='{.status.phase}' 2>/dev/null || echo "Pending")
    [ "$PHASE" = "Succeeded" ] || [ "$PHASE" = "Failed" ] && break
    sleep 5
  done

  if [ "$PHASE" != "Succeeded" ]; then
    echo "[bridge-bench] FAIL: pod phase=${PHASE} after 5min" >&2
    $KUBECTL -n "$NAMESPACE" logs "$POD_NAME" >&2 || true
    $KUBECTL -n "$NAMESPACE" delete pod "$POD_NAME" --ignore-not-found
    exit 1
  fi

  RESULT_JSON=$($KUBECTL -n "$NAMESPACE" logs "$POD_NAME" | tail -1)
  echo "${RESULT_JSON}" | jq . >/dev/null 2>&1 || { echo "[bridge-bench] unparseable result" >&2; exit 2; }
  ALL_RESULTS=$(echo "$ALL_RESULTS" "$RESULT_JSON" | jq -s '.[0] + [.[1]]')
  $KUBECTL -n "$NAMESPACE" delete pod "$POD_NAME" --ignore-not-found
done

# Emit aggregate JSON matching V23_1_SPIKE_RESULT_TEMPLATE §2 schema +
# extension fields (p99, runs[]).
echo "${ALL_RESULTS}" | jq '{
  runs: . ,
  aggregate: {
    n_attempts: (map(.runs[0].n) | add),
    success: (map(.runs[0].success) | add),
    fail: (map(.runs[0].fail) | add),
    p50_ms_median: (map(.runs[0].p50_ms) | sort | .[length/2|floor]),
    p95_ms_median: (map(.runs[0].p95_ms) | sort | .[length/2|floor]),
    p99_ms_median: (map(.runs[0].p99_ms) | sort | .[length/2|floor])
  }
}'
```

Builder validates with `shellcheck` and a dry-run loop count of 1 in the test distro before recording the canonical 3-run bench numbers.

### 4.1 Gate verdict logic (C-A1c-M7 specified)

- **PASS**: all thresholds met simultaneously on at least 2 of 3 bench runs.
- **FAIL**: any threshold missed on 2 of 3 runs OR cold-start p95 > 60s on the very first run OR first-boot wall-clock p95 > 90s on the very first run.
- **AMBIGUOUS** (1 of 3 misses): re-run protocol = `wsl --unregister musu-test` then `wsl --import musu-test <C:\Temp> <canonical-tar>` for fresh distro state; run a fresh **3-run** bench session (NOT a single re-run). If still 1-of-3 miss after the fresh re-bench → escalate per MODE_Agent_Team.md decision tree step 1 (block-on-user). 2-of-3 miss on re-run = FAIL. Rejected alternative: "1-miss = PASS with disclosure" defeats Const VI's purpose (accepting tail-latency noise as PASS is the false-negative-on-regression mode the gate exists to prevent).

### 4.2 Const VI scope clarification (Researcher OQ2)

Track A only means the bench is **Pod-only with reference systemd numbers from a different host**, not a same-time same-host comparison. Per Researcher's recommended adjudication: this satisfies Const VI gate **intent** because:

- The systemd reference numbers from a V21 dev-Linux host give the "what's the substrate cost of K3s+containerd?" calibration that Const VI exists to answer.
- The Pod-only numbers from the WSL2 distro give the "is this acceptable for prosumer deployment?" answer that V23.3 actually ships against.
- A same-host comparison would only add information about within-host noise floor, which is already absorbed by the 3-run median protocol in §5.

If Critic disagrees and insists on same-host bench, the plan escalates to operator block-on-user (request a Track B dev-Linux host setup, deferred from §1.2).

## §5 Bench protocol (concrete numbers for Builder)

Per Researcher implicit framing + plan-introduced specifics.

### 5.1 Warm-up

100 requests discarded per path. Warm-up amortizes:
- Pod first-request JIT-like effects in FastAPI's pydantic model construction.
- aiohttp connection-pool establishment.
- K3s kube-proxy iptables nat rules first-touch latency.

### 5.2 Measurement load (C-A1c-H3-followup: /api/route dropped)

- **`/health/ready`**: 5 minutes sustained at 10 RPS = 3000 requests per run.
- Total bench measurement window per run: ~5 min (excluding warm-up and tear-down).
- Per run: 3000 measured samples.
- Per bench session: 3 runs = 9,000 samples.
- `/api/route` row REMOVED per C-A1c-H3 (worker:false + auth-required; V23.4 forward-pointer when worker sidecar lands).

### 5.3 RSS sampling

`kubectl top pod -n musu musu-bridge-<suffix>` every 5 seconds for 60 seconds (12 samples) immediately after warm-up. Report mean + max RSS in MB.

**Note**: `metrics-server` may not be installed in K3s by default. If `kubectl top` returns "Metrics API not available", fall back to `kubectl exec` + `cat /sys/fs/cgroup/memory.current` (cgroup v2) or `cat /sys/fs/cgroup/memory/memory.usage_in_bytes` (cgroup v1).

### 5.4 Cold-start trials

5 trials of:

```sh
START=$(date +%s%N)
kubectl -n musu delete pod -l app=musu-bridge
# wait for Deployment to recreate
until kubectl -n musu get pods -l app=musu-bridge \
  -o jsonpath='{.items[0].status.containerStatuses[0].ready}' | grep -q true; do
  sleep 0.5
done
# Then poll /health/ready until 200
until curl -sf http://musu-bridge.musu.svc.cluster.local:8070/health/ready >/dev/null; do
  sleep 0.5
done
END=$(date +%s%N)
echo "cold_start_ms=$(( (END - START) / 1000000 ))"
```

Run 5 trials; record min, p50, p95, max.

### 5.5 Statistical reporting

For each path:
- Report `min`, `p50`, `p95`, `p99`, `max` per run.
- Across 3 runs: report `median(p99_run_i for i in 1..3)` as the "canonical p99" for gate comparison.
- Report `success_rate = sum(success_i) / sum(n_i)` across all runs; must be ≥ 99.5% to count for gate eligibility.

## §7 Risks + mitigations

### R1: hostPort:8070 might not WSL2-forward to Windows automatically

**Probability**: low (WSL2 has well-documented auto-forward of distro-bound TCP ports since WSL2 1.0). **Impact**: high (breaks the Track A contract entirely).

**Evidence** (C-A1c-L4 anchor): WSL2 auto-forward of distro-bound listeners is documented at https://learn.microsoft.com/en-us/windows/wsl/networking ("Accessing a Linux networking app from Windows (localhost)"), gated by `.wslconfig` `localhostForwarding=true` (default since WSL2 1.0). K3s default flannel-iface uses the WSL2 distro's eth0, so the chain hostPort→portmap CNI→WSL2 vEthernet→Windows localhost works under defaults.

**Mitigation**: Step 4 smoke test catches this before bench. If broken, debugging path: check `BRIDGE_HOST=0.0.0.0` env (already set per A1.b ConfigMap); check Windows-side `netsh interface portproxy show v4tov4`; check `wsl --status` for `localhostForwarding=true` in `.wslconfig`.

**Fallback**: if WSL2 doesn't auto-forward, add a `wsl --exec` socat or `netsh portproxy` step to `install-wsl2.ps1` as a follow-on (V23.4 polish; carry as forward-pointer in closure if hit).

### R2: Bench results show Pod path significantly slower than systemd reference → Const VI FAIL

**Probability**: medium. **Impact**: high (blocks V23.3 §9 acceptance #4).

**Mitigation paths** (per MODE_Agent_Team.md decision tree step 1):
- (a) Enter audit-fix loop: tune `resources.limits.cpu` upward (currently 500m = 0.5 cores), tune `MemoryMax`. To promote QoS from Burstable to Guaranteed (per C-A1c-M8 wording fix): raise `resources.requests.cpu` from 100m → 500m AND `resources.requests.memory` from 256Mi → 2048Mi (must match limits). On single-node K3s this raises scheduling-time resource reservation — verify no other workloads would be evicted.
- (b) Disable K3s features unnecessary for bridge (e.g., `--disable=traefik` already done; consider `--disable=servicelb`, `--disable-network-policy` if not needed).
- (c) Escalate to operator block-on-user: "Pod path is ~Xx slower than systemd reference for these reasons; accept anyway, or defer V23.3 ship?"

### R3: Outer tar grows beyond 380MiB after hostPort addition

**Probability**: near zero. **Impact**: low (B6 byte-identity intact regardless; size is informational).

**Mitigation**: expected delta is ~0 bytes (single YAML line change inside the existing musu-bridge.yaml file; gzip might even compress slightly differently). Step 7 captures actual delta.

### R4: Bench pod cannot resolve `musu-bridge.musu.svc.cluster.local`

**Probability**: low (CoreDNS is K3s default since 1.20). **Impact**: medium (bench can't run).

**Mitigation**: orchestrator step 3 `kubectl wait --for=condition=Ready pod -l app=musu-bridge` validates the bridge is Running before applying the bench pod. If DNS broken, bench pod logs will show `getaddrinfo` failure; fallback is to hard-code the bridge ClusterIP or use a `--add-host` style override.

### R5: `python:3.11-slim` not in airgap image cache at bench time

**Probability**: low (A1.a baked it; A1.b preserved it). **Impact**: medium (bench fails with ImagePullBackOff).

**Mitigation** (C-A1c-L6 hardened): bench pod `imagePullPolicy: IfNotPresent` + explicit precheck in `bridge-bench.sh` step 1: `crictl -r unix:///run/k3s/containerd/containerd.sock images | grep python:3.11-slim`. If missing, **abort bench** (do NOT silently retry with ImagePullBackOff hang); document missing-image as forward-pointer F-A1c-5 (bake bench-image into airgap cache during A1.a).

### R6: aiohttp not available in `python:3.11-slim` base, urllib fallback distorts timings

**Probability**: certainty (slim doesn't include aiohttp). **Impact**: low (script handles both paths; urllib synchronous loop at 10 RPS is well within capacity).

**Mitigation**: per §3 Step 2 note, Planner accepts urllib fallback for V23.3. If Critic wants true aiohttp, build a small bench-tooling layer into the airgap image cache during A1.a → carry as V23.4 forward-pointer.

### R7: bench pod's bind to `0.0.0.0` may collide with another musu-bridge listener if Step 1 hostPort fights for the port

**Probability**: false alarm (bench pod doesn't bind any port; it's a client only). **Impact**: zero.

Documented for completeness — flagged at Critic review to confirm no collision.
