### §2.1 Why B2 lands on musu-relay/signaling, not musu-pro

**v1 of this plan committed B2 to musu-pro; Critic C1 reversed it.** Reasoning:

- **Infrastructure parity**: musu-relay/signaling already ships every primitive `install_attempt` needs — `requireInstallHmac` middleware (telemetry.ts:346) demonstrates the auth-validation pattern; `requireTelemetrySecret` middleware (telemetry.ts:215) demonstrates a sibling auth scheme; express.json + rawBody capture is wired; `applyMigrations` against `better-sqlite3` is the schema-migration tool; 7 telemetry test files demonstrate the test pattern (notably `telemetry-raw-body.test.ts`). musu-pro (Next.js + Supabase) ships NONE of these — no `/api/v1/telemetry/*` routes exist, no in-code rate-limit, no schema-migration tooling against Supabase that matches musu-bee's vertical.
- **Pre-bootstrap signal**: `install_attempt` is the *pre-bootstrap* failure dump for hosts that never reach a working install. There is no DNS-vs-identity-layer argument that privileges musu-pro — the caller is a Windows PowerShell installer that has only `signaling.musu.pro` baked in as its first-reachable endpoint (per V23.2 B4b installer config). musu-pro hostname (`musu.pro` / DNS+identity layer) is logically separate but operationally unreachable for a fresh install.
- **Schema substrate**: install_attempt rows are append-only telemetry, sized identically to existing `install_failures` / `install_completed` rows. better-sqlite3 in-process on musu-relay is the natural home; Supabase would require a new vertical and offers no benefit.
- **Cost**: musu-relay placement is ~3-5d. musu-pro placement was ~5-7d because it required new infrastructure (rate-limit, raw-body capture, HMAC primitives, schema migration tooling) duplicated from musu-relay.

Trade-off considered: musu-relay placement means `signaling.musu.pro` (Fly.io) bears the unauth POST load. Mitigation: the same rate-limit `(install_id + source_ip)` design works against Fly's existing IP-based rate-limit substrate. Fly bandwidth cost for ~5 unauth POSTs per failed install is negligible.

Items intentionally dropped from wiki/378's Tier-B vs this plan:
- **B3 PSScriptAnalyzer CI** → V23.4 (Researcher HIGH-3: requires `.github/workflows/` bootstrap; 3-5d real cost; deferred to keep V23.3 ≤5w wall-clock)
- **B5 manifest URL consumption** → keep deferred to V23.4+ (LOW1 polish; not gating Tier-A acceptance)
- **B6 /etc/musu-version Node precision** from wiki/378 — kept but reordered (this plan's B6 is wiki/378's B1 SOURCE_DATE_EPOCH; the wiki/378 B6 is folded into B6 here as a tributary)

## §7 Risks + mitigations

### 7.1 A1.c cutover blast radius

**Risk**: K3s-Pod bridge ships, host-systemd bridge stays alive (parallel-path), but a host operator triggers cutover and the K3s-Pod bridge crashes / fails to bind to expected port. Service degradation between cutover trigger and rollback.

**Mitigation**:
1. `--prefer k3s|systemd` flag in `musu-bridge/cli.py` is **opt-in** (default = systemd).
2. Cutover trigger is per-host, manual, with explicit operator confirmation.
3. Rollback is `musu-bridge --prefer systemd` re-run + systemd unit kick; documented in A1.c closure.
4. Const VI bench in A1.c §5 measures rollback time (target ≤30s).

### 7.2 B2 cross-repo coordination

**Risk**: musu-pro endpoint deploys but installer caller wires up wrong URL / wrong auth shape / wrong field names; install_attempt rows never arrive in musu-pro DB.

**Mitigation**: Identical pattern to V23.2 B2-pro/B2-bee split:
1. musu-pro endpoint lands first in a deploy (Const VII gate #1 per V23.3 closure)
2. curl-smoke from any developer host before musu-bee installer caller PR opens
3. musu-bee caller PR includes a smoke test (`Invoke-WebRequest` against deployed endpoint) in install-wsl2.ps1's diagnostic block
4. Const VII gate #2 (V23.3 main-merge) blocks until smoke green

### 7.3 A2 + A3.swap main.ts collision

**Risk**: Two parallel Builder chains both edit `main.ts`, surprising merge conflicts.

**Mitigation**: §3.1 option (i) — A2 lands first; A3.swap rebases. Both edits are at well-separated line ranges (A2 at 211-218, A3.swap at 97-114). If Critic prefers option (ii), single-commit Builder pass with both diffs.

### 7.4 K3s-Pod base image choice (Alpine vs Debian-slim)

**Risk**: Alpine-musl wheels for uvicorn/fastapi/aiohttp are sparse and slow; Debian-slim adds ~50-80MB to `musu-backend.tar`.
