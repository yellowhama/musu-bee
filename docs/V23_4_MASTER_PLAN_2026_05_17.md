## §5 Sub-WS specs (concise, OQ-resolved)

### §5.1 F-B2-1: install_attempt retention sweeper (wiki/406 detail / wiki/426 closure)

**Phase 0 Researcher OQ resolutions** (from prior turn):

- **OQ1 retention window**: **30 days**. Rationale: matches Researcher recommendation; 1/3 of the telemetry module header's 90-day raw retention budget at `telemetry.ts:9`. Conservative; defensible for operational analytics window without re-litigating privacy.
- **OQ2 timer style**: **dedicated `setInterval` timer** (NOT on-hot-path). Reasoning per Researcher R1.1 + R1.4: on-hot-path opportunistic mirrors rate-limit reaper but risks O(table_size) scan once `install_attempt` grows (10K, 100K, 1M rows over months). `setInterval` is predictable + bounded-work per tick + decouples retention sweep from POST throughput.

**Implementation outline** (wiki/406 detail plan will expand; updated per Critic C1, C9, C12):

```typescript
// telemetry.ts (new constants, near telemetry.ts:308-317 rate-limit constants):
const INSTALL_ATTEMPT_RETENTION_MS = 30 * 24 * 60 * 60 * 1000; // 30 days
const INSTALL_ATTEMPT_SWEEPER_INTERVAL_MS = 60 * 60 * 1000;    // 1 hour
const INSTALL_ATTEMPT_SWEEPER_BATCH = 1000;                    // C9: bound event-loop block per tick

let _installAttemptSweeperTimer: ReturnType<typeof setInterval> | null = null;

// Exported for tests + manual triggering. Returns rows deleted this call.
export function _runInstallAttemptSweeperOnce(): number {
  const cutoff = Date.now() - INSTALL_ATTEMPT_RETENTION_MS;
  // C9: bounded per-tick to avoid >100ms event-loop block under steady-state.
  // SQLite DELETE...LIMIT requires compile-flag; use IN-SELECT-LIMIT pattern.
  const info = _db
    .prepare(
      `DELETE FROM install_attempt
       WHERE id IN (
         SELECT id FROM install_attempt WHERE received_at < ? LIMIT ?
       )`,
    )
    .run(cutoff, INSTALL_ATTEMPT_SWEEPER_BATCH);
  return info.changes;
}

// Exported for tests per C1 (so the registration path itself is covered).
export function _maybeStartInstallAttemptSweeper(): void {
  // C12 safety hatch: operator can disable for forensics / pre-snapshot.
  if (process.env.MUSU_INSTALL_ATTEMPT_SWEEPER_DISABLED === "1") return;
  // Test-env exempt: jest hangs on dangling setInterval handles.
  if (process.env.NODE_ENV !== "production") return;
  if (_installAttemptSweeperTimer) return;
  _installAttemptSweeperTimer = setInterval(
    _runInstallAttemptSweeperOnce,
    INSTALL_ATTEMPT_SWEEPER_INTERVAL_MS,
  );
}

// Test-only cleanup (mirrors _resetInstallAttemptLimiter precedent).
export function _stopInstallAttemptSweeper(): void {
  if (_installAttemptSweeperTimer) {
    clearInterval(_installAttemptSweeperTimer);
    _installAttemptSweeperTimer = null;
  }
}
```

Sweeper start invoked from `makeTelemetryRouter()` entry. Three exports for tests.

**Telemetry module header update** (per Critic C4 — wiki/425 plan-edit, NOT a Builder discretion):

Update `telemetry.ts:9` from:
```

```
to:
```

//  - telemetry_install / telemetry_nat_pierce / telemetry_agent_spawn (v40):
//    90 days raw (V23.5 sweeper TODO; currently no policy enforced)
//  - install_attempt (v42): 30 days raw (F-B2-1 / wiki/426 sweeper enforced
//    via _runInstallAttemptSweeperOnce, 1-hour cadence, batch=1000)
//  - aggregated metrics: retained indefinitely
```

This is part of the F-B2-1 commit, NOT a separate cleanup pass.

### §5.2 F-B2-2: PowerShell state file enrichment (wiki/407 detail / wiki/427 closure)

**Phase 0 Researcher OQ resolution**:

- **OQ3 state file schema_version**: **NO schema_version field added**. Backward-compat is automatic — old state files without `os_version`/`bios_vt` will resume with empty `$state.os_version` / `$state.bios_vt` (= `$null` in PowerShell), matching pre-V23.4 behavior. If V23.5+ needs structured state migration, add schema_version then (YAGNI).

**Implementation outline** (wiki/407 detail plan will expand; updated per Critic C2, C5):

**C2 latent-bug warning**: the helper at `install-wsl2.ps1:109-117` uses a **two-level guard chain** (`$script:PrereqResult -and $script:PrereqResult.PSObject.Properties["probes"] -and $script:PrereqResult.probes.PSObject.Properties["os_version"]`). Under `Set-StrictMode -Version 3`, a single-level guard collapse throws `PropertyNotFoundException` at runtime when `$script:PrereqResult.probes` is `$null` on resume. F-B2-2 implementation MUST mirror the existing two-level pattern.

Also note: the **resume-path** `Save-MusuState` call (around `install-wsl2.ps1:415`) re-serializes via `$state.PSObject.Properties`; it does NOT carry forward fields that were never written in the fresh-install save. F-B2-2 modifies ONLY the fresh-install save site; the resume-path data carry-forward relies on the fresh-install having written the fields. Old (V23.3) state files lacking `os_version`/`bios_vt` will resume with `$null` for those fields; the helper's `elseif` fallback handles this gracefully.

```powershell
# install-wsl2.ps1 ~ line 422 (fresh-install Save-MusuState hashtable — mirror two-level guard):
@{
    install_id    = $script:InstallId
    host_class    = $script:HostClass
    tar_path      = $TarPath

    musu_pro_base = $MusuProBase
    retry_count   = $script:RetryCount
    # NEW (F-B2-2) — TWO-LEVEL guard per Critic C2:
    os_version    = if ($script:PrereqResult -and `
                        $script:PrereqResult.PSObject.Properties["probes"] -and `
                        $script:PrereqResult.probes.PSObject.Properties["os_version"]) `
                    { $script:PrereqResult.probes.os_version } else { $null }
    bios_vt       = if ($script:PrereqResult -and `
                        $script:PrereqResult.PSObject.Properties["probes"] -and `
                        $script:PrereqResult.probes.PSObject.Properties["bios_vt"]) `
                    { $script:PrereqResult.probes.bios_vt } else { $null }
}

# install-wsl2.ps1 ~ lines 415-424 (resume block, restore script-scoped fallback):
$script:InstallId     = $state.install_id
$script:HostClass     = $state.host_class
# NEW (F-B2-2) — guard against old state files lacking these fields:
$script:OsVersionResumed = if ($state.PSObject.Properties["os_version"]) { $state.os_version } else { $null }
$script:BiosVtResumed    = if ($state.PSObject.Properties["bios_vt"])    { $state.bios_vt }    else { $null }
```

**Defensive helper relaxation** (`_Invoke-MusuInstallAttemptTelemetry` at ~lines 109-116) — keep two-level guard, add elseif fallback:

```powershell
$os = if ($script:PrereqResult -and `
```

### §5.3 F-B2-3: uniform DB-write try/catch (wiki/408 detail / wiki/428 closure)

**Phase 0 Researcher OQ resolution**:

- **OQ4 log shape**: `console.error(\`[telemetry] ${routeName}: db write failed: ${err.message}\`)`. Matches existing `[telemetry]` prefix pattern at `telemetry.ts:40` (ephemeral DB warning), `:433` (missing shared-secret warning), `:867` (over-length field clamp). Structured JSON logging is V23.5 horizon (separate concern).

**Implementation outline** (wiki/408 detail plan will expand; updated per Critic C3, C11):

**C11 double-response guard**: the `return` after the catch is load-bearing. Without it, control falls through to `res.status(204).end()` and Express throws "Cannot set headers after they are sent". Tests must assert no double-response in addition to the 500.

**C3 regression-scan requirement** (mandatory BEFORE F-B2-3 commits): grep `tests/*.test.ts` for any case asserting on the legacy 3 routes (`/install`, `/nat_pierce`, `/agent_spawn`) 500-response shape. Disposition documented per case in wiki/408 detail plan §"Regression scan": (a) no change needed (no test currently relies on Express-default-handler shape), or (b) test update required (re-anchor to new JSON 500 contract). If any test breaks because of F-B2-3, that is a STOP-SHIP requiring re-evaluation of the contract change.

```typescript
// telemetry.ts — applied uniformly across 4 routes (/install, /install_attempt, /nat_pierce, /agent_spawn):
try {
  _db.prepare("INSERT INTO ... VALUES (?, ?, ...)").run(/* params */);
} catch (err) {
  console.error(`[telemetry] /install_attempt: db write failed: ${err instanceof Error ? err.message : String(err)}`);
  res.status(500).json({ error: "database write failed" });
  return; // C11: load-bearing — without this, falls through to res.status(204).end() and Express throws
}
res.status(204).end();
```

Each route gets the same try/catch shape with route name in log + `return` statement.

  - Response body `{ error: "database write failed" }`

## §7 Risks + mitigations

### R1 (LOW): `setInterval` doesn't fire if Node event loop is blocked

**Mitigation**: musu-relay has no blocking work in route handlers (all DB queries are sync `better-sqlite3` returning in <1ms; no long-running tasks). If future V23.5+ adds workers or sync I/O, sweeper migrates to a dedicated worker thread. Not blocking V23.4.

### R2 (LOW): F-B2-3 try/catch changes return path

**Mitigation**: Old behavior was "Express default error handler emits HTML stack trace in dev / generic 500 in prod"; new behavior is structured JSON 500. Could break clients that pattern-match on the old error shape — but no known client does. The 4 telemetry routes are fire-and-forget from the PowerShell installer; clients ignore 5xx by design (telemetry is best-effort).

### R3 (LOW): F-B2-2 state file backward-compat

**Mitigation**: Old state files lacking `os_version` / `bios_vt` resume with `$null`. Telemetry receives empty optional fields and clamps to NULL via existing `optClamp` at `telemetry.ts:857-871`. No regression.

### R4 (LOW): Sweeper retention window debate

**Mitigation**: 30 days is OQ1-resolved with Researcher rationale. If operator later wants tighter (7 days) for privacy, change the constant — single-line edit, no schema or test churn. V23.5 horizon item if it comes up.

### R5 (LOW, per Critic C3): F-B2-3 may shift HMAC-installer retry behavior on DB-write failure

**Mitigation**: HMAC-installer routes (`recordOutcome` in `client.ts`, `emitInstallCompleted` in `main.ts`) may rely on status codes for retry policy. F-B2-3 changes failure mode from "Express default → connection drops or HTML stack trace" to "explicit JSON 500". Status-code path now reliably reports 500 instead of potentially a timeout/socket-error, which could shift retry behavior. Builder MUST re-check installer-side retry policy in wiki/408 §"Regression scan" and document any behavior delta. If installer retry assumes timeout-class failure, F-B2-3 may inadvertently increase retry rate on DB-write failures (which is benign — telemetry is fire-and-forget — but worth documenting).
