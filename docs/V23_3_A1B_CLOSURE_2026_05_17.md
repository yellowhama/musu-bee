## §1 Summary

A1.b ships the K3s Deployment + Service + ConfigMap manifest for `musu-bridge`,
baked into `musu-backend.tar` under `/var/lib/rancher/k3s/server/manifests/`
where K3s auto-applies it at first boot. `musu-init` extends to seed the
`musu-bridge` Secret (account_key + relay_token) at first start with idempotent
re-apply, then `kubectl scale deploy musu-bridge --replicas=1` to start the
Pod after the Secret exists (avoids CrashLoopBackOff before Secret creation).

## §7 Known issue — bridge image hash drift across buildah cache state

A1.b's B6 verification surfaced that the inner bridge OCI image's sha256 is NOT
stable across buildah cache state changes, even with identical source.

| Run | Bridge OCI sha256 | Manifest digest |
|---|---|---|
| A1.a-fix archive (2026-05-17 00:57) | `7e807ea2…` | `sha256:ad3097cd…` |
| A1.b Builder Build A (2026-05-17 02:04) | `d73c7bcb…` (drifted) | `sha256:760c1ed2…` |
| A1.b final canonical Build A==B (2026-05-17 02:33) | `f0c79d96…` (fresh-cache after `buildah rmi --all`) | `sha256:b484bde2…` |

**Root cause**: `python:3.11-slim-bookworm` base layer's apt metadata and
buildah's internal staging cache aren't fully byte-deterministic across cache
clears. `musu-bridge/`+`musu-core/` source byte-clean from `3830e0d` (commit
HEAD); same base image digest in both runs.

**Mitigation in this commit**: bridge image is pinned to the actual baked
sha256 at build-time; manifest.yaml + musu-bridge.yaml stay in lock-step. The
**outer** musu-backend.tar is fully Const VI byte-identical when the bridge
image is reused from cache OR rebuilt under the same cache state.

**Deferred to V23.4**: pin `python:3.11-slim-bookworm` to its digest
(`sha256:70fc1e698a0ef0f0bfdd63a5370d76c598ee98cb297a5cf30871607e52663c6d`)
and add explicit apt-archive-clear in Dockerfile to make the bridge image
itself byte-identical across cache states. Not blocking A1.b ship because the
outer-tar reproducibility (the actual Const VI gate object) is preserved.
