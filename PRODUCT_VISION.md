# MUSU 제품 비전

## 한 줄 정의

**여러 대의 내 컴퓨터 위에서 AI들이 팀처럼 일하게 하는 개인용 AI 운영체계(Control Plane).**

---

## 채팅의 포지션 (중요)

MUSU에서 “채팅”은 코어가 아니라 **원격 Web GUI surface**다.

- 기본 인터페이스: `Codex CLI` / `Claude Code` / `MCP` (각 컴퓨터에서 직접 사용)
- 채팅/웹 GUI: `musu.pro`에서 **어디서든 접속**해 상태를 보고 지시/이어받기 위한 Pro 운영면

관련 노트: `docs/NOTE_2026-04-09_chat_is_web_gui_not_core.md`

## 제품 원칙: 유저는 말만 한다

```
유저가 보는 것: 하나의 인터페이스(기본: CLI/MCP, 옵션: Web GUI/채팅)
유저가 하는 것: 말하기
나머지 전부: AI가 알아서
```

유저가 신경 쓸 건 **지금 일이 어떻게 되고 있는지** 딱 그것만.

| 영역 | 누가 하나 | 유저가 볼 일 |
|------|----------|------------|
| 대화/업무 지시 | 유저 | 항상 |
| 작업 진행 상황 | 파트장이 알려줌 | 알림으로 받음 |
| 자원 관리 (GPU/CPU/RAM) | 파트장이 알아서 | 충돌 날 때만 선택지 제시 |
| 조직 구조 (회사/프로젝트) | 파트장이 관리 | 워크스페이스 전환만 |
| 환경 관리 (도구/설정/MCP) | 파트장이 자동 | 문제 있을 때만 보고 |

**1차 유저는 AI 도구(Claude Code, Codex CLI)를 쓰는 개발자다.** CLI/MCP가 기본 인터페이스.
**2차 유저는 비개발자.** 이들을 위해 Web GUI(musu.pro)를 제공하지만, 1차 유저 경험이 우선이다.
뭔가 안 되면 파트장이 알아서 고치거나, 못 고치면 "이런 문제가 있는데 어떻게 할까요?" 하고 물어본다.

### 허락 안 구한다

일 시켰으면 알아서 한다. 매번 "이거 해도 될까요?" 안 물어본다.

```
❌ 지금 (Claude Code):
유저: "보고서 만들어줘"
AI: "파일을 읽어도 될까요?"
유저: "ㅇ"
AI: "데이터를 분석해도 될까요?"
유저: "ㅇ"
AI: "파일을 저장해도 될까요?"
유저: "ㅇ"
AI: "완료했습니다"

✅ MUSU:
유저: "보고서 만들어줘"
파트장: "완료했습니다. 📎 보고서.pdf"
```

**팀원한테 일 시켰는데 매번 허락 구하면 그건 팀원이 아니라 도구다.**

물어보는 건 **진짜 판단이 필요할 때만**:
- "디자인 A랑 B 중에 어떤 걸로 할까요?" → 유저 취향
- "서버 데이터를 삭제해야 하는데 진행할까요?" → 되돌릴 수 없는 위험한 작업
- "예산 초과될 것 같은데 계속할까요?" → 돈 문제

파일 읽기, 명령 실행, 도구 호출 같은 건 안 물어본다. 그건 일하는 과정이지 유저가 결정할 게 아니다.

### 유저가 대답 안 하면?

**기다리지 않는다. 알아서 판단하고 진행한다.**

사장이 출장 가면 팀장들이 멈추나? 아니지. 할 수 있는 일 하고, 진짜 사장 결재가 필요한 건 올려놓고 다른 일 한다.

```
파트장: "디자인 A랑 B 중에 어떤 걸로 할까요?"
(유저 10분 째 무응답)
파트장: "A가 더 나아보여서 A로 진행합니다. 마음에 안 들면 말씀해주세요."
```

```
파트장: "서버 데이터 삭제해야 하는데 진행할까요?"
(유저 무응답)
파트장: "위험한 작업이라 보류합니다. 다른 작업 먼저 하고 있겠습니다."
```

규칙:
- **안전한 판단**: 알아서 진행하고 나중에 보고
- **위험한 판단**: 보류하고 다른 일 먼저
- **절대 멈추지 않음**: 유저 응답 기다리느라 전체가 멈추는 일 없음

## 핵심 컨셉

### 유저 = 손님 (클라이언트)

유저는 일을 맡기는 사람이다. 개발자가 아니다.
카페에서 노트북을 열든, 집에서 데스크탑 앞에 앉든, `CLI/MCP(기본)` 또는 `Web GUI(옵션)`를 열고 "이거 해줘"라고 말한다.

### 기기 = 부서

각 컴퓨터가 하나의 부서다. 부서에는:
- **파트장 (AI)**: 그 기기에서 돌아가는 AI 에이전트. 일을 받고, 판단하고, 실행한다.
- **팀원 (하드웨어)**: CPU, GPU, RAM, 디스크. 파트장이 일을 시킬 수 있는 자원.

```
[노트북]              [4060Ti 데스크탑]       [5070Ti 데스크탑]
파트장: Musu-C        파트장: Musu-A          파트장: Musu-B
팀원:                 팀원:                   팀원:
- CPU 8코어           - CPU 12코어            - CPU 16코어
- RAM 16GB            - RAM 8GB               - RAM 32GB
- GPU 없음            - 4060Ti 16GB           - 5070Ti 16GB
                      - SSD 1TB               - SSD 2TB
```

### 사장 = 로테이션

유저가 말을 건 기기의 파트장이 **그 순간의 사장**이 된다.

- 노트북에서 말 걸면 → Musu-C가 사장
- 4060Ti에서 말 걸면 → Musu-A가 사장
- 사장이라고 특별한 게 아님. 그냥 유저 요청을 받아서 적절한 부서에 분배하는 역할.
- 반장 같은 임시직. 모든 파트장은 동등.

### 대화 = 업무

유저가 주 인터페이스(대부분 `CLI/MCP`, 필요 시 `Web GUI/채팅`)에 쓰는 모든 말이 업무 요청이다.

```
유저: "이 PDF 요약해줘"
사장(Musu-C, 노트북): "GPU가 필요해서 개발팀에 넘깁니다"
Musu-A(4060Ti): "받았습니다. 요약 중..."
Musu-A(4060Ti): "완료했습니다. [요약 결과]"
```

```
유저: "블로그 글 초안 써줘"
사장(Musu-A, 4060Ti): "이건 CPU로 충분하니까 제가 하겠습니다"
Musu-A(4060Ti): "초안입니다. [내용]"
```

```
유저: "이 코드 분석하고 리팩토링해줘"
사장(Musu-B, 5070Ti): "코드 분석은 제가, 테스트는 4060Ti팀에 나눕니다"
Musu-B(5070Ti): "분석 완료. 리팩토링 PR 올립니다"
Musu-A(4060Ti): "테스트 전부 통과했습니다"
```

### 왼쪽 사이드바

**채널 (자동 생성)**:
- `#general` — 모든 대화가 여기서 시작. 유저가 말 걸면 사장이 받음.
- `#tasks` — 진행 중인 작업 목록. 칸반 보드처럼.
- `#dev` — 기기 간 내부 대화 (AI끼리 협의하는 것도 볼 수 있음)
- `#alerts` — 기기 상태 변경, 에러, 완료 알림

**부서 (기기 목록)**:
- 각 기기가 부서로 표시
- 온라인/오프라인 상태
- 하위에 CPU/GPU/RAM 사용률 실시간 표시
- 클릭하면 그 기기의 상세 정보 + 직접 대화

### 채팅 영역

일반 메신저랑 똑같다:
- 유저가 메시지 보냄
- 사장(현재 기기 파트장)이 받아서 응답
- 다른 기기에 넘겨야 하면 자동으로 넘기고 경과 보여줌
- 파일 첨부, 코드 블록, 이미지 전부 지원

## 업무 분배 로직

### 자동 분배

유저가 "이거 해줘"라고 하면, 사장이 판단:

```
1. 이 작업에 뭐가 필요한가?
   - GPU 추론 → GPU 있는 기기
   - 대용량 파일 → 디스크/RAM 여유 있는 기기
   - 가벼운 텍스트 → 아무 기기나 (현재 기기 우선)

2. 지금 어디가 한가한가?
   - CPU 사용률 낮은 곳
   - GPU 여유 있는 곳
   - 작업 큐가 짧은 곳

3. 결정 → 실행
   - 현재 기기에서 할 수 있으면 바로 실행
   - 못하면 적절한 기기에 전달
   - 여러 기기가 나눠서 해야 하면 분할
```

### 수동 지정

유저가 직접 기기를 지정할 수도 있음:

```
유저: "@5070Ti 이 모델 돌려줘"
→ 5070Ti 파트장이 직접 받음 (사장 로테이션 무시)
```

## Paperclip과의 관계

Paperclip ≠ MUSU. 다른 제품이다.

| | Paperclip | MUSU |
|--|-----------|------|
| 대상 | 개발자 | 일반 사용자 |
| 모델 | 고정 CEO + 계층 구조 | 로테이션 사장 + 동등한 파트장 |
| 단위 | AI 역할 (CEO/CTO/QA) | 기기 (컴퓨터) |
| UI | 대시보드 | 업무 메신저 |
| 목적 | AI 에이전트 오케스트레이션 | 내 컴퓨터들을 팀으로 쓰기 |

Paperclip은 우리가 MUSU를 **개발하는 데** 쓰는 도구다. MUSU 자체는 Paperclip과 다른 제품.

다만 MUSU 내부에 Paperclip의 개념(이슈 추적, 에이전트 하트비트, 거버넌스)을 차용할 수는 있다.

## 조직 구조: 그룹 → 회사 → 프로젝트

대기업을 생각하면 된다. 삼성그룹 안에 삼성전자도 있고 삼성생명도 있고, 각각 건물(공장, 사무실, 지점)이 여러 채 있다. MUSU도 같은 구조.

### 계층

```
그룹 (내 MUSU 전체)
├── 회사 A
│   ├── 기기 (5070Ti, 4060Ti, ...)
│   └── 프로젝트들
│       ├── 프로젝트 1
│       ├── 프로젝트 2
│       └── 프로젝트 1 ↔ 프로젝트 2 연동
│
├── 회사 B
│   ├── 기기 (노트북, 4060Ti, ...)
│   └── 프로젝트들
│       └── 프로젝트 3
│
└── 회사 간 연동
    └── 회사 A의 결과물을 회사 B에서 사용
```

### 핵심 규칙

**기기는 여러 회사에 소속될 수 있다.**
4060Ti가 "AI 개발" 회사에서도 일하고 "콘텐츠 제작" 회사에서도 일한다. 시간/자원을 나눠서.

**프로젝트는 여러 기기에 걸친다.**
"블로그 자동 생성" 프로젝트가 노트북에서 기획하고, 5070Ti에서 이미지 만들고, 4060Ti에서 배포한다.

**프로젝트끼리 연동된다.**
"코드 리뷰 봇" 프로젝트가 "블로그 자동 생성" 프로젝트의 코드를 리뷰한다.

**회사끼리도 연동된다.**
"AI 개발" 회사에서 만든 모델을 "콘텐츠 제작" 회사에서 쓴다.

**하나의 프로젝트가 쪼개질 수도 있다.**
"앱 개발" 프로젝트가 커져서 "프론트엔드" + "백엔드" + "인프라" 세 프로젝트로 나뉜다.

### 예시

```
내 MUSU
│
├── 🏢 "AI 스튜디오" (회사)
│   ├── 🖥 5070Ti — 모델 추론, 학습
│   ├── 🖥 4060Ti — 개발, 테스트
│   ├── 📋 프로젝트: 이미지 생성 API
│   └── 📋 프로젝트: 코드 분석 도구
│
├── 🏢 "콘텐츠 팩토리" (회사)
│   ├── 💻 노트북 — 기획, 글쓰기
│   ├── 🖥 4060Ti — 이미지/영상 (AI 스튜디오와 공유)
│   └── 📋 프로젝트: 유튜브 채널 운영
│       ├── 기획 (노트북)
│       ├── 썸네일 생성 (4060Ti, "이미지 생성 API" 프로젝트와 연동)
│       └── 편집 자동화 (5070Ti)
│
└── 🔗 연동
    └── "이미지 생성 API" ←→ "유튜브 채널 운영"
        (API 호출로 썸네일 자동 생성)
```

유저가 회사를 전환하는 건 Slack에서 워크스페이스 바꾸는 것과 같다.

```

│          │                           │
│ ───────  │  채널                     │
│ 채널     │  #general                 │

## 자원 관리: 유한한 공장을 어떻게 나눠 쓸 것인가

이 제품의 핵심 문제. CPU, GPU, RAM은 유한하다. 여러 회사, 여러 프로젝트가 이걸 나눠 써야 한다.

### 공장 비유

삼성 화성 반도체 공장이 24시간 돌아가는데, 라인 A는 갤럭시용 칩, 라인 B는 엑시노스, 라인 C는 외주. 각 라인에 몇 퍼센트를 돌릴지 정하는 것과 같다.

```
5070Ti 데스크탑 = 화성 공장
├── 총 자원: GPU 16GB, CPU 16코어, RAM 32GB
│
├── 🏢 AI 스튜디오 — GPU 10GB, CPU 8코어 할당
│   ├── 📋 이미지 생성 API — GPU 6GB (라인 A)
│   └── 📋 코드 분석 — GPU 4GB (라인 B)
│
├── 🏢 콘텐츠 팩토리 — GPU 4GB, CPU 4코어 할당
│   └── 📋 유튜브 — GPU 4GB (라인 C)
│
└── 여유분: GPU 2GB, CPU 4코어 (긴급/돌발용)
```

### 할당 방식

**1. 회사별 쿼터 (예산)**

각 회사가 기기 자원을 얼마나 쓸 수 있는지 상한선을 정한다.

```
5070Ti 자원 배분:
- AI 스튜디오:    GPU 60%, CPU 50%, RAM 50%
- 콘텐츠 팩토리:  GPU 25%, CPU 25%, RAM 25%
- 여유분:         GPU 15%, CPU 25%, RAM 25%
```

**2. 프로젝트별 우선순위**

같은 회사 안에서도 프로젝트마다 급한 정도가 다르다.

```
AI 스튜디오 내부:
- 이미지 생성 API: 우선순위 높음 (매출 직결)
- 코드 분석: 우선순위 보통 (내부 도구)
→ 자원 부족하면 코드 분석이 양보
```

**3. 동적 조정 (핵심)**

고정 할당이 아니라 상황에 따라 유동적으로 바뀐다.

- 콘텐츠 팩토리가 쉬는 시간 → 그 쿼터를 AI 스튜디오가 임시로 씀
- 급한 작업 들어오면 → 낮은 우선순위 작업을 잠시 멈추고 자원 양보
- 밤에 안 쓰는 기기 → 배치 작업 몰아서 실행

### 유저한테 보이는 모습

자원 관리를 유저가 직접 설정할 필요 없다. 파트장(AI)이 알아서 판단하고, 유저한테는 자연어로 상황을 알려준다.

```
유저: "이미지 100장 생성해줘"

사장(Musu-B): "지금 5070Ti GPU가 80% 사용 중이에요.
  - 이미지 생성 API가 60% 쓰고 있고
  - 유튜브 썸네일이 20% 쓰고 있어요.

  선택지:
  1. 지금 바로 (유튜브 작업 일시정지) → 약 30분
  2. 순서대로 (현재 작업 끝나면) → 약 2시간
  3. 4060Ti에서 처리 (느리지만 바로) → 약 1시간

  어떻게 할까요?"

유저: "1번으로"

사장: "유튜브 작업 일시정지하고 이미지 생성 시작합니다.
  유튜브 작업은 이미지 끝나면 자동 재개됩니다."
```

### 자원 충돌 해결

여러 유저가 동시에 요청하거나, 여러 프로젝트가 동시에 자원을 요구할 때:

```
규칙 1: 유저 직접 요청 > 자동 배치 작업
규칙 2: 높은 우선순위 프로젝트 > 낮은 우선순위
규칙 3: 먼저 시작한 작업은 함부로 안 멈춤 (90% 이상 진행됐으면 끝까지)
규칙 4: 어떤 작업도 영원히 굶지 않음 (최소 보장 자원)
규칙 5: 자원 부족하면 파트장이 유저에게 물어봄 (자동으로 안 죽임)
```

### 기기 상태 대시보드

메신저 사이드바에서 실시간으로 보인다:

```
🖥 5070Ti Desktop
├── 🟢 온라인
├── GPU ████████░░ 80% (16GB 중 12.8GB 사용)
│   ├── AI 스튜디오: 60% (이미지 생성)
│   └── 콘텐츠 팩토리: 20% (유튜브)
├── CPU ████░░░░░░ 40%
├── RAM █████░░░░░ 50%
└── 큐: 3건 대기

🖥 4060Ti Desktop
├── 🟢 온라인
├── GPU ██░░░░░░░░ 20% (여유)

## 환경 관리: 도구/설정/MCP를 중앙에서 통제

### 지금의 문제

기기가 3대면 3대 각각에 도구를 깔고, 설정을 맞추고, MCP를 연결해야 한다. 수동으로.

```
기기 A (4060Ti):                    기기 B (5070Ti):
  RTK v0.34.3 ✅                      RTK 없음 ❌
  Claude Code ✅                      Claude Code ✅ (설정 다름 ❌)
  MCP: Notion, Paperclip ✅           MCP: 없음 ❌
  Skills: 24개 ✅                     Skills: 0개 ❌
  Codex CLI ✅                        Codex CLI 없음 ❌
  hooks: rtk-rewrite ✅               hooks: 없음 ❌
  ~/.claude/settings.json ✅          ~/.claude/settings.json ❌ 다름
```

기기 하나 추가할 때마다:
- 뭐가 깔려야 하는지 기억해야 하고
- 버전이 맞는지 확인해야 하고
- 설정 파일 하나하나 복사해야 하고
- 이름이 같은 다른 패키지 깔릴 수 있고 (RTK ↔ Rust Type Kit)
- 뭐가 빠졌는지 일일이 뒤져봐야 알고
- 문제 생기면 "니 쪽 설정 문제" "아니야 내 쪽은 됨" 이러면서 시간 날림

기기가 2대면 참지만, 5대, 10대면? 불가능.

### MUSU의 해결: 환경 매니페스트

회사(또는 프로젝트)마다 **"이 환경에서 일하려면 이것들이 필요하다"**는 매니페스트가 있다.

```yaml
# AI 스튜디오 환경 매니페스트
company: AI 스튜디오
environment:
  tools:
    - name: rtk
      version: ">=0.34.3"
      install: "npm install -g @anthropics/rtk"
      verify: "rtk --version"

    - name: claude-code
      version: ">=1.2.0"
      config:
        settings: ./configs/claude-settings.json
        hooks: ./configs/hooks/
        skills:
          - review
          - investigate
          - ship
          - careful

    - name: codex-cli
      version: ">=2.0.0"
      verify: "codex --version"

  mcp_servers:
    - name: notion
      config: ./configs/mcp/notion.json
    - name: paperclip
      config: ./configs/mcp/paperclip.json

  env_vars:
    MUSU_PORT_MANAGER_HOST: "0.0.0.0"
    CUDA_VISIBLE_DEVICES: "0"
```

### 기기 연결 시 자동 프로비저닝

새 기기가 MUSU에 연결되면:

```
1. 기기 연결
   "5070Ti가 AI 스튜디오에 합류했습니다"

2. 환경 체크 (자동)
   파트장: "환경 점검 중..."
   - RTK: ❌ 미설치 → 설치 필요
   - Claude Code: ✅ 설치됨, 설정 다름 → 동기화 필요
   - Codex CLI: ❌ 미설치 → 설치 필요
   - MCP Notion: ❌ 미연결 → 연결 필요
   - Skills: 3/24 설치됨 → 21개 추가 필요

3. 유저에게 보고
   파트장(5070Ti): "AI 스튜디오 환경에 맞추려면:
   - 도구 2개 설치 (RTK, Codex)
   - 설정 1개 동기화 (Claude Code)
   - MCP 1개 연결 (Notion)
   - 스킬 21개 설치

   자동으로 진행할까요?"

4. 유저: "ㅇㅋ"

5. 파트장: 자동 설치 + 설정 동기화 + 검증
   "완료. 모든 도구가 AI 스튜디오 표준에 맞춰졌습니다."
```

### 버전 충돌 방지

```
파트장(5070Ti): "RTK 설치 중에 문제가 있습니다.
  이 기기에 'rtk' (Rust Type Kit, reachingforthejack/rtk)가
  이미 설치되어 있어요. 이건 다른 패키지입니다.

  선택지:
  1. Rust Type Kit 제거하고 MUSU RTK 설치
  2. MUSU RTK를 다른 이름(musu-rtk)으로 설치
  3. 건너뛰기

  어떻게 할까요?"
```

### 설정 드리프트 감지

기기들의 환경이 시간이 지나면서 서로 달라지는 걸 자동으로 잡는다.

```
[매일 아침 모닝 리뷰]

파트장들 환경 점검 결과:
┌─────────────┬──────────┬──────────┬─────────┐
│ 도구        │ 4060Ti   │ 5070Ti   │ 노트북   │
├─────────────┼──────────┼──────────┼─────────┤
│ RTK         │ 0.34.3 ✅│ 0.34.3 ✅│ 0.33.1 ⚠│
│ Claude Code │ 1.2.3 ✅ │ 1.2.3 ✅ │ 1.2.3 ✅│
│ MCP Notion  │ 연결 ✅  │ 연결 ✅  │ 끊김 ❌  │
│ Skills      │ 24/24 ✅ │ 24/24 ✅ │ 20/24 ⚠│
└─────────────┴──────────┴──────────┴─────────┘

⚠ 노트북: RTK 버전 오래됨 (0.33.1 → 0.34.3 업데이트 필요)
❌ 노트북: Notion MCP 연결 끊김 (재연결 필요)
⚠ 노트북: Skills 4개 누락 (canary, benchmark, cso, codex)
```

### 핵심 원칙

1. **유저는 각 기기에 SSH 들어가서 설정 안 한다.** MUSU가 알아서 한다.
2. **환경 매니페스트가 SSOT(Single Source of Truth).** 매니페스트 하나 바꾸면 전 기기 반영.
3. **기기 추가할 때 0-config.** 연결만 하면 자동 프로비저닝.
4. **드리프트는 자동 감지.** 사람이 일일이 확인 안 해도 됨.
5. **충돌은 물어봄.** 자동으로 막 지우지 않음.

## 기술 스택 매핑

이미 만들어진 것들이 어떻게 연결되는지:

```
유저 화면
  │
  ├── musu-bee (로컬 Next.js 웹앱, localhost:3001)  ← 기능 전부
  │     ├── 채팅 UI (React)
  │     ├── 기기 상태 패널 (React)
  │     └── 작업 보드 (React)
  │
  └── musu.pro (Vercel, relay 통해 원격 접근)  ← 계정/결제/랜딩
              │
              ▼
        musu-port (각 기기의 로컬 control plane)
        ├── 서비스 라우팅
        ├── 기기 상태 API (/health, /routes)
        └── 작업 큐 관리
              │
              ▼
        musu-connects (기기 간 연결)
        ├── QUIC 보안 통신
        ├── 라우트 교환
        └── 피어 인증 (Tailscale/직접)
              │
              ▼
        MUSU-WORKS (업무 분배 엔진)
        ├── 작업 큐 / 레인
        ├── 자원 기반 라우팅 (GPU 필요 → GPU 있는 기기)
        └── 거버넌스 (승인, 에스컬레이션)
              │
              ▼
        musu-computer-tools (AI가 기기를 제어)
        ├── 스크린샷, 마우스, 키보드
        ├── 터미널 제어 (PTY)
        └── 파일 읽기/쓰기
```

## MVP 정의

### 최소 시나리오: "두 대 컴퓨터에서 채팅으로 일 시키기"

1. 유저가 4060Ti 앞에 앉아서 MUSU 메신저를 연다
2. "이 이미지 분석해줘" 라고 보낸다
3. Musu-A(4060Ti, 사장)가 받는다
4. "GPU가 필요해서 5070Ti에 넘깁니다" 라고 답한다
5. Musu-B(5070Ti)가 분석하고 결과를 채팅에 올린다
6. 유저가 채팅에서 결과를 본다

### MVP에 필요한 것

- [x] 채팅 UI (React, musu-bee 웹앱)
- [x] 파트장 AI (각 기기에서 돌아가는 에이전트)
- [x] 사장 로테이션 로직
- [x] 기기 간 메시지 전달 (musu-relay WebSocket 위에)
- [x] 기기 상태 표시 (get_dashboard + list_nodes)
- [x] 작업 분배 판단 (route_task recommended 전략)

### MVP에 필요 없는 것 (나중에)

- 칸반 보드
- 파일 공유
- 음성/영상
- 3대 이상 연결
- 자동 스케일링
